3

What one agent list does — the run loop

A card on an agent list fires one run. The list hands the session a closed set of valid next-list names, and the only thing the engine trusts back is the last line.

① Card enters an agent list

By one of three triggers set per list: immediate on entry, the list's own scheduler beat (e.g. Test every 5h), or manual via Start. verify the seed lists carry no trigger yet, so today they all behave as immediate.
engine mints/threads runId · builds prompt↓

② Combined prompt is assembled

execute-prompt (evokes the work, which loads the skill) + router-prompt + the explicit validNext[] + the card's runDir so the skill writes per-run + a /goal prefix if goalMode. Implement also hands the writer the plan slice, acceptance, and architecture doc. Attached as a hint, not text: the list's {taskType, tier}.
POST /chat · hint honored in both gateway modes↓

③ Orchestrator resolves the route engine

honor the {taskType,tier} hint → resolveRole (exception > cell > row > column > global) → role → Profile's roleMap → target. applySwitch: model/effort change is slash-injected; provider/soul change respawns. On a Codex list the operative is modest, since the cross-model strength comes from the codex CLI the skill shells out to.
runs the combined prompt↓

④ Session does the work

The operative runs the prompt, the skill loads off the content, drives the app or shells out to Codex, and writes artifacts under the run dir. Concurrent code edits across cards are kept safe by agent_mail file claims and beads awareness. verify today this is one pooled, serialized operative.
reply · last line = one next-list name↓

⑤ Engine reads the verdict

It exact-matches the last non-empty line against validNext[]. The run's session and artifacts are linked onto the card either way.
✓ exact match
Move the card forward. From Review the line is Adv Review.
✗ no match · error · iteration cap
Park in needs-attention and notify. Any failed gate ends with Implement and the card loops back.
Convergence in Implement is the iteration cap. Prepending /goal is only prompt text here; the autothing goal loop never fires for a Kanban run. The per-card iteration cap plus the loop-back edges is the guard.

Flow A — infer the title, then build

The default path. Drop a card, it names itself, press Start, and it walks the pipeline on its own, looping back to Implement on any failed gate.

1
Drop a card into Backlog
Infers the title eagerly, and the project only at 70% or more, otherwise it parks in needs-attention until you set it.
2
Press ▶ Start
The card carries its description to Plan. The engine mints a runId for the card here.
3
Plan
autothing-plan writes the plan and acceptance under the run dir, ends with Implement.
4
Implement
autothing-implement builds, handed the slice, acceptance, and architecture doc. Ends with Review.
5
Review
Same-model code-review. Pass ends with Adv Review, fail with Implement.
6
Adversarial Review
Codex tries to break the diff. Pass ends with Test, fail with Implement.
7
Test, on its beat
The project's waiting cards are tested together in one session. Pass ends with Adv Test, fail with Implement.
8
Adversarial Test
Codex drives the app with its own functional pass. Pass ends with Walkthrough, fail with Implement.
9
Walkthrough
autothing-walkthrough records the verified video and links it onto the card. Ends with Validate.
10
Validate
autothing-validate checks the Definition of Done, video included, and ends with Done.
11
Done
Terminal manual column. The card still links its plan, sessions, and evidence.

Flow B — conversation first

An alternative entry. Before planning, the card goes through Discuss to turn a rough idea into a brief, with you in the loop. Interactive only for now.

1
Drop a card into Backlog, title inferred
Same start as Flow A.
2
Move it to Discuss instead of Plan
You route here when the idea needs shaping first.
3
Talk it through in the web chat
The operative in James mode thinks it through with you, responsive and read-aloud, and writes a brief to disk. The card links the session and the brief. You press ⤷ Move when ready.
4
Brief goes to Plan
Plan builds from that brief and the card rejoins Flow A at step 3.

Auto-infer of the brief is parked until there have been enough real interactive conversations to decide what an automatic version should do.

8

Decided, and what's left

The four forks are settled. What remains is verification during the build and the build brief itself.

Decided.
  • One skill family (autothing-*); doctrine in CLAUDE.md and docs; resources via the install coupling; one net-new verb, autothing-validate.
  • Each pass is its own list, including the two cross-model Codex passes, for observability.
  • Discuss is James mode plus brief-to-disk, interactive only, manual advance. Auto-infer parked.
  • Isolation: per-card runId for artifacts (the engine mints and threads it), and agent_mail + beads for concurrent code edits. Full concurrency, no per-project serialization.
  • Convergence in Implement is the iteration cap.
  • Walkthrough is in, using the existing skill; Validate keeps its video gate.
  • Gateway: target the mode that honors the routing hint, and add a small fix so it is honored in both modes, so the board is never fragile to composition changes.
  • One chat surface: the context-driven web channel for Dev Env, Kanban, and standalone; operative test interface untouched.
  • Link, never duplicate, and fully responsive, phone-first.
Deferred, on purpose
  • Codex serialization. Two adversarial passes colliding is unlikely; if it bites, a board-level lock is the fix.
  • Auto-infer of the Discuss brief.
  • Autothing-as-a-preset (a dedicated session running the whole skill for true per-card convergence).
Verify during the build verify
  • autothing-implement reads doctrine from docs as well as when handed the area skill. Test one slice.
  • Where the walkthrough skill writes its video, so it can be linked.
  • The Claude Code session transcript path under ~/.claude.
  • The operative / Dev Env PTYs are tmux-attachable.
  • autothing-validate and the others honor a passed-in runDir.
  • The three-trigger model and the Test scheduler beat need wiring.
