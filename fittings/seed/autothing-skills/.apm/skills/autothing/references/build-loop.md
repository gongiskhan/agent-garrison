# Build Loop (Phases 3-4)

The gated, autonomous build. Runs under the operator's `/goal` + auto mode — never pause, fix forward, log blockers, continue.

## Resume FIRST (every invocation, including after compaction)
Reconstruct progress from this run's durable files, never from memory. **The full resume protocol is the SINGLE SOURCE in `autothing/SKILL.md` → "Resume mode"** — discovery, the ownership lock + single-owner check (a fresh `owner.lock.json` / sentinel another session holds means the run is owned: log and stop; a stale lock >10 min is claimable), the ground-truth re-run of the last passed gate's suite on THIS machine, untrusted uncommitted working-tree scratch, re-arm with old-shape `gatesConfig` compatibility (old `review`+`adversarialReview` → the merged review; missing keys → defaults; profiles apply to NEW runs only), the `RESUME` entry, and post-compaction mini-resume. **All run state lives under the per-run `<runDir>` (`docs/autothing/runs/<runId>/`) chosen in Phase 0 — never a shared `docs/FLOW_PLAN.md`**, so concurrent runs cannot clobber it. In brief: find `<runDir>` (same session → the per-session sentinel; fresh session → newest `runs/<runId>/` with globalGate ≠ `passed`, after the single-owner + stale-lock checks), re-run the last passed gate's checks, verify-or-discard uncommitted scratch, and log a `RESUME` entry. Then do the build-loop scan:
1. Read `<runDir>/FLOW_PLAN.md` slice table.
2. Read each `<runDir>/slices/<slice>/gate-status.json`.
3. Read `<runDir>/evidence-index.json`.
4. Compute **buildable-remaining** = count of slices whose status is neither `passed` nor `blocked` (a slice counts as `blocked` only if it carries rule-2 attempted-remediation evidence).
A slice with `status: passed` and `video.status` in {verified, failed-but-unblocking} is **done** — skip it. Resume at the first non-done slice.

At the end of EVERY turn, after this scan, print one line so exhaustion is visible to the transcript-only `/goal` evaluator:
`PROGRESS: passed <p>/<total> · blocked <b> (remediation-attempted) · buildable-remaining <r>`
A `GLOBAL GATE:` verdict may NOT be printed while `buildable-remaining > 0` — continue to the next buildable slice instead.

**When the coord stack is connected (coord-mcp + agent-mail — see SKILL.md "Cross-session coordination"), the resume scan ALSO polls the other sessions.** Read `coord_digest` + `fetch_inbox`; handle inbound messages (`acknowledge_message` + steer if another session asks you to avoid a path or announces a change); and print a second ledger line `COORD: leases <held> · digest-conflicts <n> · inbox <unread>`. A new digest conflict on an unbuilt slice's files re-orders the loop (build a non-conflicting slice first) exactly as a failing gate would — cross-session awareness is part of choosing the next buildable slice, not a side task.

## Per-slice loop
For each slice (its acceptance is in FLOW_PLAN; route via the routing index). **Each step below is its own STANDALONE skill that autothing orchestrates** — and any gate that finds real issues sends the slice **back to `autothing-implement`** (the loop), bounded by the **5-attempt ceiling**. Run the steps in order; each prints its own `GATE …` line in the lead context (see the lead-print invariant). The same skills are usable on their own outside a build (then they just report instead of looping).

**Skipped gates (operator / kind / profile — see SKILL.md "Operator gate toggles", "Pipeline consolidation", "Run profiles").** Before running steps 2-6, check `gatesConfig` (from `evidence-index.json`, fallback the sentinel's `gates`) AND the slice's `kind` + the run's `profile`. A step that does not run is **skipped, not run**, recorded in that gate's slot of `gate-status.json` with the reason and printed — never a silent pass:
- operator-disabled flag → `{"status":"skipped","reason":"operator-disabled"}`, `GATE <name>: skipped (operator-disabled)`
- kind-conditional (design-audit + walkthrough for `ui`/`mixed` only; per-slice adversarial-test for `ui`/`mixed`, `api` batched run-level) → `{"status":"skipped","reason":"kind-conditional"}`
- profile-skipped (a gate the run's `patch`/`feature` profile excludes) → `{"status":"skipped","reason":"profile:<name>"}`
- `codexSliceReview` conditional (step 3b — `build` always; `feature` only security-boundary slices; never `patch`) → not-selected records `{"status":"skipped","reason":"profile-conditional"}`; a Codex quota/auth/availability death records `{"status":"degraded","reason":"codex-unavailable"}` and CONTINUES (never blocks, never fakes a verdict)
A skipped gate never sends the slice back to implement and never counts toward the full-bar `passed`. Steps 1 (implement) and 7-10 (record/self-unblock/blocked/friction) always run. **The floor never lowers (Part 11.3):** the deterministic wall (typecheck/lint/structural greps/`securityWall`) + committed tests always run, and a security-boundary diff always gets the step-3 review regardless of profile.

0. **Claim the slice's files (coord stack connected only)** — before editing, take an agent-mail **file reservation** for the slice's files (`macro_file_reservation_cycle`: paths + TTL + heartbeat + reason) and `declare_intent(area, files, reason)`. Check `coord_digest`: if another live session holds any of those files, treat them as shared → serialize behind it, pick a different buildable slice, or `send_message` to negotiate ownership. Renew the lease + `plan_heartbeat` on long slices; **release the reservation + intent as soon as this slice reaches `passed`/`blocked`** (step 7/9). Absent stack → skip; the disjoint-files discipline is the whole contract.
1. **Implement** → **`autothing-implement`** — explore (vision-first for UI, via `/run` + `/verify` + playwright-cli), then write the slice's code per its acceptance + the project's area skill.
2. **Deterministic wall + Test (correctness gate)** → **`autothing-test`** — FIRST the deterministic wall (Part 8.3: `typecheck` / `lint` / structural greps + the universal **`securityWall`** — gitleaks + semgrep + dependency-audit, zero tokens), THEN a COMMITTED, re-runnable test + run it, plus `tests`/`e2e`/`build` exit 0. **Default to e2e through the UI** (Playwright / playwright-cli) + unit tests; a CLI/TUI deliverable uses a committed driver + asciinema. **No model-judgment gate (steps 3-5) runs until this wall is green.** Any failure → **classify before looping (Part 5.1)**: `code` → **back to step 1** (consumes the ceiling); `test-bug` → fix the test, log a `DECISION` (no ceiling cost); `infra-flake` → fix infra, log a `DECISION`, append to `docs/autothing/known-flakes.md`, re-run (no ceiling cost). Ephemeral `.playwright-cli/` logs are exploration, never the gate. **`patch` profile:** the core gate is regression-test-first (a committed FAILING test reproducing the bug, then the fix, then green).
3. **Review (fresh-context — THE per-slice review)** → **`autothing-adversarial-review`** — Part 8.1 merged the old same-model `autothing-review` into this single review: a fresh-context Anthropic session (Fable-pinned, no implementer notes/rationale) reviews the slice diff against ONLY its acceptance + the spec/plan sections it cites, gathering its OWN build/typecheck/lint/test evidence and citing a spec/acceptance section or attaching failing output for every finding (evidence-mandatory — opinions without evidence are not findings). Real, evidence-backed findings → **back to step 1**. A **security-boundary diff gets this review regardless of profile** (rubric: authorization on every touched path, injection surfaces, secret handling, tenant/org scoping, input validation at the boundary — Part 12.7). The same-model `autothing-review` is retired from the pipeline (its unique catches were ~zero once the deterministic wall + fresh-context passes existed) — usable standalone only.
3b. **Per-slice Codex adversarial pass — CONDITIONAL, runs ALONGSIDE step 3** (`build` profile: always; `feature`: only on slices whose diff touches security-boundary code — the Part 11.3 trigger; `patch`: never; skip entirely with `--no-codex-slice`). Evidence: run `20260706` RUN_LOG (per-slice Codex caught real Criticals at G4/G7/G7A/G7B/G8). ONE **SERIAL** `codex exec`, **GPT-5.5 at high reasoning** (`-m gpt-5.5 -c model_reasoning_effort=high`), diff-scoped **narrow rubric** (the slice's invariants + acceptance + the diff, prompt "find violations" — never open-ended). Print `CODEX CALL: gate=codexSliceReview model=… effort=high slice=… diff=[<shortstat>]` in the lead context FIRST; then print `GATE codex-slice: <approve|needs-work|degraded|skipped> — <summary>`. Record `codexSliceReview: {verdict, by, actualModel, durationMs}`. A Claude-agreed real, material **`needs-work`** → **back to step 1** within the slice's EXISTING 5-attempt ceiling (Claude is the deciding authority; rebut the rest). A slice the conditional does not select records `{"status":"skipped","reason":"profile-conditional"}`. **Credit/availability death (Part 3 honesty):** a `codex exec` quota/auth/availability failure records `codexSliceReview: {status:"degraded", reason:"codex-unavailable", actualModel:null}`, emits the mid-run notification (cause `codex-unavailable`), logs a `DECISION`, and **CONTINUES** — never blocks the run, never fakes an `approve` (the fresh-context step 3 still stands). All `codex exec` calls serialize with the run-level checkpoint. Auth via **API KEY, not ChatGPT sign-in**, with a budget cap (see `autothing-codex-checkpoint/references/codex-checkpoint.md`) — an unattended run must not drain the interactive account's 5-hour/weekly pool. Full policy: SKILL.md "Per-slice Codex adversarial pass".
4. **Adversarial test (independent)** → **`autothing-adversarial-test`** — a fresh-context Anthropic session (no diff, no committed test) independently drives the running app through the acceptance with its own self-written Playwright probes. `fail` with a reproducible probe → **back to step 1**. **Kind-conditional (Part 8.2):** per-slice for `ui`/`mixed`; for `api` slices this runs as ONE batched run-level pass over the API surface (per-slice slot records `{"status":"skipped","reason":"kind-conditional"}`).
5. **Design audit (ui/mixed only, Part 8.2)** → **`autothing-design-audit`** — judges the running app against the project's design tokens (uses `frontend-design` / `huashu-design`). Record `designAudit: {verdict, by, at, issues}`. `issues` → **back to step 1** (same ceiling). `api` slices record `{"status":"skipped","reason":"kind-conditional"}`.
6. **Evidence (ui/mixed walkthrough; api = asciinema)** → **`autothing-walkthrough`** — self-verified video in **`mode: evidence`** (Part 4: one beat per acceptance criterion, short holds, per-beat vision + inline asserts, an emitted `evidence.json`, no holistic gate). A `walkthrough` STUCK becomes `video.status: failed-but-unblocking` and the run CONTINUES; an evidence-mode caption/legibility failure whose functional assert passed is recorded `evidence-degraded` (see "Evidence" below).
7. **Record durably + RUN_LOG GATE entry** — write `<runDir>/slices/<slice>/gate-status.json` (schema: `assets/gate-status.example.json`, incl. `adversarialReview` + `adversarialTest` + per-gate `durationMs`) and upsert the slice into `<runDir>/evidence-index.json`; set the slice's Status in `<runDir>/FLOW_PLAN.md` to match; append one **`GATE`** entry to `RUN_LOG.md` per gate (green condition, evidence pointer, review verdicts **with the actual model each ran on**, checkpoint commit + tag, and **wall-clock duration + the model each step ran on** — Part 12.1: `date -u` timestamps, never fabricated token counts). Refresh `owner.lock.json` + rewrite `status.json`. **Print every `GATE …` line for this slice in the lead context — never delegate a gate print into a workflow agent's output** (see *Gate lines must print in the lead context (ultracode-safe)*). **The determinism ratchet (Part 9.1):** each accepted review finding ships WITH a deterministic guard (lint rule / Semgrep pattern / grep gate / test) in the SAME fix, recorded finding → guard, so that bug class is machine-caught forever.

8. **Self-unblock before blocking** — if the slice is failing because a tool, dependency, or binary is missing, attempt to install it ONCE with the ecosystem's standard command (`uvx` / `pipx install` / `brew install` / `npm i -g` / `pip install --break-system-packages` / `cargo install`) and re-run. A missing tool is NOT a blocker until that install has been tried. Only an install that FAILS and genuinely needs operator credentials or hardware the operator lacks is a legitimate external blocker. (This install attempt is remediation, not a slice fix-attempt — it does not consume the slice's 5-attempt ceiling.)
9. **Blocked path** — a slice may end ONLY as `passed` or `blocked`; "deferred", "consolidated later", and "interim proof" are FORBIDDEN, never terminal states. Mark `blocked` ONLY after either (a) the slice's 5-attempt ceiling is hit on genuinely buildable work — at **attempt 3 the retry runs FRESH-context** (Part 5.2: a new agent with only the slice acceptance + findings, no prior-attempt transcript — repeated failure is often context poisoning; log the escalation) — or (b) step 8's remediation failed on an external cause. Append a blocker to `docs/decisions.md` naming the **external cause and the exact remediation command that failed** ("not installed" alone is never a blocker), mark the slice `blocked`, and move to the next slice. Do not stop the run while any buildable slice remains.
10. **Record friction (signal only)** — when you work around autothing itself being silent, missing a step, or wrong, append one line to `docs/autothing/friction-log.md` in the format `- <UTC> [<skill>] <what happened> → <suggested prose fix>` (Part 6.1: `date -u` stamp, at every gate boundary + run end; never feedback about the product being built). autothing NEVER acts on this and never edits skills; the nightly `skill-improver` reads it as its main improvement signal. This is the only "learning" autothing does.

The per-slice adversarial gates are now decorrelated by **fresh context**, not a second vendor — their mechanics (what they may/may not receive, evidence-gathering, verdict shape) live in `autothing-adversarial-review`'s and `autothing-adversarial-test`'s own SKILL.md files, and they also run standalone. The genuine cross-model check (Codex) now runs ONCE, in the final phase, via **`autothing-codex-checkpoint`** — its operational rules (serial `codex exec`, `</dev/null`, preflight/auth, output schema) live in that skill's own `references/codex-checkpoint.md`, which supersedes `references/codex-verification.md` for cross-model duty (that file is retained for history only).

## Gate builds must not clobber a live dev server
Before ANY gate `build` (per-slice step 2 or Phase 4): check whether a dev server is already serving this project dir from outside the run (`lsof -iTCP:<port> -sTCP:LISTEN -n` on the project's known dev port). **Any process on that port whose PID is NOT a direct child of this run is an external dev server** — launchd-managed and system-spawned processes are always external even when they serve the same repo. `next build` and `next dev` share `.next/`, so a gate build over a live dev server corrupts its incremental state — and the breakage is invisible to a static-route smoke (already-compiled routes keep serving 200 while dynamic/not-yet-compiled routes 404). When a live external dev server is detected, run the gate build where it cannot write the live `.next/`: in a `git worktree` copy, or with an isolated dist dir. The gate itself is unchanged — `build` must still exit 0. If the shared `.next/` was touched anyway, restart that dev server and smoke a DYNAMIC route (not just a static one) before calling the environment healthy. The same hazard applies to dev servers the run itself starts (an e2e sandbox, a second `next dev`): two dev servers sharing one `.next/` poison each other's route cache exactly like a build does — give any run-spawned dev server its own isolated dist dir too, or treat the live server as touched (restart + dynamic-route smoke) when it shared `.next/`.

## Gate lines must print in the lead context (ultracode-safe)
Every `GATE <name>:` line (including `GATE security:` / `GATE adversarial-review:` / `GATE adversarial-test:` / `GATE codex-slice:` per slice and `GATE mutation:` / `GATE security-review:` / `GATE codex-checkpoint:` in the final phase), every per-turn `PROGRESS:` line, every `CODEX CALL:` pre-call line (now printed once per checkpoint scope, not per slice), and the terminal `GLOBAL GATE:` line MUST be printed in the **lead context** — after any fanned-out workflow returns — and **NEVER emitted from inside a workflow agent**.

**Why.** The transcript-only `/goal` evaluator, the goal-loop `Stop` hook (`hooks/goal-stop.sh`), and the durable resume scan all read ONLY the main transcript. A gate line whose only print-site is inside a workflow agent reaches the lead solely as the workflow script's return value and is otherwise lost — a silent gate disappearance. This is the same class of silent failure as "asserted-not-lived by choice": the un-surfaced part is exactly where the regression hides.

**The terminal verdict carries a machine signature — keep it.** The goal-loop Stop hook keys on the verdict's **`(run <runId>)` tag + `videos:<n>/<n>` token** (e.g. `(run 20260622-143110-ab12cd34) … videos:3/3`). The runId makes it match THIS run's real verdict ONLY — never the *quoted* `"GLOBAL GATE: passed"` /goal target (no `videos:N/N`), and never a stray/example/quoted/prior-run `GLOBAL GATE` line (different or absent runId). NEVER drop the `(run <runId>)` tag or the `videos:<verified>/<total>` field, and never let a workflow be the verdict's only print-site — any of those silently breaks the loop's completion detection (it would run to the turn cap, or release early).

**ultracode may stay ON — it is additive.** ultracode = `xhigh` effort + automatic per-*task* workflow orchestration; it operates per task, not per skill, so it does not wrap autothing's flow. Workflows return control to the lead between phases, so a workflow spawned inside one gate step cannot reach across into the next gate — **as long as this lead-print invariant holds.** Test verdict: **ULTRACODE-SAFE** (see `references/decisions.md`): a live 2-slice run under `/effort ultracode` had 2 of 4 tasks genuinely fan out to task-scoped workflows; all 4 gate lines reached the main transcript in order; no workflow spanned more than one named task. A hazard probe proved the converse mechanically — a gate line whose only print-site is inside a workflow agent is lost to the lead.

**Known limitation (record, do not solve here).** Only single-phase fan-outs were tested. A multi-phase workflow that internally chains understand→change→verify within ONE named task was NOT tested; the invariant protects it in principle, but a gate line meant to print *between* a workflow's internal phases is untested.

## A brief naming an external resource is a claim, not authority to provision
Before provisioning into (or writing to) any external, billable, or shared resource the brief names — cloud project, billing account, bucket, remote DB — confirm the **operator's own identity** can access it (e.g. `gcloud projects describe <id>` as the logged-in account). If the operator's accounts are permission-denied on a brief-named resource, that is a **contradicted premise**: do NOT route around it with alternate credentials (a service account or key found locally) — provisioning into a resource the operator cannot access is provisioning into someone else's property. Mark the step blocked, log the contradiction in `docs/decisions.md`, and continue. Fix forward applies to the build, not to someone else's cloud.

## Evidence — delegate to `walkthrough`, reconcile its STUCK exit
- Invoke `walkthrough` on the slice, passing the **slice diff + task context + acceptance** so its flow selection is accurate. It owns recording, captions, frame extraction, vision self-verification, its own retry ceiling, honest failure rendering, its notes file, and publishing the Tailscale link + gallery.
- **walkthrough's ceiling ends in "write STUCK.md and ask the user." autothing must NOT wait for input.** Treat a STUCK/ask-user return as: set `video.status: "failed-but-unblocking"`, record the STUCK.md path + link (if any), append a blocker to `docs/decisions.md`, and **continue**. The slice still counts as having evidence for the global gate (an unblocking failure), but is flagged.
- A genuine feature failure that walkthrough renders honestly (`flagged: true`) is recorded, not faked green.
- Ensure walkthrough's deps are satisfied once per machine (Phase 1 ran its preflight; on macOS `brew install asciinema agg` if missing).
- After it returns, **confirm the gallery URL actually resolves** (the `walkthrough` serve must be running); if it is down, (re)start it so the recorded link is live, not just written into `evidence-index.json`.
- **Terminal / CLI (TUI) slices have their own evidence path — a browser walkthrough does not fit one.** When the slice's deliverable is a terminal app or CLI (not a web flow), record evidence as an asciinema/PTY capture of the real flow (asciinema + agg already ship in walkthrough's preflight) alongside the step-1 committed re-runnable assertion. Do not substitute this ad-hoc each time; treat it as the first-class evidence for that slice class.
- **For an event-streamed or dynamic flow (escalation, an agent run, a live status/progress stream), the evidence MUST be a LIVED capture of that exact flow end-to-end** — not a mechanism proxy or a stand-in that exercises only part of the path. "Verified by mechanism" (a label-free banner capture, a unit test of the handler) is how a frozen-looking UX regression slips through: the un-lived dynamic part is precisely where the bug hides. This is the same non-negotiable as "asserted-not-lived by choice" being a forbidden terminal state — make the capture drive the real streamed turn.

## Parallelism — earn it, then prefer agent teams (full guide: the `autothing-parallel-work` skill)
- Parallelize a group of slices ONLY when they own **disjoint files** AND don't share a stateful runtime. No mechanism makes same-file edits safe — decompose at plan time (split monolith files) to earn parallelism, and log the parallel-vs-serial choice per group.
- **Prefer agent teams** for parallel slice implementation when the operator enabled them (`CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`); else use a **dynamic workflow** (with `isolation: 'worktree'` for file isolation — needs git). Keep team bursts short-lived (create → batch → gate → shut down → clean up): in-process teammates do not survive `/resume`.
- **Preflight the mechanism once, before the first parallel batch:** probe that agent-team creation does NOT raise an approval prompt under auto mode (create a trivial throwaway team and confirm it proceeds, or rely on the known approval behavior). If team creation would block a headless run, fall back to **dynamic workflows for the whole run** — an approval prompt that stalls unattended is itself a silent killer of autonomy.
- **Workers/teammates do not inherit the lead's context.** Paste the area skill (or path + read directive), the slice acceptance, and the exact file-ownership boundary into each spawn prompt.
- **Cross-session safety (first-class when the stack is connected).** The operator runs many sessions on `main` at once. Claim each unit's files as agent-mail reservations + `declare_intent` (and check `coord_digest` + `fetch_inbox`) before editing — the disjoint-files rule made enforceable BETWEEN sessions — AND stay aware: a live intent/lease or an inbound message from another session re-orders your slices or narrows your area, exactly like a gate finding. Use it when the coord stack is present; fall back to disjoint-files alone when absent, never hard-block. Full mechanics: `autothing-parallel-work` ("Cross-session coordination"); autothing's per-phase touchpoints: SKILL.md "Cross-session coordination".
- **Always serialize the shared runtime**: one dev-serve / bundle / `walkthrough` recorder (unique ports + run dirs). Parallelize the EDITING; serialize the BUILD → VERIFY → RECORD tail. There is no per-slice Codex call to serialize anymore — the one place Codex runs (`autothing-codex-checkpoint`, final phase) is a single run-level step, not part of the parallel slice batch; see `references/codex-checkpoint.md` under that skill for its own serial-call rule.
- Never let parallelism skip a gate — every slice passes the full **enabled** sequence (committed test + clean build + **fresh-context review + independent test pass** + design audit + verified video, minus any gate the operator disabled via `gatesConfig`), and the run as a whole still gets its final-phase codex checkpoint. Parallelism must never *silently* drop a gate; only an explicit `gatesConfig` toggle removes one, openly recorded.

## Loop lifecycle — deliberate-red, active waiting, graceful turn-cap landing
**Deliberate-red, once per run (Part 5.5, default ON for runs ≥ 3 slices; `--no-deliberate-red`).** Near the start, plant one violation each for the run's structural gates (a boundary-breaking import, a fake secret for the secrets scanner), show the gate genuinely fail (record the red evidence), then revert. A gate that has never been seen red is not known to work. Record `deliberateRed: {status, gatesProven}` in `globalGate`.

**Active waiting — never end an empty turn (Part 5.7).** When the lead has no buildable action (workers/subagents mid-flight, a long external wait), wait ACTIVELY: run a bounded `sleep` (30–90s) in a shell between status checks, then check worker output and end the turn — every turn must contain a real action. Ending an empty turn under the goal loop produces a stop-block busy loop: the hook re-blocks instantly, the iteration counter freezes, and blocks burn against the cap doing nothing. (`hooks/goal-stop.sh` MAY add a short cooldown when it observes repeated blocks with no transcript growth — message/timing only, the matching logic stays frozen.)

**Graceful turn-cap landing (Part 5.3).** At ~90% of the turn cap: stop starting new slices, finish or cleanly PARK the current one, commit, and write a `PAUSED` entry to RUN_LOG (state, next slice, anything in flight) — so a cap landing is a clean resume boundary, never a mid-slice corpse. Notify the operator (`PAUSED`, Part 6.2).

## Phase 4 — global gate + handover
**No voluntary deferral, and no verdict while buildable work remains.** A slice may end only as `passed` or `blocked` — "deferred", "consolidated later", "interim proof", and "asserted-not-lived by choice" are NOT terminal states and are FORBIDDEN as blocker reasons. **buildable-remaining** = count of slices whose status is neither `passed` nor `blocked` (and `blocked` only counts when it carries the step-8 attempted-remediation evidence). **A verdict line MAY NOT be printed while `buildable-remaining > 0`** — return to the per-slice loop and build the next buildable slice instead.

Once `buildable-remaining == 0`, run the **run-level closing gates IN ORDER** BEFORE deciding the terminal state:
1. **Mutation testing (Part 9.2, runs ≥ 3 slices unless `--no-mutation`).** Wire Stryker (or equivalent) over the CHANGED modules with a ratchet (score never decreases) — the zero-token, systematic version of deliberate-red: "do the committed tests actually detect bugs?" answered with math. Record `mutation: {status, score, baseline}` in `globalGate`; print `GATE mutation: <score|off> — <summary>`.
2. **Built-in security review (Part 12.6).** Run Claude Code's built-in `security-review` ONCE over the finished repo, BEFORE the codex checkpoint — so the cross-model pass adversarially checks what the built-in review already cleaned (two decorrelated passes, not two overlapping ones). Profiles: `build` always; `feature` when security-boundary surfaces were touched; `patch` never un-escalated. Real findings loop to `autothing-implement`; record `builtinSecurityReview: {status, findings, at}` in `globalGate`; print `GATE security-review: <clean|issues-fixed|issues-open|skipped> — <summary>`.
3. **Codex checkpoint** (`autothing-codex-checkpoint`, once for the whole run — unless `gatesConfig.codexCheckpoint === false`): it checks the security-critical scopes (whole-repo security, `shared/` contract, anonymisation/egress pipeline, auth/session, plus anything the brief named) with a small number of high-effort, narrowly-scoped `codex exec` passes. Real findings are fixed and the affected scope re-checked (ceiling: 2 rechecks per scope); record `codexCheckpoint: {status, scopes, at}` in `evidence-index.json`'s `globalGate` and print `GATE codex-checkpoint: <clean|issues-fixed|issues-open|skipped> — <summary>`.

Two terminal states — never conflate them, and BOTH require `buildable-remaining == 0`:
- **`passed`** only when ALL of the ENABLED gates hold (a gate the operator turned off via `gatesConfig` is **excluded** from this list, rendered `off` in the verdict, and makes the pass a declared reduced-bar): **buildable-remaining == 0**; zero blockers; full `npx playwright test` exits 0 (if `test` on); the **`securityWall` (gitleaks/semgrep/dep-audit) is clean** (always — the floor never lowers); `/verify` on the whole app is clean; `build`/`typecheck`/`lint` exit 0; full design audit clean (if `design` on); **every slice has a clean fresh-context review `approve` and an independent-test `pass`** (`decorrelatedVerification.reviewAllApproved && decorrelatedVerification.pwTestAllPassed`, if the adversarial gates are on); **every slice that ran the per-slice Codex pass got a clean `codexSliceReview`** (if `codexSliceReview` on; a `degraded (codex-unavailable)` slot does NOT block `passed`, but is named in LANDING); **every slice's video is `verified`** (`everySliceVideoVerified: true`, if `walkthrough` on); the **mutation** ratchet holds (if on); the **built-in security review** is `clean`/`issues-fixed`; AND **the codex checkpoint is `clean` or `issues-fixed`** (if `codexCheckpoint` on). With no gates disabled this matches the operator's `/goal` condition "a self-verified video exists for every slice" plus decorrelated verification at both the slice and run level; with gates disabled the verdict's `gates-disabled:<list>` token names exactly what did not run.
- **`completed-with-blockers`** legitimate ONLY when **buildable-remaining == 0** AND EITHER ≥1 slice is `blocked` with a **named external cause + the exact remediation command that failed** (a revoked/absent `codex login` that survived the self-unblock attempt is such a cause) OR the built-in security review / codex checkpoint ended `issues-open` (a real, agreed violation survived its 2-recheck ceiling). A `failed-but-unblocking` or `evidence-degraded` video is recorded evidence but keeps the global gate out of `passed`.

Never repeat failed work unproductively, but ALWAYS continue while any buildable (non-blocked) slice remains. The verdict is printed only when buildable work is exhausted — reaching the turn cap with buildable work left is a failure of the loop, not a stop condition. Never fake `passed`.

Set `globalGate.status` in `<runDir>/evidence-index.json`, then do these in order — **the verdict line is LAST because it is the completion signal**: the `/goal` evaluator reads only the transcript and clears on it, ending the session, so anything after it never runs.
0. **Release cross-session coordination (coord stack connected only)**: `release_file_reservations` + `release_intents` + `end_planning`, and `send_message` a short completion summary to the project channel (what landed on `main`, which files, the commit range) so the other sessions can adjust. (`autothing-report`'s Slack message is for the operator; this is for the sibling sessions.)
1. **Write the landing packet** — `<runDir>/LANDING.md` (Part 12.2): the gates summary table, every `DEVIATION`/`DECISION`/`AMBIGUITY`, the assumptions-ledger status (RUN_SPEC runs), model-fallback events, the friction-log items appended this run, and a "needs human eyes" list. The operator's post-run review starts from this one file; `autothing-report` links it.
2. **Print the handover**: stack chosen + why, how to install/invoke, the evidence-gallery URL, an **explicit enumeration of any blockers** (from `docs/decisions.md`), and known limitations (flow selection improves with use; vision verification cannot confirm business correctness). Describe the outcome in prose — do NOT put an assertive `GLOBAL GATE:` line here, so the evaluator doesn't read the handover as the verdict. (The same token in the invocation handoff is a quoted target, not a verdict; the evaluator distinguishes the two.)
3. **LAST line of the run** — print exactly one:
   `GLOBAL GATE: <status> (run <runId>) — e2e:<exit> verify:<clean|issues> build:<exit> typecheck:<exit> lint:<exit> adversarialReview:<approved>/<total> adversarialTest:<passed>/<total> design:<clean|issues> codexCheckpoint:<clean|issues-fixed|issues-open|skipped|off> videos:<verified>/<total> gates-disabled:<list|none> profile:<name> codexSlice:<approved>/<ran|off>` — append ` model-fallbacks:<n>` ONLY when n>0. `codexSlice:` counts only slices that actually RAN the conditional per-slice Codex pass (`off` when `--no-codex-slice`; `0/0` when the profile/boundary conditional never selected a slice); a `degraded (codex-unavailable)` slice is excluded from `approved` and noted in LANDING, never faked into the count.
   **The signature through `videos:<verified>/<total>` is byte-identical to before** — `profile:<name>`, `codexSlice:<approved>/<ran|off>`, and the conditional `model-fallbacks:<n>` are all APPENDED after `gates-disabled:` exactly the way `gates-disabled` itself was added, so the hook-matched fields are untouched. **Any operator-disabled gate renders as `off`** in its field (`e2e:off` / `adversarialReview:off` / `adversarialTest:off` / `codexSlice:off` / `design:off` / `codexCheckpoint:off` / `videos:0/0`) — never as a passing count — and is named in the `gates-disabled:` token (`gates-disabled:none` on a full-bar run). Nothing follows this line.
