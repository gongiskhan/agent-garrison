# autothing

Orchestrates an unattended, gated build of a **significant code change** — a feature, a module/service/page/endpoint, a substantial behavior change, a multi-file refactor, or a whole project — and proves each slice with a self-verified evidence artifact (a browser walkthrough video for web flows, an asciinema capture for CLI/TUI). This skill is USER-scope; the foundation it writes is PROJECT-scope (inside the target repo). The `walkthrough` skill is SEPARATE — call it, never rebuild it.

## Scope — use autothing for any non-trivial code work (NOT only whole projects)
autothing is the default for **work that meaningfully adds or restructures code, at ANY size**: a single feature or page, a new module/service/endpoint, a substantial change to existing behavior, a multi-file refactor, up to a whole new app. A one-slice feature is a valid, common autothing run — the same gates apply at every size, they just scale down (a small feature is often a single slice with one evidence artifact and a near-empty foundation step).

**AUTO-invocation is SOFTWARE-ONLY.** The model may self-trigger autothing ONLY when BOTH (a) the task implements or changes **runnable software** in a code project — primarily JS/TS/Node web apps and services, also Python — where the result is verifiable by build/tests/running the app, AND (b) it is **not** prose, documentation, markdown, configuration, data files, or Claude skills/agent instructions (editing skills or docs **never** auto-triggers autothing, even a large multi-file edit — per-slice build gates are meaningless for prose). Beyond that, **do NOT self-trigger autothing on** (reach for the normal tools, `/run`, `/verify`, or plan mode instead): bug fixes, one-line or small edits, single-function tweaks, formatting/renames/dependency bumps, running or starting the app, writing or running tests on their own, or pure research/exploration/questions. This bounds when the model should *auto*-invoke — it is **NOT** a refusal list for explicit invocations.

**An explicit invocation NEVER refuses — it assesses and adapts.** When the operator explicitly runs autothing on any code task, it does not decline as "too small"; it picks a **run profile** (`patch` / `feature` / `build` — see "Run profiles" below) that scales the pipeline to the work and runs it. A tiny bug fix becomes a `patch` run (regression-test-first + the deterministic wall + secrets scan), not a refusal. **Never refuse a qualifying task because it is "not a whole project" or "too small."** Scale the plan to the work — fewer slices, a lighter (often no-op) foundation step, a smaller profile — but always run that profile's full gate sequence on whatever you do build.

## Run profiles — scale down, never refuse (Part 11)
A run's **profile** is assigned at planning (Phase 1 / `autothing-plan`) by the **same sizing that derives the turn cap** — operator flags always override. The profile is recorded in the RUN_LOG `RUN-START` entry and in `evidence-index.json`'s `gatesConfig`, and the terminal verdict carries a `profile:<name>` token (extended the same way the `gates-disabled` token is — the hook-matched fields stay untouched). Three profiles:
- **`patch`** (bug fix / tiny change, ≤ ~1 sizing point): a micro-plan (a few lines, no RUN_SPEC). **Regression-test-first is the core gate** — a committed FAILING test that reproduces the bug, then the fix, then green. Plus the deterministic wall + secrets scan. Fresh-context review runs ONLY if the diff is security-boundary-adjacent (else skipped, recorded). No design audit / walkthrough / adversarial-test / mutation / deliberate-red / codex checkpoint. Asciinema evidence optional. **RUN_LOG is still written** (one `GATE` entry).
- **`feature`** (1–3 slices): the per-slice pipeline with the kind-conditional gates (see "Pipeline consolidation" below). RUN_SPEC per the 10.1 threshold. Mutation + deliberate-red OFF by default. The codex checkpoint runs only if security-relevant surfaces were touched.
- **`build`** (≥ 4 slices): everything — RUN_SPEC, full per-slice gates, deliberate-red, mutation, built-in security review + codex checkpoint.

**The floor never lowers (11.3).** No profile skips typecheck, lint, structural greps, the `securityWall` secrets scan, or committed tests. **Any diff touching security-boundary code** (auth, crypto, egress, parsing, path containment) **gets the fresh-context review regardless of profile.** Every profile-skipped gate is recorded `{"status":"skipped","reason":"profile:<name>"}` — visible in `gate-status.json` and printed as `GATE <name>: skipped (profile:<name>)`, never a silent pass.

**Escalate, never squeeze (11.4).** If implement discovers the work is larger than the profile assumed, or touches boundary code the sizing missed, the run **escalates the profile mid-run** (`patch → feature → build`), logs a `DECISION` in RUN_LOG, re-derives the turn cap, and re-plans slices if needed. Big work never ships through a small pipeline because of an initial label.

## Operating assumptions (the operator sets these; autothing cannot set session switches)
- Launched under **/effort ultracode + auto mode**. The goal loop is **armed automatically** by autothing's Phase 0 run sentinel + the goal-loop `Stop` hook in `settings.json` (`hooks/goal-stop.sh`) — **no manual `/goal` needed**. **If the hook is not yet wired into this config, Phase 0 self-installs it** (`hooks/install.sh`) and carries on. The completion condition is the global gate (including "a self-verified evidence video exists for every slice") plus a turn cap.
- autothing **arms its own goal loop** (Phase 0 writes the run sentinel; the `Stop` hook keeps the session taking turns until the terminal `GLOBAL GATE:` verdict prints, then deletes the sentinel and releases the session). **It does NOT prompt for `/goal` when the hook is already active** — it prints a one-time `/goal` ONLY on a bootstrap run where Phase 0 just installed the hook (a freshly-added hook activates next session, so that one run can't auto-loop yet), or when hooks are disabled. Once the hook is in this machine's `settings.json` (committed/deployed), every run auto-loops with no `/goal`. `/effort` and auto mode are session toggles the operator flips once; autothing cannot set them. **autothing inherits the session effort (it does not pin its own); run it under `/effort ultracode` (or at least a high effort) — autothing does not self-escalate.**
- **Parallelism (optional, operator-set):** if `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` is set, prefer **agent teams** for parallel slice implementation; otherwise use **dynamic workflows**. autothing cannot enable that env var itself. See the **`autothing-parallel-work`** skill.
- **Cross-session coordination (FIRST-CLASS when present — see the "Cross-session coordination" section below):** the operator runs **many Claude sessions on `main` at once**, so at any moment other sessions may be editing the same repo. When the **coord stack** is connected (coord-mcp planning gate + digest, agent-mail identity/inbox/file-reservations, beads), autothing does NOT just protect files during parallel fan-out — it stays **aware of what the other sessions are doing across the whole run and steers to accommodate them** (re-orders slices away from files another session holds, coordinates by message, defers a contended area). When the stack is absent (no Garrison / MCP not connected), fall back to the disjoint-files discipline — **never hard-block on it.**
- Therefore behave **fully autonomously**: never pause for approval, never ask the user, fix forward, log blockers and continue.
- **Trade-off (noted):** no `allowed-tools` restriction — it both sets up and builds, so it may use any tool. It is now **dual-mode** (slash- AND model-auto-invocable); the safety boundary is its own gates — its first real phase is read-only planning (`autothing-plan`) + writing the run sentinel, so auto-starting is not immediately destructive. **Optional operator hardening (documented, NOT built):** a single `UserPromptSubmit` hook that *suggests* autothing on matching prompts, if auto-trigger ever proves flaky — add it deliberately; do not rely on it by default.

## Owns vs delegates
autothing is the **orchestrator**: it arms the goal loop, sequences the per-slice pipeline, owns the retry ceiling + loop-back, and writes durable evidence. Each STEP of the pipeline is its own standalone `autothing-*` skill (each also usable on its own). autothing never reimplements a step — it invokes the step skill and, on a real failure, loops back to `autothing-implement`.
- **Owns:** arming the goal loop (Phase 0 sentinel + `hooks/install.sh`), the per-slice pipeline orchestration (sequence + 5-attempt ceiling + loop-back), and durable evidence (gate-status + evidence-index).
- **The pipeline (each a standalone skill autothing orchestrates), cheapest gates first (Part 8.3 — the deterministic wall runs before any model judgment):** plan → **`autothing-plan`**; implement → **`autothing-implement`**; the **deterministic wall** (typecheck / lint / structural greps + the universal `securityWall`: gitleaks + semgrep + dependency-audit) then the correctness test (e2e through the UI) → **`autothing-test`**; **the per-slice review** (fresh-context, Fable-pinned, evidence-mandatory — Part 8.1 merged the old same-model review into this single review) → **`autothing-adversarial-review`**, plus — CONDITIONALLY (`build` always; `feature` only on security-boundary slices; NEVER `patch` — see "Per-slice Codex adversarial pass" below) — a **cross-model Codex adversarial pass** (`codexSliceReview`, serial `codex exec`, GPT-5.5/high) ALONGSIDE it; independent test pass → **`autothing-adversarial-test`**; design audit (ui/mixed only) → **`autothing-design-audit`**; evidence → **`autothing-walkthrough`**. A gate that finds real issues sends the slice back to `autothing-implement`. Once all slices are done, the final phase (before the terminal verdict) runs the **built-in security review**, then the run-level cross-model check → **`autothing-codex-checkpoint`** (two decorrelated passes, not overlapping), then notify → **`autothing-report`** (Phase 5).
- **Also delegates (do not reimplement):** foundation detect + scaffold → **`autothing-project-foundation`**; how to parallelize → **`autothing-parallel-work`**; cross-model engine → the **`codex` CLI**, used at TWO points now: the CONDITIONAL per-slice `codexSliceReview` (build / boundary-feature, see below) AND the final-phase `autothing-codex-checkpoint` (its own `references/codex-checkpoint.md`) — both serialize on the shared OAuth/API token; run-level security review → Claude Code's built-in **`security-review`** (final phase, before codex); per-slice review engine → the fresh-context **`autothing-adversarial-review`** (the built-in **`code-review`** skill remains available STANDALONE via `autothing-review`, but is no longer a per-slice pipeline gate — Part 8.1); design engines → `frontend-design` / `huashu-design` (via `autothing-design-audit`); evidence engine → `walkthrough` (via `autothing-walkthrough`); research → `deep-research`; run/verify → `/run`, `/verify`.

## Operator gate toggles (optional parameters)
By default autothing runs the **full** gate sequence on every slice. The operator can turn individual gates OFF for a run by putting flags anywhere in the invocation args (they are stripped from the brief before planning). **Plan and implement are never disableable** — you cannot build without a plan and code. Everything else is optional:

| Flag(s) | Turns OFF | Pipeline step |
|---|---|---|
| `--no-test` / `--skip-test` | correctness test gate | `autothing-test` |
| `--no-adversarial-review` / `--no-review` | **the per-slice review** (fresh-context Anthropic, the merged review — Part 8.1) | `autothing-adversarial-review` |
| `--no-adversarial-test` | independent Anthropic test pass | `autothing-adversarial-test` |
| `--no-adversarial` | BOTH fresh-context Anthropic gates (convenience) | both adversarial skills |
| `--no-codex-slice` | the CONDITIONAL per-slice cross-model Codex adversarial pass (build / boundary-feature) | build-loop (codexSliceReview) |
| `--no-codex` | the final-phase (RUN-LEVEL) cross-model Codex checkpoint ONLY — distinct from `--no-codex-slice` | `autothing-codex-checkpoint` |
| `--no-design` / `--no-design-audit` | subjective design/UX audit | `autothing-design-audit` |
| `--no-walkthrough` / `--no-evidence` / `--no-video` | video/asciinema evidence | `autothing-walkthrough` |
| `--no-deliberate-red` | the once-per-run deliberate-red gate proof (Part 5.5; default ON for runs ≥ 3 slices) | build-loop (deliberate-red) |
| `--no-mutation` | mutation testing at the run-level phase gate (Part 9.2; default ON for runs ≥ 3 slices) | build-loop (mutation) |
| `--no-report` | the end-of-run Slack notification | `autothing-report` |
| `--no-foundation` | Phase 2/3 foundation detect + scaffold (assume the repo is already founded) | `autothing-project-foundation` |

**`--ask-questions`** (default OFF) is a planning flag, not a gate toggle: with it, planning pauses exactly ONCE after `RUN_SPEC.md` is written to let the operator answer the open decisions, then runs fully autonomously (see "Spec-first planning" below). It is recorded in `gatesConfig` so a resume knows the spec was operator-locked.

By design, no flag disables more than its own row now — `--no-adversarial-review`/`--no-adversarial-test` only ever turn off the per-slice Anthropic gates, `--no-codex-slice` only the CONDITIONAL per-slice Codex pass, and `--no-codex` only the run-level checkpoint; they no longer alias each other's scope. (`--no-review` is accepted as a spelling of `--no-adversarial-review` because the merged per-slice review is now the only review — Part 8.1 retired the separate same-model review from the pipeline.)

Two convenience forms:
- **Allowlist:** `--gates test,adversarial-review,walkthrough` (or `--only test,adversarial-review`) — run ONLY the named gates; every other gate is OFF. Names: `test adversarial-review adversarial-test codex-slice design walkthrough deliberate-red mutation codex-checkpoint`.
- **Preset:** `--minimal` (alias `--fast`) — keep only the correctness **test** gate; turn off both adversarial gates, the per-slice codex pass, design audit, walkthrough, deliberate-red, mutation, report, and the codex checkpoint. The lightest honest build (still committed + tested + the deterministic security wall, which the floor never lets go).

**Defaults require no flags.** A plain `autothing` invocation runs the full per-slice Anthropic gates on every slice AND the codex checkpoint once in the final phase — nothing above needs to be typed to get the full bar.

**Honesty is preserved, not bypassed.** A disabled gate is **recorded and printed as `off`, never as a silent pass**: its `gate-status` entry is `{"status":"skipped","reason":"operator-disabled"}`, it prints `GATE <name>: skipped (operator-disabled)`, and the terminal verdict renders that gate as `off` (e.g. `adversarialReview:off`, `codexCheckpoint:off`, `videos:0/0`) plus a `gates-disabled:<list>` token. A `passed` verdict that carries disabled gates is explicitly a **reduced bar** and says which gates did not run — it is never presented as the full-gate `passed`. The independent-verification non-negotiable below forbids *silently* skipping any of these gates; an explicit `--no-*` flag is an openly-declared, openly-recorded operator choice, which is different.

## Cross-session coordination — first-class throughout the flow (coord-mcp + agent-mail)
The operator keeps **several Claude sessions running on `main` at the same time**. This coordination stack is how autothing avoids trampling those sessions AND stays aware of them so it can steer its own work around theirs. Two MCP servers, each independently optional:
- **coord-mcp** — the per-repo **planning gate** + shared awareness: `begin_planning(repo, summary)` / `end_planning`, `plan_status`, `plan_heartbeat`, `declare_intent(repo, area, files, reason)`, `release_intents`, and **`coord_digest`** (the cross-session snapshot: who is planning, what intents/leases are in flight).
- **coord-agentmail** — per-session **identity + inbox** + **file reservations (advisory leases)**: register/identity + `ensure_project`, `fetch_inbox` / `fetch_summary` / `summarize_recent`, `send_message` / `reply_message` / `acknowledge_message`, and the reservation cycle (`macro_file_reservation_cycle` / `file_reservation_paths` / `renew_file_reservations` / `release_file_reservations` / `force_release_file_reservation`), plus `whois`, `health_check`, `install_precommit_guard`.

**Detect once, degrade gracefully.** If these MCP tools are not connected (a plain session, no Garrison), skip ALL of the below and run exactly as autothing otherwise would — the intra-run disjoint-files discipline (`autothing-parallel-work`) is then the whole contract. **Never hard-block on coordination**; a WAIT or a busy lease steers you, it never hangs the run.

**Awareness is the point, not just file locking.** Read `coord_digest` + your agent-mail inbox at the coordination touchpoints below and let what other sessions are doing *change your plan*: sequence away from files another session holds, pick a different slice, narrow your area, or send a message to negotiate. Treat another session's declared intent/lease on a file the same way you treat an intra-run shared file — **shared ⇒ serialize or avoid**, never edit under it.

Coordination touchpoints, woven through the phases:
- **Phase 0 (arm):** if the stack is connected, `health_check`, register this session's agent identity + `ensure_project` for the repo, and read `coord_digest` + `fetch_summary` **before planning** so the plan is shaped by what is already in flight. `declare_intent` the run's overall area (the slices/files you expect to touch) so other sessions see autothing arrive.
- **Phase 1 (plan):** `autothing-plan` already takes the `begin_planning` lock (honor a **WAIT** — re-check with a bounded backoff; never hang) and folds its read-bundle in. On top of that, **steer the slice plan around live intents/leases from the digest** — do not plan a slice whose files another session currently holds; if unavoidable, order it last and flag it for message-coordination. `end_planning` when the plan file is written.
- **Phase 4 (build loop), per slice, BEFORE editing its files:** claim the slice's files as an agent-mail **reservation** (`macro_file_reservation_cycle`: path + TTL + heartbeat + reason) and `declare_intent` for the slice. If the reservation/intent **collides** with another live session, treat those files as shared → serialize behind them, pick a different buildable slice, or `send_message` to negotiate ownership. During long slice work, `renew_file_reservations` + `plan_heartbeat` so the leases don't lapse. **Release the slice's reservation/intent as soon as the slice is `passed`/`blocked`** (don't hold files you're done with).
- **Every turn (with the resume scan):** poll `coord_digest` + `fetch_inbox`. Handle inbound messages — if another session announces a change or asks you to avoid a path, `acknowledge_message` and steer (re-order, avoid, or reply to coordinate). Add coordination state to the per-turn ledger, e.g. `COORD: leases <held> · digest-conflicts <n> · inbox <unread>`.
- **Phase 5 (end):** `release_file_reservations` + `release_intents` + `end_planning`, then **`send_message` a short completion summary** to the project channel (what landed on `main`, which files, the commit range) so other sessions can adjust to autothing's changes. `autothing-report`'s Slack notification is for the operator; this mail summary is for the other sessions.
- **Optional hardening:** `install_precommit_guard` so a commit that touches another session's reserved files is caught before it lands on `main`.

Precedence: the intra-run disjoint-files + serialize-shared-runtime rules ALWAYS hold; the coord stack ADDS cross-session safety and awareness on top. Full mechanics + the agent-teams-vs-workflows decision live in **`autothing-parallel-work`** (its "Cross-session coordination" section) — this section is autothing's contract to actually USE them at every phase, not just during a parallel fan-out.

## Workflow

### On invocation — Phase 0: self-install, then arm the goal loop
**Step A — ensure autothing is wired into this Claude config (self-install).** Before anything else, run the idempotent installer so the goal loop works even on a machine/config where it was never set up:
```bash
bash ~/.claude/skills/autothing/hooks/install.sh
```
It is a no-op when already configured; otherwise it adds the `Stop` + `SessionStart` hooks and `CLAUDE_CODE_STOP_HOOK_BLOCK_CAP` to `settings.json` (idempotent, non-duplicating, backs up to `settings.json.autothing.bak`) and makes the hook scripts executable. **Capture WHICH it reported — this decides whether a `/goal` is needed:**
- **`already configured`** → the hook was in `settings.json` at session start, so it is **active now** and will drive the loop autonomously. **No `/goal` — do not prompt for one.**
- **`installed/repaired`** → the hook was added THIS session. A freshly-added hook does **not** take effect until a new session (hooks load at session start), so THIS run cannot auto-loop yet — it needs a **one-time** `/goal` (or a session restart). This is a bootstrap-only situation, not the normal path.

Either way **carry on with the run — never stop to ask.** If the installer warns that `disableAllHooks`/`allowManagedHooksOnly` is set, or that it cannot write `settings.json` (no `jq`), the hook cannot drive the loop → use the `/goal` fallback. (`install.sh --check` reports status without writing.)

**Step B — pick a run id + run directory (per run — never a shared path).** Concurrent autothing runs (in other sessions, even the same repo) must not clobber each other, so ALL of this run's durable state lives under a unique per-run directory:
- `SID="$CLAUDE_CODE_SESSION_ID"` (always set in the Claude Code shell — this is THIS session's id, and the same id the Stop hook receives).
- `runId="<YYYYMMDD-HHMMSS>-${SID:0:8}"` (timestamp + short session suffix — unique across concurrent sessions). `runDir="docs/autothing/runs/<runId>"`. `mkdir -p "<runDir>/slices"`.
- This run's plan is `<runDir>/FLOW_PLAN.md`; its evidence index is `<runDir>/evidence-index.json`; per-slice gate-status is `<runDir>/slices/<slice>/gate-status.json`. **Do not write to a shared `docs/FLOW_PLAN.md`.**
- **Durable run record + liveness (full mechanics in "Durable run record" below).** Create/append **`RUN_LOG.md` at the repo root** with a `RUN-START` entry (runId, one-line brief summary, session model + effort, gatesConfig, `profile` [or `pending-sizing` until Phase 1 assigns it], host — **timestamps ONLY via `date -u +%Y-%m-%dT%H:%M:%SZ`**, never model-composed). Write the **ownership lock** `<runDir>/owner.lock.json` ({sessionId, host, pid, heartbeatAt} — this session claims the run) and the first **`<runDir>/status.json`** heartbeat. Both are gitignored (Part 2.3).
- **Preflight doctor (Part 12.4).** Check the tool inventory this run needs (asciinema, agg, codex, gitleaks, semgrep, playwright-cli, ffmpeg, the project's test-runner binaries), record versions in the RUN-START entry, and read `docs/autothing/known-flakes.md`. Missing tools are installed (self-unblock) or logged as a `DECISION` before any slice starts — environment failures found at minute zero, not mid-gate.

**Step C — arm the goal loop.** Pick a turn cap (default **250**; scale to ≈20–30 turns × expected slices for large builds). Write the **per-session** sentinel so the `Stop` hook (`hooks/goal-stop.sh`) loops the session to completion with **no manual `/goal`**:
- `mkdir -p ~/.autothing/sentinels`; write `~/.autothing/sentinels/$SID.json` (keyed by session id, so concurrent sessions never collide):
  ```json
  { "runId": "<runId>", "runDir": "<absolute runDir>", "project": "<project>", "turnCap": 250, "iteration": 0, "armedAt": "<iso>", "gates": { "test": true, "adversarialReview": true, "adversarialTest": true, "codexSliceReview": true, "design": true, "walkthrough": true, "deliberateRed": true, "mutation": true, "report": true, "foundation": true, "codexCheckpoint": true }, "condition": "loop until this run's terminal GLOBAL GATE line (with (run <runId>) + videos:<n>/<n>) prints" }
  ```
  (`turnCap` is the default here; Phase 1 UPDATES it to the plan-derived `max(300, 80 × slices)` and logs the resize as a `DECISION` — see "Phase 1". With `--ask-questions`, DEFER writing this sentinel until the operator's answers arrive, so the goal loop does not fight the one planned pause — Part 10.2.)
  The hook releases the session the moment THIS run's terminal `GLOBAL GATE: … (run <runId>) … videos:<n>/<n>` verdict appears in the transcript (matched by `runId`, so stray/quoted/prior verdicts can't false-release), OR at the turn cap; it then deletes the sentinel. **Resume:** in the same session (incl. after compaction) read `runDir` from `~/.autothing/sentinels/$SID.json`; in a fresh session resuming an interrupted run, pick the newest `docs/autothing/runs/<runId>/` whose `evidence-index.json` globalGate is not `passed`, and re-arm a sentinel for the new session pointing at it — after first confirming no other live session's sentinel/lock owns that run (one session owns a run), and verifying the interrupted run's markers before trusting them. **The full protocol is the "Resume mode" section below (the single source);** in brief: re-run the last passed gate's checks on THIS machine; uncommitted working-tree changes from the interrupted session are untrusted scratch; record the resumption.

**Step B.5 — parse the operator gate toggles (see "Operator gate toggles" above).** Scan the invocation args for the `--no-*` / `--gates` / `--only` / `--minimal` / `--ask-questions` / `--no-deliberate-red` / `--no-mutation` / `--no-codex-slice` flags, build a `gates` config object (each of `test adversarialReview adversarialTest codexSliceReview design walkthrough deliberateRed mutation report foundation codexCheckpoint` → `true`=run | `false`=off; default all `true`, plus `askQuestions` default `false` and `profile` — a `patch|feature|build` override if the operator forced one, else `pending-sizing` until Phase 1 assigns it), and **STRIP the flags from the brief** before Phase 1 planning (they are directives, not part of the goal). Note (Part 8.1): the old same-model `review` key is gone — the per-slice review is now the single `adversarialReview`; `--no-review` maps onto `adversarialReview:false`. `deliberateRed` + `mutation` default ON only for runs ≥ 3 slices (Phase 1 sizing decides), OFF below — an operator `--no-deliberate-red`/`--no-mutation` forces OFF regardless. `codexSliceReview` defaults ON but RUNS conditionally (profile `build` always; `feature` only on security-boundary slices; NEVER `patch` — see "Per-slice Codex adversarial pass"); `--no-codex-slice` forces it OFF entirely. Persist the config so a resumed run in a fresh session honors the same choices:
- add `"gates": { … }` to the sentinel JSON in Step C (below), and
- add the same `"gatesConfig": { … }` to `evidence-index.json` when you first write it (Phase 4).
On resume, read `gatesConfig` from `evidence-index.json` (fallback: the sentinel's `gates`) rather than re-parsing — a compacted session no longer has the original args. If NO flags were passed, `gatesConfig` is all-`true` (the normal full-gate run) — record it explicitly so a reader sees the full bar was in force.

**Step D — finish Phase 0 per the Step-A signal. DO NOT prompt for `/goal` when the hook is already active.**

- **Hook already active** (Step A said `already configured`) — the normal path. The Stop hook drives the loop autonomously. ## Durable run record — RUN_LOG, ownership lock, status heartbeat (Parts 1, 2)

**`RUN_LOG.md` (repo root, append-only, versioned).** Created/appended at Phase 0, **never rewritten**. One journal per repo; each run is delimited by its `RUN-START`. Entry types: **`RUN-START / GATE / DECISION / AMBIGUITY / DEVIATION / ABORT / RESUME / PAUSED`**. Every timestamp comes ONLY from `date -u +%Y-%m-%dT%H:%M:%SZ` — never model-composed (mixed local/UTC labels read as fabrication in audits).
- **RUN-START** records: runId, one-line brief summary, session model + effort, gatesConfig, profile, host, preflight tool versions.
- **GATE** (one per gate): green condition, evidence pointer, review verdicts **with the actual model each ran on**, ledger update, checkpoint commit + tag, and **wall-clock duration + the model each pipeline step ran on** (Part 12.1 — token counts are NOT reliably readable in-session; never fabricate them, duration + model only).
- **DECISION / AMBIGUITY / DEVIATION** record autonomous choices made on the operator's behalf, unresolved ambiguities steered past, and any deviation from plan.
- **ABORT / PAUSED / RESUME** mark run lifecycle transitions (see Resume mode + the graceful turn-cap landing in build-loop.md).
If a project spec mandates its own journal discipline, MERGE into this one RUN_LOG.md (the spec's format wins; autothing's entry types map onto it). **Never two journals.**

**Ownership lock (Part 2.1).** `<runDir>/owner.lock.json` `{sessionId, host, pid, heartbeatAt}` — gitignored, written at start/resume, **heartbeat refreshed every turn** with the per-turn ledger. A session that finds a FRESH lock it does not own MUST NOT touch the run — log and stop. One session owns a run, always. A stale lock (heartbeat > 10 min) is claimable (see Resume mode).

**Status heartbeat (Part 2.2).** `<runDir>/status.json` `{slice, gate, lastAction, at (UTC), model, iteration}` — gitignored, rewritten every turn. **This is THE liveness check for the operator: `cat <runDir>/status.json`** — use it INSTEAD of opening a second session "to see how it's going" (a second session against the same run is exactly how collisions happen).

**Git hygiene (Part 2.3 — enforced by `autothing-project-foundation` at Phase 0/3).** An idempotent marker block (`# >>> autothing >>>` … `# <<< autothing <<<`) in the target repo's `.gitignore` IGNORES evidence binaries + liveness state (`**/*.cast`, `*.webm`, `*.mp4`, `*.png`, screenshot dirs, `status.json`, `owner.lock.json`, workflow/cache dirs under `.claude/`) and KEEPS VERSIONED the small resume-critical files (`FLOW_PLAN.md`, `evidence-index.json`, `gate-status.json`, `RUN_LOG.md`, `RUN_SPEC.md`, `LANDING.md`) — cross-machine resume depends on exactly these; **never exclude the whole runs dir.** Because the binaries are unversioned, `evidence-index.json` carries `videoSha256` + `videoBytes` per artifact so the evidence stays auditable from the versioned index.

## Resume mode — first-class (Parts 1.2 / 1.3)
Triggered by `--resume`, "resume the run", "continue the autothing run", "pick up where it left off", or automatically on any invocation that finds an unfinished run it owns. **This is the SINGLE SOURCE for resume; build-loop.md's "Resume FIRST" points here.**
- **Discovery.** An arg-named runId wins; else the newest `docs/autothing/runs/<runId>/` whose `evidence-index.json` globalGate ≠ `passed`. Multiple unfinished runs → newest, plus an `AMBIGUITY` entry listing the rest.
- **Ownership.** No live sentinel or FRESH `owner.lock.json` (heartbeat ≤ 10 min) may belong to another session — if one does, the run is owned; **log and stop, do not touch it.** A stale lock (heartbeat > 10 min) is claimable — the takeover is logged in the `RESUME` entry.
- **Ground truth over memory.** Trust committed git (tags, checkpoint commits) + RUN_LOG over any in-context memory or a single "passed" marker. **Re-run the preflight doctor (Part 12.4)** and **the last passed gate's suite on THIS machine** before building forward. Uncommitted working-tree changes from the interrupted session are untrusted scratch — verify or discard, log which. Infra/environment failures are NOT red gates — fix infra, log a `DECISION`, re-run.
- **Re-arm.** Write a fresh sentinel for this session pointing at the runDir; inherit `gatesConfig` from `evidence-index.json`. **Mid-run compatibility (in-flight runs must survive this upgrade):** a pre-upgrade `gatesConfig` in the old shape maps onto the new structure — old `review`+`adversarialReview` keys → the merged review gate; missing new keys → their defaults — and the run CONTINUES under the gate structure its FLOW_PLAN recorded (a project spec's gate template always wins over skill defaults; profiles apply to NEW runs only, so an in-flight run keeps the shape its planning chose). Resume at the first non-green slice; never redo green gates; never re-plan unless FLOW_PLAN is missing/corrupt (log it, don't silently re-plan).
- **RESUME entry** logs: what was verified green, what was discarded, session model (noting any model handoff), resumption point.

**Post-compaction = mini-resume (Part 1.3).** A compaction inside the SAME session triggers the same ground-truth re-read (sentinel → status.json → RUN_LOG tail) before the next action. Durable state is authoritative over reconstructed context.

## Model honesty — fallback logging + classifier redirects (Part 3)
Whenever a gate's pinned model is replaced (rate limit, credit exhaustion, capacity fallback, or a safety-classifier redirect): `gate-status` records `actualModel` + `fallbackCause`; a `DECISION` entry names which model replaced which and why; the terminal verdict carries a `model-fallbacks:<n>` token when n > 0. **A degraded review is recorded as degraded, never presented as the pinned bar.** Policy by cause — **one retry maximum** (looping a deterministic cause burns budget for the same answer):
- **Classifier redirect** (content-deterministic — adversarial security reviews, PII/anonymisation test content, and exploit-shaped regression cases can false-positive frontier safety classifiers, which *redirect* to a fallback model rather than refuse): retry ONCE on the pinned model immediately; if it repeats, accept the fallback and record `degraded (classifier-redirect)`. Redirects are per-request and bounce back on innocuous turns.
- **Capacity fallback** (time-deterministic, e.g. HTTP 529 overloaded): retry the pinned model ONCE after a 2–5 min backoff (do useful work or active-wait meanwhile); if still overloaded, accept and record `degraded (capacity)`. Capacity/limit fallbacks are sticky.
Distinguish the causes in the `DECISION` entry, and notify the operator the moment a fallback happens (naming the cause) via `autothing-report` (Part 6.2).

## Per-slice Codex adversarial pass — conditional cross-model gate (`codexSliceReview`)
Evidence (run `20260706` RUN_LOG: per-slice Codex caught real Criticals at G4/G7/G7A/G7B/G8 under diff-scoped rubrics) reinstates a **per-slice cross-model Codex adversarial pass that runs ALONGSIDE** the fresh-context `autothing-adversarial-review` — it does NOT replace it, and it is distinct from the run-level `autothing-codex-checkpoint` (which is UNCHANGED). It runs only where it earns its cost.

**When it runs (conditional):**
- **`build` profile:** ALWAYS, on every slice.
- **`feature` profile:** ONLY on slices whose diff touches security-boundary code (the Part 11.3 trigger: auth, crypto, egress, parsing, path containment).
- **`patch` profile:** NEVER.
- Off entirely with `--no-codex-slice`. A slice that does not run it by the conditional records `{"status":"skipped","reason":"profile-conditional"}`; operator-disabled records `{"status":"skipped","reason":"operator-disabled"}` — visible, never silent.

**Mechanics:** one `codex exec` call, **GPT-5.5 at high reasoning** (`-m gpt-5.5 -c model_reasoning_effort=high`), **SERIAL** (all `codex exec` calls serialize run-wide — concurrent calls revoke the shared token). Diff-scoped **narrow rubric**: the slice's invariants + acceptance + the diff, prompt "find violations" (never an open-ended review). Print `CODEX CALL: gate=codexSliceReview model=… effort=high slice=… diff=[<shortstat>]` in the lead context before the call. Record the verdict in gate-status as `codexSliceReview: {verdict, by, actualModel, durationMs}`. A **`needs-work`** verdict (a real, agreed violation) loops the slice back to `autothing-implement` within the slice's EXISTING 5-attempt ceiling (no separate budget) — Claude is the deciding authority: apply only findings it independently agrees are real, material defects (record a one-line rebuttal otherwise).

**Credit / availability death — never block a run on a dead meter, never fake the verdict (Part 3 honesty).** If `codex exec` fails on quota / auth / availability mid-run: record the gate slot **degraded** — `codexSliceReview: {status: "degraded", reason: "codex-unavailable", actualModel: null}` — emit the mid-run notification (Part 6.2, cause `codex-unavailable`), log a `DECISION`, and **CONTINUE**. The slice still has its fresh-context `adversarialReview`; a degraded codex slot does NOT fail the slice and does NOT fake an `approve`. This is the G8 credit-death failure mode made survivable.

**Authenticate via API KEY, not ChatGPT sign-in.** autothing's unattended Codex usage MUST authenticate with a **Codex/OpenAI API key** (do NOT `codex login` with the interactive ChatGPT account) so an unattended run does not drain the interactive account's shared **5-hour / weekly** usage pool — the exact mechanism behind the G8 credit-death. Set a **budget cap** on the key so a runaway loop cannot exhaust the account. Full auth + budget mechanics: `autothing-codex-checkpoint/references/codex-checkpoint.md`.

### Phase 4 — Build loop
Per `references/build-loop.md` (which is the single source for the resume protocol — see "Resume mode"). Resume from durable files first (FLOW_PLAN + gate-status + evidence-index, incl. `gatesConfig`), then for each non-done slice run the per-slice pipeline of standalone skills, **cheapest gates first (Part 8.3 — no model-judgment gate runs on a slice that has not passed typecheck, lint, structural greps, the `securityWall`, and its committed tests):** **`autothing-implement`** → **`autothing-test`** (the deterministic wall + the correctness gate) → **`autothing-adversarial-review`** (THE per-slice review — fresh-context, Fable-pinned, evidence-mandatory; Part 8.1 merged the old same-model `autothing-review` into this one) → **`autothing-adversarial-test`** (independent Anthropic test pass) → **`autothing-design-audit`** (ui/mixed only) → **`autothing-walkthrough`** (ui/mixed; `api` slices use asciinema) → write `gate-status.json` + upsert `evidence-index.json` (under `<runDir>`). **Kind-conditional gates (Part 8.2):** design-audit + walkthrough run for `ui`/`mixed` only; per-slice `autothing-adversarial-test` runs for `ui`/`mixed`, while `api` slices get ONE batched run-level independent-test pass over the API surface. A once-per-run **deliberate-red** proof (Part 5.5, runs ≥ 3 slices) and a run-level **mutation** gate (Part 9.2) round out the structural coverage. **Coordination (when the stack is connected):** before editing a slice's files, claim them as an agent-mail **file reservation** + `declare_intent` and check `coord_digest` for collisions with other sessions on `main` — a collision means those files are shared, so serialize behind them, pick a different buildable slice, or `send_message` to negotiate; renew leases + `plan_heartbeat` during long slices; release them the moment the slice is `passed`/`blocked` (see the "Cross-session coordination" section). **Parallelize by default** where slices own disjoint files — **agent teams when enabled, else dynamic workflows** — and serialize only the shared runtime (one dev-serve / bundle / recorder). **ALL `codex exec` calls serialize run-wide** — the CONDITIONAL per-slice Codex pass (`codexSliceReview`, build/boundary-feature) AND the Phase-5 `autothing-codex-checkpoint` both drive Codex, and concurrent calls revoke the shared token, so never overlap a per-slice pass with the checkpoint or with each other (see that skill + "Per-slice Codex adversarial pass"). Decompose at plan time to EARN parallelism; log the parallel-vs-serial choice. See the **`autothing-parallel-work`** skill — incl. its cross-session rules (claim each unit's files via agent-mail leases + `declare_intent` when the coord stack is present; fall back to disjoint-files when absent). **Before the first parallel batch, preflight the mechanism:** probe once that agent-team creation does NOT raise an approval prompt under auto mode (a trivial throwaway team that proceeds, or the known approval behavior); if it would block a headless run, fall back to dynamic workflows for the whole run — an approval prompt that stalls unattended is itself a silent killer of autonomy.

### Phase 5 — Security review + Codex checkpoint, global gate + handover
Per `references/build-loop.md`. **A verdict line MAY NOT be printed while `buildable-remaining > 0`** — `buildable-remaining` = count of slices whose status is neither `passed` nor `blocked` (and `blocked` only counts when it carries the attempted-remediation evidence from the non-negotiables). If it is > 0, the run returns to the per-slice loop and builds the next buildable slice.

Once `buildable-remaining == 0`, run the two run-level security passes IN ORDER (two decorrelated passes, not two overlapping ones):
1. **Built-in security review (Part 12.6).** Run Claude Code's built-in `security-review` ONCE over the finished repo, BEFORE the codex checkpoint — so the cross-model pass adversarially checks what the built-in review already cleaned. Profiles: `build` always; `feature` when security-boundary surfaces were touched; `patch` never reaches it un-escalated (a boundary patch escalates per Part 11.4). Findings triage like any gate — real ones loop to `autothing-implement`; record `builtinSecurityReview: {status, findings, at}` in `globalGate` and print `GATE security-review: <clean|issues-fixed|issues-open|skipped> — <summary>`.
2. **Codex checkpoint.** Invoke **`autothing-codex-checkpoint`** ONCE (skip only if `gatesConfig.codexCheckpoint === false`) — the run-level cross-model check over the security-critical scopes (whole-repo security, the `shared/` contract, the anonymisation/egress pipeline, auth middleware, plus anything the brief named). Real findings loop the affected scope back to `autothing-implement` as an ad-hoc fix (bounded by that skill's own 2-recheck ceiling per scope, not the per-slice 5-attempt ceiling). Record `codexCheckpoint` in `evidence-index.json`'s `globalGate` and print `GATE codex-checkpoint: <clean|issues-fixed|issues-open|skipped(operator-disabled)> — <summary>`.

Then decide the terminal state and write `globalGate.status` to `evidence-index.json`:
- **`passed`** only when **buildable-remaining == 0**, zero blockers, and **every ENABLED gate is green**: full e2e + `/verify` + build/typecheck/lint exit 0 and the `securityWall` (gitleaks/semgrep/dep-audit) clean (if `test` on), a clean fresh-context `approve` + independent-test `pass` on every slice (if the adversarial gates are on), a clean `codexSliceReview` on every slice that ran it (if `codexSliceReview` on; a `degraded (codex-unavailable)` slot does NOT block `passed` but is named in the LANDING packet), a clean design audit (if `design` on), every slice's video `verified` (if `walkthrough` on), the **built-in security review** `clean`/`issues-fixed`, AND the codex checkpoint is `clean` or `issues-fixed` (if `codexCheckpoint` on). **Gates the operator disabled (`gatesConfig.<gate> === false`) are excluded from this check** — they render as `off` in the verdict, never as a pass, and their presence makes the `passed` an explicitly **reduced-bar** pass (the verdict's `gates-disabled:<list>` token names them). A full-bar `passed` is `gates-disabled:none`.
- **`completed-with-blockers`** legitimate ONLY when **buildable-remaining == 0** AND (≥1 slice is `blocked` with a named external cause + the exact remediation command that failed) OR the built-in security review / codex checkpoint ended `issues-open` (a real, agreed violation survived its recheck ceiling). (A `failed-but-unblocking` or `evidence-degraded` video is recorded evidence but is NOT self-verified, so it keeps the gate out of `passed`.)

Never repeat failed work unproductively, but ALWAYS continue while any buildable (non-blocked) slice remains. The verdict is printed only when buildable work is exhausted — reaching the turn cap with buildable work left is a failure of the loop, not a stop condition. Never fake `passed`.

Then, in order: **(0)** if the coord stack is connected, **release** this run's file reservations + intents + `end_planning`, and `send_message` a short completion summary to the project channel (what landed on `main`, which files, the commit range) so other sessions can adjust; **(1)** write **`<runDir>/LANDING.md`** — the run's own audit packet (Part 12.2): the gates summary table, every `DEVIATION`/`DECISION`/`AMBIGUITY`, the assumptions-ledger status (RUN_SPEC runs), model-fallback events, the friction-log items appended this run, and a "needs human eyes" list. The operator's post-run review starts from this one file; `autothing-report` links it. **(2)** print the handover as prose — stack + why, install/invoke, evidence-gallery URL, an explicit enumeration of blockers (each with its external cause + failed remediation command), **which gates were operator-disabled**, and known limits; **(3)** invoke **`autothing-report`** to send the operator a Slack notification (work summary + the walkthrough video-gallery Tailscale URL + the `LANDING.md` link + Tailscale links to the run's logs/artifacts, served in place) — **skip this step when `gatesConfig.report === false`**. This is a side-effect, not a transcript print, so it is safe before the verdict line; **(4)** as the LAST action, print the terminal verdict line (exact format in `references/build-loop.md` — it carries `(run <runId>)` + `videos:<n>/<n>` so the goal-loop hook matches THIS run, plus a `gates-disabled:<list|none>` token, a `profile:<name>` token, a `codexSlice:<approved>/<ran|off>` token, and a `model-fallbacks:<n>` token when n>0; disabled gates render as `off`, e.g. `adversarialReview:off videos:0/0 codexSlice:off`). The handover comes first because the verdict line is the completion signal: the `/goal` evaluator clears on it and the session stops, so nothing after it runs. Don't put an assertive `GLOBAL GATE:` line in the handover. (The same token in the invocation handoff is a quoted target, not a verdict; the evaluator distinguishes the two.)

## Durable markers — the contract
Every gate (objective AND subjective) leaves **two** traces:
- **A durable file** — so the build resumes after compaction and the human can inspect it:
  - per slice → `<runDir>/slices/<slice>/gate-status.json` (schema: `assets/gate-status.example.json`)
  - per run → `<runDir>/evidence-index.json` (schema: `assets/evidence-index.example.json`) — `<runDir>` is `docs/autothing/runs/<runId>/` from Phase 0, never a shared path
  - the run journal → `RUN_LOG.md` (repo root, append-only), plus `<runDir>/RUN_SPEC.md` (spec-first runs) and `<runDir>/LANDING.md` (written when the verdict prints); all versioned. Liveness (`status.json`, `owner.lock.json`) is gitignored (Part 2.3).
- **A printed line in the transcript** — because the `/goal` evaluator is a small fast model that reads ONLY the conversation; it does not open files or run commands. Print each gate's exit code inline (`GATE <name>: exit <code> — <summary>`), and as the **last line of the run (after the handover)** print the final `GLOBAL GATE: <status> (run <runId>) — <checklist>` block (the `(run <runId>)` tag binds it to this run so the goal-loop hook matches only THIS verdict). The evaluator can confirm only what autothing has surfaced in the transcript.
- **A per-turn progress ledger** — at the end of EVERY turn, after the resume scan, print one line: `PROGRESS: passed <p>/<total> · blocked <b> (remediation-attempted) · buildable-remaining <r>`. This makes exhaustion (or its absence) visible to the transcript-only evaluator. A `GLOBAL GATE:` verdict may NOT be printed while `buildable-remaining > 0`. **Also every turn:** refresh the `owner.lock.json` heartbeat + rewrite `status.json` (Part 2.2), so the operator can `cat status.json` for liveness instead of opening a second session. **When the coord stack is connected, append a coordination ledger too** — `COORD: leases <held> · digest-conflicts <n> · inbox <unread>` — after polling `coord_digest` + `fetch_inbox`, so cross-session state (and any steering it forced) is visible in the transcript.
- **Active waiting — never end an empty turn (Part 5.7).** When the lead has no buildable action (workers/subagents mid-flight, a long external wait), it waits ACTIVELY: run a bounded `sleep` (30–90s) between status checks so every turn contains a real action, then check worker output and end the turn. Ending an empty turn under the goal loop produces a stop-block busy loop — the hook re-blocks instantly, the iteration counter freezes, and blocks burn against the cap doing nothing.
- **Context pressure — seek a boundary before compaction hits.** When session context is visibly high (~75%+ of the window), **prefer landing the current slice at its next gate and flushing durable state** (append the pending `RUN_LOG` entries, write `gate-status.json` + upsert `evidence-index.json`, rewrite `status.json`) **over starting new sub-work** — so if compaction fires it lands on a gate boundary, not mid-edit, and durable state is already the source of truth. Do not start a fresh slice, a parallel fan-out, or a long uncommitted edit under high context; finish or cleanly park what is in flight (a `PAUSED` entry if you must stop mid-slice, per the graceful turn-cap landing in build-loop.md). After compaction, the **post-compaction mini-resume (Part 1.3)** applies as specced: re-read sentinel → `status.json` → RUN_LOG tail before the next action, durable state authoritative over reconstructed context.
- **Mid-run notifications (Part 6.2, via `autothing-report`).** Notify the operator immediately on: `ABORT`, a model-fallback (naming the cause — limit / capacity / classifier-redirect), `PAUSED` (turn-cap landing), a blocked slice, and a detected busy-loop (repeated stop-blocks without progress). Optional `--notify-gates` adds a one-liner per gate PASS. The operator watching from Slack then never opens a second session — and learns of a mid-run degradation the second it happens.

Never claim a gate passed without both its file trace and its printed line.

## Non-negotiables
- **Idempotent + non-clobbering** (enforced by `autothing-project-foundation`): missing → create; existing canonical file → never autonomously rewrite.
- **Autonomous**: never pause; fix forward; append blockers to `docs/decisions.md` and continue. A `walkthrough` STUCK/ask-user return becomes `video.status: failed-but-unblocking` + a logged blocker — never a wait for input.
- **No voluntary deferral.** A slice may end the run only as `passed` or `blocked`. "Deferred", "consolidated later", "interim proof", and "asserted-not-lived by choice" are NOT terminal states and are FORBIDDEN as blocker reasons. If work is buildable, it must be built before any verdict is printed.
- **Self-unblock before blocking.** Before marking any slice `blocked` on a missing tool, dependency, or binary, attempt to install it ONCE using the ecosystem's standard command (`uvx` / `pipx` / `brew` / `npm i -g` / `pip --break-system-packages` / `cargo install`). Only an install that FAILS and genuinely requires operator credentials or hardware the operator lacks is a legitimate blocker — and the blocker line must name the exact failed command and its error. "Not installed" alone is never a blocker.
- **Honest**: a failing slice is shown failing and flagged, never edited to look passed.
- **Independent verification is real and never silently skipped.** Every slice gets a fresh-context review and an independent evidence-based test pass (decorrelated by context and evidence, not vendor — `autothing-adversarial-review` / `autothing-adversarial-test`); CONDITIONALLY (build always; boundary-feature) every slice ALSO gets a cross-model Codex adversarial pass (`codexSliceReview`, see "Per-slice Codex adversarial pass"); and every run gets, in the final phase, Claude Code's built-in security review followed by a targeted cross-model checkpoint (Codex, high effort) over the security-critical scopes (`autothing-codex-checkpoint`) — two decorrelated passes, the cross-model one adversarially checking what the built-in review already cleaned. A disabled gate is recorded as `off`, never a silent pass (`--no-adversarial-review` / `--no-adversarial-test` / `--no-codex-slice` / `--no-codex`, recorded openly as `adversarialReview:off` / `adversarialTest:off` / `codexSlice:off` / `codexCheckpoint:off`). **All `codex exec` calls serialize** (one at a time, run-wide — concurrent calls revoke the shared token; Codex is called at TWO points now — the conditional per-slice `codexSliceReview` and the run-level checkpoint). A Codex quota/auth death is recorded `degraded (codex-unavailable)`, never blocks the run, and never fakes a verdict (Part 3). If Codex is missing, self-unblock the install; if `codex login` genuinely fails, that is an external blocker for the checkpoint — log it and let the global gate fall to `completed-with-blockers`, never report a full-bar `passed` without it. The prohibition is on a gate *disappearing unrecorded*; an explicit operator toggle is a declared, logged choice, not a silent skip. Full recipes: `autothing-adversarial-review`/`autothing-adversarial-test` SKILL.md (fresh-context mechanics) and `autothing-codex-checkpoint/references/codex-checkpoint.md` (the Codex mechanics, superseding `references/codex-verification.md` for cross-model duty — that file is retained for history, marked superseded).
- **The correctness gate is a COMMITTED, re-runnable assertion** — a test file, or (where no runner ships, e.g. a Cortex/artifact bundle) a committed `playwright-cli` driver script that re-drives the flow and asserts. Ephemeral `.playwright-cli/` logs and the walkthrough video are EXPLORATION/EVIDENCE, never the gate. No committed re-runnable assertion ⇒ the slice is not done. (Resolves the tension with the `e2e-testing` skill: drive with playwright-cli, but COMMIT the driver + assertions so later slices catch regressions.)
- **Foundation + git before build** — Phase 3 via `autothing-project-foundation` (lean CLAUDE.md, the docs, area skills, `git init`) is a hard prerequisite for Phase 4; a gap is filled or logged with a reason, never silently skipped.
- **Workers/teammates get explicit context** — they do not inherit the lead's history; paste the relevant skill/doc + acceptance + file-ownership boundary into each spawn prompt.
- **Record build friction as a signal — autothing never self-edits.** At every gate boundary and at run end, append one line per observed friction to `docs/autothing/friction-log.md` — where the SKILL's OWN instructions were ambiguous, wrong, wasteful, or fought the situation (never feedback about the product being built). **Format (Part 6.1):** `- <UTC> [<skill>] <what happened> → <suggested prose fix>` — UTC-stamped (`date -u`), terse, one line each. autothing does NOT act on it and never edits any skill; the nightly `skill-improver` is the single mechanism that improves skills, and it reads this log as its main improvement signal (autothing's unattended builds have no user present, so the log IS that signal — close the loop from the producer side).
- **The determinism ratchet (Part 9.1).** Every accepted review finding must, where the class of bug is mechanically expressible, ship WITH a deterministic guard in the SAME fix — a lint rule, a Semgrep pattern, a grep gate, or a test — so that class is machine-caught forever and reviews stop re-finding it. The fix commit references the guard; the gate entry records finding → guard. Reviews trend toward judgment-only over the life of a project; that is the goal.
- **Verify every new dependency before install (Part 12.3).** Every NEW package an agent adds is checked first — it exists on the registry, is actively maintained, and is the intended package (not a typo/slopsquat neighbor). Agent-written code makes hallucinated dependencies an attack surface; one lookup per new dependency closes it. The check is noted in the slice's gate entry. (Enforced in `autothing-implement`.)
- **The floor never lowers (Part 11.3).** No profile, kind-skip, or `--no-*` toggle ever removes typecheck, lint, structural greps, the `securityWall` secrets scan, or committed tests; any diff touching security-boundary code (auth, crypto, egress, parsing, path containment) gets the fresh-context review regardless of profile. Every profile-/kind-skipped gate is recorded `{"status":"skipped","reason":"profile:<name>"|"kind-conditional"}` — visible, never silent.

- `hooks/probe.sh` — goal-loop **liveness probe** (`arm` → end the turn → `check`): confirms Claude Code actually honors the Stop hook's `decision:block` auto-continue in a session (proves the no-`/goal` loop works end-to-end) using the real `goal-stop.sh`. Self-bounded (cap 2), refuses to clobber a live run. Run it in a session that is NOT executing a build, ideally before the first long unattended run on a machine.

## Delegated skills (separate skills — invoke, never reimplement; each is usable on its own)
The per-slice pipeline (each a standalone `autothing-*` skill autothing orchestrates; a failing gate loops back to `autothing-implement`):
- **`autothing-plan`** — Phase 1 planning: reproduces Claude Code plan mode (read-only `Explore` + `Plan` subagents, then a concise durable plan) WITHOUT the native `EnterPlanMode`/`ExitPlanMode` tools. Writes the run's `<runDir>/FLOW_PLAN.md` (never a shared path).
- **`autothing-implement`** — the code-writing step: explore (vision-first for UI) then write the slice. The step every gate loops back to.
- **`autothing-test`** — the correctness gate: a COMMITTED re-runnable **e2e-through-the-UI** test (Playwright / playwright-cli) + unit tests + `typecheck`/`lint`/`build`. CLI/TUI deliverables use a committed driver + asciinema capture.
- **`autothing-adversarial-review`** — **THE per-slice review** (Part 8.1): fresh-context, Fable-pinned, evidence-mandatory Anthropic review (no implementer context, gathers its own build/typecheck/lint/test evidence, every finding cites a spec/acceptance section or attaches failing output).
- **`autothing-review`** — same-model review via the built-in `code-review` skill. **No longer a per-slice pipeline gate** (Part 8.1 merged it into `autothing-adversarial-review`); retained as a STANDALONE skill.
- **`autothing-adversarial-test`** — independent Anthropic functional pass (fresh context, self-written Playwright probes). Per-slice for `ui`/`mixed`; ONE batched run-level pass for `api` slices (Part 8.2).
- **`autothing-design-audit`** — subjective design/UX gate (ui/mixed only, Part 8.2) via `frontend-design` / `huashu-design`.
- **`autothing-walkthrough`** — self-verified evidence via the `walkthrough` skill in `mode: evidence` (Part 4); reconciles a STUCK to `failed-but-unblocking`.
- **`autothing-codex-checkpoint`** — Phase 5, run once AFTER the built-in security review: the genuine cross-model check via the `codex` CLI, high effort, narrowly scoped to the security-critical surfaces (whole-repo security, `shared/` contract, anonymisation/egress pipeline, auth/session).
- **`autothing-report`** — Phase 5 final step: Slack notification (work summary + the walkthrough video-gallery Tailscale URL + Tailscale links to the run's logs/artifacts, served in place via a small standing Node server; no duplication). Sends via a Slack incoming webhook, falls back to the Slack MCP. Standalone too.

Foundation + support:
- **`autothing-project-foundation`** — Phase 2 detect + Phase 3 scaffold: manifest/idempotency, /docs, lean CLAUDE.md, area skills, `git init`. Carries the doc/CLAUDE.md/area-skill assets.
- **`autothing-parallel-work`** — Phase 4 parallelization: agent teams vs workflows vs serial, disjoint-file decomposition, what must serialize, AND cross-session coordination (coord-mcp planning gate / agent-mail file leases / beads) when the coord stack is present.
- **`walkthrough`** — the evidence engine (used by `autothing-walkthrough`, in `mode: evidence`). Built-in **`security-review`** — the run-level security-review engine (final phase, before the codex checkpoint). **`code-review`** — the same-model review engine (used by the standalone `autothing-review`; no longer per-slice). **`frontend-design`** / **`huashu-design`** — design engines (used by `autothing-design-audit`). **`deep-research`** — research. **`/run`** + **`/verify`** — drive the running app.
