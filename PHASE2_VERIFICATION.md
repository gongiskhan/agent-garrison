## T4 — Heartbeat re-enable + suggestion loop

**What it asserts:** the heartbeat ticks every `cadence_minutes`,
each tick POSTs a synthetic prompt to the gateway `/jobs`
endpoint, and the Operative responds by either suggesting one or
two Trello tasks via Slack or staying silent if nothing is
actionable.

**Implementation:**

- `fittings/seed/loop-heartbeat/scripts/heartbeat.mjs` — replaced
  the Phase-1 stub (which logged once and exited) with three
  modes: `--probe`, `--once`, `daemon`. Default cadence still
  40 min via `GARRISON_HEARTBEAT_MINUTES`. Daemon sleeps first
  then ticks (avoids firing while the operative is warming up
  on startup).
- `fittings/seed/loop-heartbeat/apm.yml` — verify hook upgraded
  from `test -f ... && echo ok` to `node ... --probe`. Same
  check, slightly more useful (catches script-broken-but-file-
  exists failure).
- `fittings/seed/personal-operative/.apm/prompts/personal-operative.prompt.md`
  — replaced the **"Heartbeat behavior (Phase 1: off)"** block
  with a **"Heartbeat behavior"** section telling the operative
  how to handle a tick. Five rules: suggest-don't-execute,
  silent-if-empty, dedup-against-previous-tick, plan-on-approval,
  24h-decline-cooldown.

**Scope reduction (versus the original plan):**

The original T4 plan called for a separate JSON state file
storing decline timestamps and last-suggested IDs. **Dropped for
v1** — the Operative manages dedup and cooldown via in-session
memory + the compiled memory layer (it already sees prior
suggestions in conversation history, and the decline-cooldown
window is well within the memory-compiler retention window).

without shipping any capability the prompt rules don't already
cover. If the rules drift in practice, the JSON file is the
fallback to add — but only after observing a real failure.

The `list_open_tasks() / mark_done(id) / add_comment(id, ...)`
shim from the plan is also dropped from heartbeat.mjs itself —
the Operative invokes the existing Trello skill (already

## T5 — Scheduler Fitting

**What it asserts:** the operative can register cron-style jobs
that fire once-per-minute when their cron matches local time,
and the scheduler survives a Garrison restart by reading
`jobs.json` on every tick.

**Implementation:** `fittings/seed/scheduler/`. Greenfield —
no prior empty seed. Stdlib-only Node, ~250 LOC.

- `apm.yml` — `faculty: scheduler`, shape `script`, provides
  `automation-runner:scheduler`. Two config keys
  (`jobs_file`, `log_file`) defaulting under the composition's
  `data/` dir.
- `scripts/scheduler.mjs` — single CLI exposing `--probe`,
  `list`, `add <id> <cron> <cmd>`, `remove <id>`, `run-now`,
  `tick` (one-shot), `daemon` (long-loop).
- `scripts/setup.sh` — Node ≥ 18 check.
- `.apm/skills/scheduler/SKILL.md` — operating principles,
  cron grammar, integration paths for daemon mode.

**Cron grammar:** 5 fields. Supports `*`, `*/N`, single values,
comma lists, and ranges (`a-b`). No `@yearly` aliases, no
seconds, no named months/days. Covers all Phase 2 patterns:
`*/5 * * * *`, `0 8 * * 1-5`, `0 18 * * *`.

**Tick semantics:** at most once per minute per job. `last_run_minute`
is written to `jobs.json` *before* the job spawns, so a crash
mid-run doesn't double-fire on restart. Match is against local
time of the tick.

**Daemon spawning (the wiring decision deferred):** the runner
doesn't currently auto-spawn long-running Fitting processes
beyond the gateway. Three integration paths documented in
SKILL.md:
1. Manual launchctl asuser / tmux pane.
2. Heartbeat-driven `tick` (T4 will wire this).
3. Host crontab calling `tick` every minute.

A future runtime-SDK milestone may add an `x-garrison.daemon`
declaration; deliberately out of scope for Phase 2.

**Verified offline (2026-05-07T01:03Z):**

- `add hello "* * * * *" "echo hello-world"` → registers, persists.
- `tick` (current minute matches `* * * * *`) → ran the job,

- `tick` again same minute → no-op (`{"ran":[]}`) — dedup works.
- `add never "0 3 1 1 *" "echo never"` → `tick` doesn't fire it
