# Garrison — Findings + Lead Synthesis (Garrison-as-agent-engine feasibility)

## Part 4 — What Garrison offers to leverage

### 4.1 — The Orchestrator / V2 Router as a consumable engine

The "orchestrator" in Garrison is **two distinct things that share a name**. (1) The **Orchestrator Faculty / prompt projection** (`orchestrator-projection.ts`, `runner.ts`) — assembles soul + behavior + capabilities + routing and *projects it as a standing rule into the user's real `~/.claude`* — is deeply entangled with disk, APM, and provenance and is NOT cleanly library-callable. (2) The **V2 Router / policy engine** — a separate Fitting at `fittings/seed/model-router/` whose entire policy brain lives in three pure, dependency-free `.mjs` modules — IS importable standalone, in-process, with zero Garrison/Next.js/`~/.claude` dependencies.

#### Orchestrator state (the routing model)

FINDING: [orchestrator-state] Six routing **roles** form a fixed shared vocabulary: `export const ROLES = ["expert", "standard", "fast", "image", "video", "review"];` (`fittings/seed/model-router/lib/routing-core.mjs:18`). Distinct from these, there are **8 Faculty roles** (`orchestrator`, `channels`, `gateway`, `runtimes`, `memory`, `observability`, `sessions`, `surfaces`) in `src/lib/types.ts:10-24` — composition *slots*, not routing roles. Don't conflate them.

FINDING: [orchestrator-state] A **Profile** is "just its roleMap (role -> target id) + discipline overrides" (`routing-core.mjs:14-16`). Three seed profiles — `balanced` (active), `economy` (local ollama), `premium` (all-opus) — each a `roleMap` + `disciplineOverrides` (`config/routing.seed.json:64-103`). Swapping provider/going-local changes only the map, never the policy.

FINDING: [orchestrator-state] The **matrix** is `task-type × tier → role` with documented inheritance `cell > row-default > column-default > global-default` (`routing-core.mjs:51-62`). Task types: `code, review, research, image, video, writing, ops, other`; tiers: `T0-trivial, T1-standard, T2-deep` (`routing-core.mjs:19-20`). **Exceptions** are ordered, first-match-wins, and resolve straight to a role, bypassing the matrix (`routing-core.mjs:47-50`). Two-stage resolution: `resolveRole(config, classification)` → role, then `resolveRoute(config, profile, classification)` maps role→targetId and looks up the concrete target (`routing-core.mjs:44-71`).

FINDING: [orchestrator-state] **Discipline** is the post-task gating spec (`["review","testing","evidence","distribution"]`, `routing-core.mjs:23`), resolved per-tier as `override ?? base ?? "none"` (`routing-core.mjs:74-81`). It is **NOT enforced in code — it is compiled into the prompt as instructions to the operative** (`renderDiscipline()`, `routing-core.mjs:133-142`); there is no runtime check that the operative ran tests/review. **Continuations** (verbs `["store","ask","route","notify"]`, `routing-core.mjs:22`) are likewise compiled to prompt text, not executed (`routing-core.mjs:144-163`).

FINDING: [orchestrator-state] `compileRouting(config, profile)` produces a byte-stable `routing.md` section (`routing-core.mjs:169-201`) injected via the `{{routing}}` placeholder, instructing the operative to end every reply with `[route: <target-id> | rule: <rule-id> | profile: <name>]`. The one real gating/feedback loop is `checkHonored(route, replyText)` which parses that token and diff-checks it against what the gateway resolved, logging `honored:false` on mismatch (`routing-telemetry.mjs:50-57`) — "the misroute signal the Improver feeds on."

FINDING: [orchestrator-state] REAL and shipping: the entire pure resolution layer plus the live gateway Stage-A/B (`RoutedGateway` in `gateway-routing.mjs:204-588`) — classify → resolve → log → slash-inject/respawn → honor-check. NOT wired into the default operative: `{{routing}}` injection is a **no-op when the orchestrator prompt has no placeholder** (`runner.ts:600-606`); the live `garrison-orchestrator` rule does not use routing. STILL A STUB at platform level: the hosted-session launcher (RC4) that would pass `--append-system-prompt` is "not yet wired"; `orchestratorAppendSystemPrompt()` (`orchestrator-projection.ts:137-139`) is a verbatim pass-through awaiting that lane.

#### Policy resolution as an in-process library (orchestrator-as-library)

FINDING: [orchestrator-policy-resolution] A per-turn classifier exists and is hybrid. Pure prompt builder + parser: `buildClassifierPrompt(config, userPrompt)` (`routing-core.mjs:207-230`) and `parseClassification(replyText, config)` which clamps out-of-vocab values (taskType→`other`, tier→`T1-standard`) and returns null only on total JSON failure (`routing-core.mjs:258-269`). A **deterministic keyword fast-path** precedes the LLM classifier: `classifyByKeywords(message, config)` matches an exception's `keywords` array and short-circuits the LLM (`gateway-routing.mjs:193-202`, called first at `:433-438`). The live LLM classifier call is the ONLY model dependency in the per-turn path: `this.classifier.session.runTurn({...})` against a pinned warm haiku session (`gateway-routing.mjs:443`).

FINDING: [orchestrator-as-library] **The V2 router's policy resolution is cleanly importable standalone.** `routing-core.mjs` has **zero imports** and is explicitly "deterministic (NO Date.now / Math.random / I/O)" (`routing-core.mjs:1-3`). `stage-b.mjs` (the switch planner + launch-env builder) also has **zero imports** and is "Pure decision + env builders (no I/O)" (`stage-b.mjs:9`). Exported pure functions Cortex can call directly: `resolveRole`, `resolveRoute`, `resolveDiscipline`, `compileRouting`, `validateRoutingConfig`, `buildClassifierPrompt`, `parseClassification` (`routing-core.mjs:44-298`) — all take a plain JSON `config` + plain args, return plain objects/strings. Hand-written `.d.mts` types ship for TS consumers without a build step (`routing-core.d.mts:1-124`). Telemetry record-builders are pure too: `decisionRecord`, `formatRouteToken`, `parseRouteToken`, `checkHonored`, `promptDigest` (`routing-telemetry.mjs:12-57`).

FINDING: [orchestrator-as-library] **Self-evidence that the pure core runs standalone with no Garrison infra:** the verify probe `probe.mjs` and the CLI `compile.mjs` both import `routing-core.mjs` directly, read only a JSON config file, and produce output ("Pure: the compiler does no I/O beyond reading the config and writing --out", `compile.mjs:11`). A minimal Cortex call is `import { resolveRoute, resolveDiscipline, compileRouting } from "<path>/routing-core.mjs"` + a JSON config object → call. No Next.js, no `~/.claude`, no vault, no APM, no filesystem writes.

FINDING: [orchestrator-side-effects] The I/O-only entanglement is *avoidable*: `routing-telemetry.mjs` mixes pure builders with two disk functions (`appendDecision`, `readDecisions`) importing `node:fs/promises`+`node:crypto` (`routing-telemetry.mjs:9-10,63-85`) — node-stdlib only, optional, skippable. The truly entangled layers Cortex must NOT import: `gateway-routing.mjs` (dynamic-imports cores from disk, depends on `@garrison/claude-pty`, spawns PTYs, writes `decisions.jsonl`, slash-injects keystrokes), `runner.ts:resolveRoutingSection()` (reads composition `<dir>/.garrison/routing.json`, hard-coded repo path import), and `orchestrator-projection.ts` (writes APM files, mutates the global manifest, runs `apmInstall`, writes through the `~/.claude` symlink, records provenance). Even the "pure" `buildOrchestratorInstructions()` (`orchestrator-projection.ts:54-62`) transitively pulls the runner module graph via `substituteCapabilitiesPlaceholder`/`substituteRoutingPlaceholder` imported from `runner.ts`, so it is not cleanly extractable as-is.

> **Cross-reference to Cortex.** Cortex already has its own LLM router (`src/llm-router.ts`: `classifyForSdk` → `{modelId, tier, effort}`, FAST/Haiku → EXPERT/Opus per FINDING [model-call], [chat-invocation]). Garrison's `routing-core.mjs` is a *more expressive* policy engine (matrix × tier × exceptions × discipline × continuations) that Cortex *could* adopt as a drop-in library if it wants role/discipline semantics — but Cortex does not need Garrison runtime infra to do so, and its existing router already covers model selection. Garrison's discipline/continuation layer (prompt-compiled, not enforced) maps conceptually onto Cortex's own prompt-injected governance, not onto Cortex's *enforced* tool/registry layer.

### 4.2 — The RuntimeAdapter contract + DECISIVE FINDING [runtime-percall]

#### The contract as actually built

FINDING: [runtime-contract] The adapter interface (`packages/claude-pty/src/runtime-adapter.mjs:28-37`) is `spawn`, `awaitReady`, `sendTurn`, `awaitResponse`, `setModel`, `setEffort`, `resume`, `teardown`. `spawn(config) -> session`, `sendTurn(session, text) -> void`, `awaitResponse(session) -> {text, artifacts?}`. The "spec" surface is split: a **per-session `config` at spawn** and a **plain `text` string per turn**. There is no per-turn spec object — `sendTurn(session, text)` takes only a string (`runtime-adapter.mjs:61`, `agent-sdk-adapter.mjs:112`).

FINDING: [runtime-contract] Three+ concrete adapters implement it: `ClaudeCodeAdapter` (PTY/TUI reference), `AgentSdkAdapter` (no-PTY structured SDK, `fittings/seed/agent-sdk-runtime/lib/agent-sdk-adapter.mjs:40-186`), plus `CodexAdapter`/`GeminiAdapter`. The pool (`multi-runtime-pool.mjs:34`) and the delegation bridge (`runtime-bridge.mjs:93-98`) drive *only* these 8 methods, runtime-agnostically. The runtime-to-runtime delegation path (`runtime-bridge.mjs:70-132`) DOES accept a structured per-call `task_spec` (`{task, paths, model, constraints, expectedSchema, cwd}`, `validateTaskSpec` at `:28-38`), but it collapses that spec into a **plain-text prompt** via `renderTaskPrompt` (`:134-141`) before `adapter.sendTurn(session, prompt)`; the only structured fields that survive to the runtime are `model`/`cwd`/`env` inside `spawnConfig`.

#### What is per-spawn config vs install-time disk state

FINDING: [runtime-per-launch-override] **System prompt is per-spawn via `--append-system-prompt-file`** (`packages/claude-pty/src/session.mjs:84-86`); the interactive TUI has no string system-prompt override, only the append-file form (`session.mjs:13-15`). The runner writes the assembled prompt to `.garrison/assembled-system-prompt.md` and hands its path through (`runner.ts:611-614` → `GARRISON_SYSTEM_PROMPT_PATH` → `pty-operative.mjs:21`). The Agent SDK adapter takes it as a per-spawn config field `appendSystemPrompt`/`leanPrompt` (`agent-sdk-adapter.mjs:49-52,93`).

FINDING: [runtime-per-launch-override] **MCP config is per-spawn via `--mcp-config <path> --strict-mcp-config`** (`spawn-soul.mjs:57-59,239`) BUT the path points to a **single shared, statically-generated file**: `writeSharedMcpConfig()` writes ONE composition-wide `mcp.json` containing exactly one `garrison` stdio server (`gateway.mjs:122-156`), written once at boot and reused for every spawn (module-global `mcpConfigPath`, `gateway.mjs:72`). It is NOT derived from a per-call spec. **Tool allow/deny is per-spawn via `--allowedTools`/`--disallowedTools`** but sourced from a static soul `spawnConfig` (`spawn-soul.mjs:68-76,241-247`), not a per-call request. **Model and effort are settable mid-session** via `setModel`/`setEffort` — slash-injected into the TUI (`runtime-adapter.mjs:74-81`) or mutated on the SDK session (`agent-sdk-adapter.mjs:167-177`) — the closest thing to a per-call override, but limited to model/effort.

FINDING: [runtime-install-time-state] **Skills and commands are read from disk (`~/.claude` + project `.claude`), never passed per-call.** `buildClaudeArgs` deliberately omits `--setting-sources` so the operative discovers the user's/project's commands+skills from disk (`session.mjs:67-72,88`). The orchestrator's standing capability/behavior is install-time disk state: `apm install` materialises `apm_modules/_local/*` (`runner.ts:118`) and the orchestrator prompt is projected onto disk as `~/.claude/rules/garrison-orchestrator.md` (`orchestrator-projection.ts:80-131`); the durable delivery is the on-disk rule, with `--append-system-prompt` only the per-launch higher-authority fallback (`orchestrator-projection.ts:31-37,133-139`). The capabilities block is computed from the **installed** composition entries (`renderCapabilitiesBlock`, `runner.ts:692-743`), not from a per-call spec. The single `garrison` MCP server is the only one wired by the gateway path; any *other* MCP servers reach the operative only because they live in `~/.claude`/project config on disk.

#### THE DECISIVE VERDICT

FINDING: [runtime-percall] **The runtime contract is PER-SPAWN config + install-time disk state — NOT per-call.** A turn is `sendTurn(session, text:string)` (`runtime-adapter.mjs:61`, `agent-sdk-adapter.mjs:112`): no skills, no MCPs, no system prompt travel with a turn. The only things bound per *launch* are (a) the append-system-prompt FILE path (`session.mjs:84-86`), (b) a single SHARED static `mcp.json` (`gateway.mjs:122-156`), and (c) allow/deny tool lists from a static soul `spawnConfig`. Skills, commands, the orchestrator rule, and the capabilities block are all read from `~/.claude`/`apm_modules` on disk written by `apm install`. **A Cortex agent's per-call spec (its own system prompt + its own skill set + its own MCP set, chosen at call time) cannot be handed to the runtime as built**: the runtime would need a fresh spawn per spec, and even a fresh spawn cannot vary skills (disk-only) or vary MCP servers beyond the one shared file without new code.

FINDING: [runtime-percall] What would have to change, ranked by cost: **(1) System prompt per-call — cheapest, ~90% there:** `appendSystemPromptFile` is already a `spawn(config)` field (`session.mjs:108`) and the gateway already writes per-spawn temp files (`writePromptTempFile`, `spawn-soul.mjs:295-301`); the only missing piece is decoupling it from the single composition-wide `assembled-system-prompt.md` (`runner.ts:611`) so each call gets its own file. **(2) MCPs per-call — wiring change:** `--mcp-config`/`--strict-mcp-config` already accept an arbitrary path and the SDK adapter already accepts `config.mcpServers` per session (`agent-sdk-adapter.mjs:107`); missing is a per-spec config builder feeding it + removing the module-global singleton (`gateway.mjs:72`). **(3) Skills per-call — genuinely new infrastructure:** skills are discovered from `~/.claude`/project dirs (`session.mjs:67-72`); the only knob is `--setting-sources`, which scopes WHICH on-disk dirs are read, not an in-memory set. A dynamic per-call skill set needs either materialising the spec's skills into a temp `.claude` dir per spawn and pointing `CLAUDE_CONFIG_DIR`/`settingSources` at it (`agent-sdk-adapter.mjs:27-28` already reads `CLAUDE_CONFIG_DIR`), or a new spawn parameter the Claude binary does not expose today. **(4) Per-turn variation is impossible without a respawn** — any change to prompt/tools/MCPs/skills requires `teardown` + new `spawn` (the codebase already treats a different base-URL model as "a new spawn, not a setModel", `agent-sdk-adapter.mjs:167-170`). A spec-per-call design maps onto **one ephemeral spawn per call** via `delegate()`/`oneShotTurn` (`runtime-bridge.mjs:90-102`, `session.mjs:277-300`), paying cold-boot cost unless the warm pool is pre-seeded per spec-shape — and the pool warms per *runtime id*, not per spec (`multi-runtime-pool.mjs:21-39`).

> **Decisive cross-reference to Cortex.** Cortex's agents are exactly per-call specs: each `query()` passes a freshly-assembled `systemPrompt` (preset + `buildAppendContent` with per-turn transcript, memory, integration data, catalog — FINDING [model-call], [chat-prompt]), per-call `allowedTools`/`mcpServers` (agent-face wires `mcp__ekoa-local__*` per run — FINDING [skills-loading]), and a per-turn model from `classifyForSdk`. Cortex's chat path is explicitly "a stateless one-shot SDK call per turn" (FINDING [session-model]). Garrison's contract, being per-spawn, maps onto Cortex's model only by treating **one Cortex call = one Garrison spawn** — viable for the system prompt (change 1, cheap) but NOT for dynamic skills (change 3, new infra) and only with new code for dynamic MCPs (change 2). See Part 5.3 for the verdict.

### 4.3 — Skills / Modes / Souls + whether a soul is a clean per-agent identity

#### Skills

FINDING: [skills-definition] Skills are NOT a Faculty — they are a **Quarters platform primitive** surfaced over the real `~/.claude`. `claude-scan.ts` reads them off disk: each subdir of `skills/` containing a `SKILL.md` (`src/lib/claude-scan.ts:49-56`); on-disk shape `skills/<name>/SKILL.md`. The file-surface union is `FileSurface = "skill" | "command" | "rule"` (`claude-scan.ts:15`), edited per-surface in `quarters-detail.ts:14,37`. APM is the single writer; Garrison authors loose files then captures them (`quarters.ts:113`). The `agent-skill` capability KIND was **retired** in the 2026-06-07 Quarters pivot (`docs/CAPABILITIES.md:34`); `src/lib/metadata.ts` rejects it today. (Parked soul fittings still declare the legacy `faculty: skills` + `provides: agent-skill` and no longer parse against the shrunk schema — `fittings/seed/README.md:13-17`.)

#### Modes

FINDING: [modes-system] There is **no Gary/Joe/James "mode" system in the code at all.** Repo-wide grep for `\b(Joe|James)\b` returns zero matches. "Gary" exists only as conversational/doc references — the name the user gave the single main Operative ("our main operative", `docs/conversations.md:13789`); the code's default identity is **"Verity"** (`DEFAULT_SOUL_PROMPT`, `src/lib/compositions.ts:62`). The word "mode" in code refers to unrelated concepts: `WorktreeBinding.mode: "headless"|"interactive"`, `permissions_mode`, `devMode`, the runner "Dev mode" watcher (`src/lib/types.ts:192,283,345`; `runner.ts:551`).

UNKNOWN: [modes-system] A distinct named "Gary/Joe/James" trio of agents/modes could not be located anywhere in the codebase, docs, or fittings. If "Joe"/"James" are intended Cortex-side names they are not represented in Garrison — only "Gary" (the single operative name) and the five parked *soul* archetypes exist. The closest Garrison analog to a multi-persona system is the **soul** mechanism, not a mode system.

#### Souls

FINDING: [souls-model] A **soul** is an identity/persona system prompt ("identity, tone, voice, and boundaries", `docs/FACULTIES.md:160-162`) that the runner folds in ahead of behavior — `buildOrchestratorInstructions` pushes soul first, then orchestrator behavior ("Identity leads so it lands before the long behavior section buries it", `orchestrator-projection.ts:54-61`); assembled as `[fallbackSoul, "", orchestratorRouted].join("\n")` (`runner.ts:610`). The default soul lives at `.garrison/prompts/soul.md` (`compositions.ts:174,205,242`; `runner.ts:585-587`). There is **no separate Soul Faculty anymore** — "identity folds into the assembled prompt ahead of behavior" (`runner.ts:583-584`); the `soul` capability kind was retired (`docs/CAPABILITIES.md:34`). In the *current shipped* runtime there is effectively **one soul per composition** (a singleton).

FINDING: [souls-model] The richer multi-soul model is **parked but on disk**: five distinct soul fittings — `soul-engineer`, `soul-architect`, `soul-researcher`, `soul-assistant`, `soul-companion`. Each is a full system-prompt persona with its own `provides: agent-skill name: soul.<role>` and a `spawn` block declaring a sandbox preset: `soul-engineer` uses `preset: claude_code` (full FS+shell, `soul-engineer/apm.yml:25-26`); the others use `preset: none` (constrained tool sets). They were parked in the 2026-06-07 pivot, de-listed from `data/library.json`, kept on disk for history.

FINDING: [souls-model] Souls are still a **live per-turn routing dimension at the gateway layer** (model-router). A soul is one of two launch-fixed dimensions (with provider): `targetKey = ${provider}::${soul}` (`stage-b.mjs:70-71`); a turn carries `soul?` (`routing-core.d.mts:18`). **Soul change forces RESPAWN-WITH-RESUME, not a slash-inject** ("provider or soul differ → RESPAWN-WITH-RESUME (launch-fixed)", `stage-b.mjs:5-8`); the respawn applies the soul as the append-system-prompt file (`stage-b.mjs:130-136`); context is preserved via `--continue`/carryover (`stage-b.mjs:111-125`), validated by `scripts/probe-soul-switch.mjs`. Souls are addressable as first-class session keys: `POST /sessions/by-soul/<soul>/end` ends a soul's active sub-session; `WorktreeBinding` is keyed by `soul` + `sessionId` (`src/lib/types.ts:189-191`).

#### Soul as a clean per-agent identity (Cortex mapping)

FINDING: [soul-as-agent-identity] **Yes — a Garrison "soul" IS a clean, self-contained per-agent identity** and is the natural mapping target for distinct Cortex agents. Each soul fitting is an independent APM package with (a) its own persona system prompt (`*.prompt.md`); (b) a unique capability name `soul.<role>`; (c) a declared sandbox `spawn` preset / tool boundary (`preset: claude_code` vs `preset: none`); (d) its own `for_consumers` usage contract; (e) its own `base_path`/workdir config. That is a complete, swappable agent identity unit. The runtime already treats souls as **multiple concurrent distinct sessions, one per soul** (resolve per turn, respawn on change, one `WorktreeBinding{soul,sessionId}` per running agent, per-soul lifecycle API). The Orchestrator "handles the soul routing — figures out which soul to invoke" and "the five souls live behind the orchestrator" (`docs/conversations.md:15073-15075`). **So 4 Cortex agents could each map cleanly to a Garrison soul (1 soul = 1 agent identity = 1 routable, respawnable, sandbox-scoped session).**

FINDING: [soul-as-agent-identity] **Two caveats.** (1) The *current shipped* runtime collapsed to a **single soul per composition** (one `.garrison/prompts/soul.md`, soul Faculty/kind retired); the five-soul multi-agent generation is **parked**, and the per-turn soul routing lives in the **model-router gateway fitting**, which is itself not in the survivor stack. (2) Each soul is a *prompt-level* identity (system prompt + tool preset), not a separate process namespace by default — distinctness is enforced by spawn preset and session keying, not by OS isolation. To run 4 Cortex agents as 4 simultaneous souls you would need the parked multi-soul/gateway routing path revived (or each soul stationed as its own composition/session).

> **Cross-reference to Cortex.** A Cortex "agent" today is a **filesystem plugin** (`plugin.json` + composed `system-prompt.md` + skills, FINDING [agent-plugin]); the four agents (chat, coding, brand-research, integration-builder) are either a markdown skill in the `agent`/`claude-code` plugin (chat FINDING [chat-location], coding FINDING [coding-location]) or a hardcoded TS handler with an inline `SYSTEM_PROMPT` (brand-research FINDING [brand-research-location], integration-builder FINDING [integrations-location]). A Garrison soul is a *cleaner, more complete* identity unit than a Cortex plugin (it bundles the sandbox preset and capability contract), so a Cortex agent's persona **fits a soul well** — but per [runtime-percall] the soul is bound at spawn, so this is a per-spawn mapping, not per-call.

### 4.4 — Knowledge faculty + FINDING [knowledge-single-tenant]

FINDING: [knowledge-state] The memory **role** (`faculty: memory`, `src/lib/types.ts:20`) is backed by the active default fitting **`basic-memory`** ("Obsidian-native, plain-markdown memory backed by Basic Memory; indexes the vault into a local SQLite knowledge graph and exposes write/search/read MCP tools to Claude, Codex, and Gemini", `fittings/seed/basic-memory/apm.yml:13`; `provides: memory-store`). It stores a single plain-markdown Obsidian vault (default `~/ObsidianVault`), registered as Basic Memory project `main`, with a `SessionEnd`/`PreCompact` capture hook writing secret-redacted checkpoints into `Memory/` (`scripts/setup.sh:11-114`). A second composite fitting **`knowledge`** ("federates memory + context projections + code graph + symbol-nav behind ONE contract", `provides: memory-store`) is also selected in the default composition. Adjacent: `improver` (nightly dream consolidation) and `vault-git-sync` (scheduled commit+push). `memory-store` is the only relevant capability kind (`src/lib/types.ts:59`).

FINDING: [knowledge-single-tenant] **DEFINITIVELY SINGLE-TENANT.** Exactly one vault = the one user's, with no tenant/account/org partitioning primitive anywhere in Garrison's code. Evidence: (a) the memory SKILL states "Operative memory is a **single** Obsidian vault" (`garrison-memory/SKILL.md:8`); (b) the vault is registered as a single fixed Basic Memory project `main` (`scripts/setup.sh:14`), not keyed by caller identity; (c) the secret vault (`src/lib/vault.ts`) is one global encrypted file at a single `VAULT_PATH` with one `secrets: Record<string,string>` map and a process-global singleton runtime (`vault.ts:8-35`) — no per-tenant keyspace; (d) explicit positioning: "Local-first, single-user. No auth, no multi-tenant, no cloud sync in v1." (`docs/SPEC.md:363`), "There is no multi-tenant isolation to design for." (`docs/GOVERNANCE.md:41`).

FINDING: [knowledge-tenant-scoping-absent] **NO tenant-scoping primitive exists.** A repo-wide grep for `tenant|tenantId|accountId|orgId|workspaceId` across `src/` and the memory fittings returned **zero matches** (the only `tenant` hits are vendored SDK comments). `multi-tenant`/`multitenant` appear only to declare absence (`docs/SPEC.md:355,363`, `docs/GOVERNANCE.md:41`, `docs/GARRISON_EXPLAINED.md:52`). The closest things to "scoping" are not tenant partitioning: `namespace` is the artifact-store content category; "owner-scoped" is per-fitting hook ownership via `_garrison: "fitting:<id>"` tags; APM "project-scoped" is the install cwd. Basic Memory *has* a `project` concept (and `create_memory_project`/`list_memory_projects` MCP tools), but Garrison pins it to the single project `main` and never exposes per-user/per-tenant projects.

> **Decisive cross-reference to Cortex.** This is the **central architectural mismatch**. Cortex is multi-tenant by `ownerUserId`/`userId` enforced in application code on every read/write (FINDING [storage-tenant-scoping], [integrations-storage]); its memory store is per-user (`resolveMemories` keyed by user, FINDING [chat-touchpoint]). Garrison's memory faculty is a single global vault with **no concept of a caller**. ## Part 5 — LEAD SYNTHESIS (authoritative)

The decision this feeds: **can each Cortex agent's reasoning run as a Garrison operative, with everything it needs from Cortex arriving through a tool/MCP boundary, while ALL tenant-scoped state stays in Cortex?**

The synthesis rests on three load-bearing facts established above:
1. **[runtime-percall]** Garrison's runtime is **per-spawn config + install-time disk state, not per-call.** A Cortex per-call spec maps onto **one Garrison spawn**, viable for system-prompt-per-call (cheap), needing new code for MCP-per-call, and needing genuinely new infrastructure for skills-per-call.
2. **[knowledge-single-tenant] / [knowledge-tenant-scoping-absent]** Garrison has **zero tenant-scoping primitive**; it is categorically single-user. Tenant state therefore *must* stay in Cortex — which matches the brief's constraint and is the right architecture, not a limitation to engineer around.
3. **[governance-external-risk]** (from Cortex) Cortex's auth/scope/ownership/credential governance lives at the **data boundary** and *survives* external reasoning **iff** the operative is forced through Cortex's authenticated tool endpoints; the **agent-process** governance (tool/permission confinement, billing/quota accounting, SDK sealing, prompt-injected guardrails) is **lost** and must be reconstituted Garrison-side.

### 5.1 — Per-agent verdict (all four)

The split for every agent (established Cortex-side) is identical in shape: the **reasoning** (system prompt + skill/persona + output parsing) is movable; the **tenant-state/infra** must stay in Cortex behind a tool/MCP boundary. The differentiator is how much tenant coupling rides *inside the loop* vs *as prompt-injected context*.

**1. Integration-Builder Agent — PORTABLE WITH A TOOL BOUNDARY (cleanest).**
- Blocker class: minimal. The agent core (`integration-agent.ts`: `SYSTEM_PROMPT` + `buildPromptWithHistory` + `parseIntegrationOutput`) has **ZERO direct Cortex coupling** — no DB, no Supabase, no stores, no auth, only its own session Map (FINDING [integrations-coupling]). All tenant/infra coupling lives in the thin handler (encrypt_credential, integrationStore upsert, save/list skill, log_activity, integration_ready SSE, test_integration_action, billing, memory affinity — FINDING [integrations-touchpoint]). The split line is exactly the `executeExternal(...)` call at `integration-builder-handler.ts:132`.
- Verdict: **portable with a tool boundary.** Move the prompt+parser into an operative (a Garrison soul is an ideal home); re-expose the ~8 handler touchpoints as Cortex tools/MCPs. **Specific blocker:** none architectural — only that secrets (`encrypt_credential`) must NEVER cross to the operative (the agent's own prompt rule already forbids it, `integration-agent.ts:33-36`), so the `encrypt_credential`/`upsert_integration_config` tools must run Cortex-side and return only non-secret results.

**2. Chat Agent — PORTABLE WITH A TOOL BOUNDARY.**
- Blocker class: moderate. The persona is a markdown skill (`chat-agent/SKILL.md`) with no coupling — movable verbatim (FINDING [chat-coupling]). But the *value* of the chat agent is the per-turn injected context (session history, memory, integration live-data prefetch, automation/Ekoa-action catalog, company/branding, starting points, settings — all TENANT-STATE, FINDING [chat-touchpoint]) and the `call_*` action tools, plus the marker-protocol state machine (`<ekoa-build-redirect/>`, `<ekoa-context>`) whose *parsing* is Cortex-side.
- Verdict: **portable with a tool boundary.** **Specific blocker:** the chat agent is per-call by construction (stateless one-shot, FINDING [session-model]) and its system prompt is re-assembled every turn from tenant state. Garrison's per-spawn contract ([runtime-percall]) means this works only via "one Cortex call = one Garrison spawn" with the per-call system prompt written to a per-spawn file (change 1, cheap) — and the `call_*` tools + catalog must be exposed as a per-call MCP set (change 2, new wiring), since the catalog is per-user and changes between calls.

**3. Brand-Research Agent — PORTABLE WITH A TOOL BOUNDARY.**
- Blocker class: moderate, infra-heavy. The movable reasoning is the prompt assembly + color-selection rules (`branding-handler.ts:176-279`) and output parsing/validation guards (FINDING [brand-research-coupling]). Everything else is INFRA: four signal extractors (Playwright browser pool, dembrandt CLI, Haiku vision) and tenant-state writes (companyStore, memoryStore, jobStore, brand-asset filesystem) — FINDING [brand-research-touchpoint].
- Verdict: **portable with a tool boundary.** **Specific blocker:** the agent runs *text-only with zero tools today* and is fed pre-extracted signals as prompt text. To run as an operative it would call Cortex extraction tools (`extract_site_signals`/`extract_design_system`/`extract_visual_vibe`) and persistence tools (`save_company_branding`/`upsert_company_memory`/`store_brand_asset`) — i.e. the extraction infra (browser pool + dembrandt binary + vision) must stay Cortex-side as MCP tools; the operative must not run Playwright/CLI itself.

**4. Coding Agent — PORTABLE WITH A TOOL BOUNDARY (but the heaviest infra coupling, and the best architectural fit).**
- Blocker class: highest count of touchpoints, but the *cleanest existing shape*. It already runs as an isolated Claude Code SDK session confined to `cwd=HOME=projectDir` with `bypassPermissions` and **no MCP servers today** (FINDING [coding-coupling]) — tenant context is prompt-injected, not live tools. The movable reasoning is the SDK `query()` itself + prompt/skill assembly + slug-text gen + in-build intent classifier. The must-stay set is large: managed-OAuth store, billing ledger, jobStore/artifactInstanceStore/sessionStore, scaffold + **esbuild bundler** + appRegistry, git/GitHub version history + secret guard, screenshot capture, conversation/orchestrator/guidance state, company branding, integration config + `callPlatformIntegration` (prime MCP candidate), memory, SSE transport.
- Verdict: **portable with a tool boundary.** Moving it is "(a) keep the same SDK session, (b) replace prompt-injected tenant context + post-run lifecycle with MCP tools/hooks pointing at the Cortex stores" (FINDING [coding-coupling]). **Specific blocker:** the **build/scaffold/serve lifecycle MUST stay in Cortex** — Cortex owns the authoritative esbuild build, dist-wipe + IIFE validation, registration, and serving; the operative produces only the source tree + `manifest.json` and must NOT pre-build to ESM (FINDING [app-build-step], [app-handoff]). This is the cleanest fit because the handoff is already filesystem-based (the source tree IS the deliverable), which maps directly onto a Garrison worktree the operative is dropped into.

**No agent is "portable as-is"** (every one needs tenant data/credentials/infra that cannot leave Cortex) and **none is "too coupled to move"** (every one's reasoning core is cleanly separable, because Cortex already isolates reasoning from tenant state — the four agents are either markdown skills or inline `SYSTEM_PROMPT` strings with the coupling in handlers/adapters). The universal verdict is **portable with a tool boundary**, gated on the per-call/per-spawn mismatch and the governance reconstitution below.

### 5.3 — Can the runtime contract accept Cortex's agent specs? (DECISIVE, depends on [runtime-percall])

**NO, not as built — and the gap is precise.** Per [runtime-percall], Garrison's runtime takes a **per-spawn `config` + install-time disk state**, while every Cortex agent is a **per-call spec** (freshly-assembled system prompt + per-call tools/MCPs + per-turn model — FINDING [model-call], [chat-prompt], [session-model]). Mapping one Cortex call onto one Garrison spawn is the only viable shape, and it requires, in ascending cost:

- **System prompt per-call — DELIVERABLE CHEAPLY (~90% there).** `appendSystemPromptFile` is already a `spawn(config)` field and the gateway already writes per-spawn temp files; the only change is decoupling from the single composition-wide `assembled-system-prompt.md` so each spawn gets its own file ([runtime-percall] change 1). Cortex's `buildAppendContent` output drops straight into that file.
- **MCPs per-call — NEEDS NEW WIRING (not new infra).** The `--mcp-config`/`--strict-mcp-config` plumbing and the SDK adapter's per-session `config.mcpServers` already accept arbitrary input; missing is a per-spec `mcp.json` builder + removal of the module-global `mcpConfigPath` singleton ([runtime-percall] change 2). Required because Cortex's catalog/`call_*` tool set is per-user and per-call.
- **Skills per-call — NEEDS GENUINELY NEW INFRASTRUCTURE.** Skills are disk-discovered from `~/.claude`/project dirs; the only knob (`--setting-sources`) scopes *which on-disk dirs* are read, not an in-memory set. A per-call skill set requires materialising the spec's skills into a temp `.claude` dir per spawn (pointing `CLAUDE_CONFIG_DIR` at it) or a spawn parameter the Claude binary does not expose ([runtime-percall] change 3). **This is the decisive blocker** if Cortex agents need to vary their skill set per call.
- **Mitigant:** Cortex's coding/chat agents largely *don't* vary skills per call — chat uses a fixed `agent` plugin, coding a fixed `claude-code` plugin (FINDING [agent-plugin], [coding-location]). If the skill set is **fixed per agent identity** (one soul = one agent with a fixed on-disk skill set, [soul-as-agent-identity]), then skills become install-time state per soul and change 3 is avoided — at the cost of one Garrison composition/soul per Cortex agent type, and a respawn (not a fast per-turn switch) whenever the spec's *prompt* changes (which it does every turn → change 1 covers that via the append-file, so the respawn cost is the per-turn `teardown`+`spawn` unless the warm pool is pre-seeded per spec-shape, which it is not — [runtime-percall] change 4).

**Decisive verdict:** the runtime contract can accept Cortex's specs **only after change 1 (cheap) + change 2 (new wiring)** and **only if per-agent skill sets are treated as fixed install-time state per soul** (avoiding change 3). If any Cortex agent must choose its skills dynamically per call, the runtime **cannot** accept that spec without new infrastructure Garrison does not have today.

### 5.4 — Do bridge and outpost unify? (DECISIVE, depends on [bridge-vs-outpost])

**They are the same archetype but DO NOT unify as-is.** Cortex's bridge and Garrison's outpost are both "executor dials out to driver over a WS, capability-call + streaming-step semantics, NAT-friendly" (FINDING [bridge-vs-outpost]) — the topology is identical. But they sit at **opposite ends of the trust spectrum**, and the gap is in auth, isolation, and validation, not topology:

- **Auth class:** Cortex = short-TTL (600s) signed HS256 JWT, audience `ekoa-bridge`, `sub`+`connectionId` claims, header-borne (FINDING [bridge-auth]). Garrison = long-lived static plaintext pre-shared machine token, no TTL/signing/audience (FINDING [bridge-outpost-auth]).
- **Tenancy:** Cortex = mandatory per-owner isolation with ownership-mismatch rejection. Garrison = single global machine namespace, no users (FINDING [bridge-vs-outpost]).
- **Wire trust:** Cortex = strict Zod validation, drop-on-invalid ("never trust the wire"). Garrison = unvalidated `JSON.parse` routing on a trusted LAN.
- **Unauthenticated surface:** Garrison's `/io` PTY socket has no auth at all — intolerable in multi-tenant (FINDING [bridge-outpost-trust]).
- **Abort liability:** Cortex aborts orphaned destructive work on timeout (it is liable for what runs on a user's machine); Garrison deliberately keeps remote PTYs alive (operator owns both ends).

**The trust gradient is inverted:** in Cortex the *executor* (user's machine) is the privacy-sensitive asset and the *driver* (hosted brain) is untrusted SaaS; in Garrison the *driver* (your laptop) is trusted and the *executor* (another Mac) is reached into. **Dropping Garrison's plaintext-token + no-auth-PTV onto Cortex would be a tenant cross-contamination and credential-leak vulnerability** (FINDING [bridge-outpost-trust]).

**Decisive verdict:** they unify **only as a shared executor-dial-out WS correlator + capability/step protocol with auth/tenancy/validation as PLUGGABLE policy injection points** — and only if Cortex's secure defaults are never downgraded by sharing code with the localhost-trusting outpost (FINDING [bridge-vs-outpost]). The Cortex `BridgeConnection` shape is the better base (single socket, validated, cancel-capable); Garrison's split HTTP-RPC + WS-PTY would have to collapse onto the single socket. For the Garrison-as-agent-engine question this matters because **if a Cortex agent runs as a Garrison operative, it cannot reach Cortex tenant data through Garrison's outpost** (no tenant isolation, no validated wire) — it must reach Cortex through the §5.2 tool/MCP contract over Cortex's own authenticated boundary. The outpost is for the *single-user driver→executor* case only.

UNKNOWN: [bridge-vs-outpost] The remote-side daemons are not on disk in the Garrison tree (Cortex's `ekoa-local` control client and Garrison's `garrison-outpost-bridge`, cloned at runtime), so executor-side capability implementations and any executor-side sandboxing are inferred from the controller/broker contract only.

### 5.5 — Consolidated FLAG list (governance breakage / tenant-data exposure)

**FLAG-1 — Garrison memory faculty cannot hold Cortex tenant data (CRITICAL).** Garrison is categorically single-tenant: one global vault, one global secret vault, one Basic Memory project `main`, no `tenant/userId/orgId` primitive anywhere (FINDING [knowledge-single-tenant], [knowledge-tenant-scoping-absent]). Cortex memory is per-`userId` (FINDING [chat-touchpoint], [storage-tenant-scoping]). **Pointing a Cortex agent at Garrison's memory faculty would commingle all tenants in one unisolated store.** Tenant memory MUST stay in Cortex behind `memory.search`/`memory.upsert` tools (§5.2-C). Garrison's memory faculty is usable only for the operative's own non-tenant working notes, if at all.

**FLAG-2 — Outpost is unsafe for tenant reach (CRITICAL).** Garrison's outpost has a static plaintext token, no tenant isolation, unvalidated wire, and an *unauthenticated* `/io` PTY socket (FINDING [bridge-outpost-auth], [bridge-outpost-trust]). A Garrison operative must reach Cortex tenant data ONLY through the §5.2 tool/MCP contract over Cortex's authenticated boundary, NEVER through the outpost. Do not route tenant traffic over the outpost.

**FLAG-3 — Billing/quota accounting breaks under external reasoning (HIGH).** Billing is gated only at Cortex's two agent entry points and metered *after* the SDK call inside Cortex (FINDING [governance-external-risk]). If reasoning + token spend happen in a Garrison operative and only side-effecting tool calls return to Cortex, the spend is invisible to `checkBillingAllowance` and the pre-alpha hard cap is bypassed. MUST re-plumb via `billing.checkAllowance()` (pre-turn) + `billing.recordUsage()` (post-turn) tools (§5.2-F), or route every turn through `/api/v1/request` first.

**FLAG-4 — Agent-process tool/permission confinement is LOST and must be reconstituted (HIGH).** Cortex's autonomous-loop containment (`bypassPermissions` + `cwd:projectDir` for the coding agent; the agent-face default-DENY `canUseTool` allowlist) lives entirely in the SDK options of the Cortex-hosted loop — there is no per-tool-call server interception on the chat/build path (FINDING [governance-external-risk]). A Garrison operative does NOT inherit this; the only thing stopping arbitrary host commands is whatever sandbox/`allowedTools` *Garrison* sets. Each Cortex agent's confinement must be re-expressed as the operative's soul `spawn` preset + `allowedTools`/`disallowedTools` ([soul-as-agent-identity] preset `none` vs `claude_code`), and verified — Garrison's default permission mode is `bypassPermissions` (CLAUDE.md), which is the *widest* posture and wrong for a tenant-facing reasoner without an explicit allowlist.

**FLAG-5 — Credentials must never enter the operative context (HIGH).** Cortex's invariant is that decrypted OAuth tokens / API keys / the encryption key never reach the model — only the HTTP response crosses the boundary (FINDING [integrations-oauth], [governance-enforcement]). This survives external reasoning IFF the operative calls `integrations.callAction`/`encrypt_credential`/`test_integration_action` (which JIT-decrypt server-side and return only results) and is NEVER handed raw `integration-configs.json` or the privileged `integrations execute` intent that returns decrypted creds. The §5.2 contract preserves this; any path giving the operative filesystem access to Cortex's data dir breaks it.

**FLAG-6 — Server must own all path/identity construction (MEDIUM).** Per-user isolation in Cortex is partly structural (sandbox path `~/.ekoa/sandboxes/user-<id>/`, `projectDir` computed server-side from `user.id` — FINDING [governance-external-risk], [app-serving-lifecycle]). If a Garrison operative supplies its own `cwd`/`projectDir`, isolation breaks. The §5.2 `project.provision` tool must compute and return the path from the verified token; the operative receives a path, never chooses one.

**FLAG-7 — Prompt-level guardrails + catalog are advisory and Cortex-assembled (MEDIUM).** Cortex's memory-guardrail `RULE:` injection and the automation/integration catalog are prompt-assembly steps inside Cortex (FINDING [governance-external-risk]). An operative that builds its own prompt loses them. If they matter for safety, Cortex's assembled prompt must be the one written to the operative's per-spawn append-file (§5.3 change 1), not a Garrison-side prompt.

**FLAG-8 — The improver/automation self-healing must NOT be repointed at tenant data (MEDIUM).** Garrison's Improver is tenant-safe ONLY because its target set is hardcoded to two single-user system surfaces (`memory/vault`, `skill`) with no caller dimension (FINDING [improver-tenant-safety]). Repointing it at Cortex tenant artifacts would require owner-scoping Garrison categorically lacks (FINDING [knowledge-tenant-scoping-absent]). Cortex's own self-healing automation is *more* mature than Garrison's (FINDING [automation-self-healing], [garrison-automation-gap]) and MUST stay Cortex-side; do not attempt to move automation execution/replan into Garrison.

**FLAG-9 — The `POST /api/apps/:appId/compile` endpoint is unauthenticated (LOW, pre-existing).** Independent of the pivot, this Cortex endpoint has no auth (FINDING [governance-enforcement]); it is a standing gap that any external-reasoning architecture inherits and should close before broadening the attack surface.

### Bottom line

All four Cortex agents are **portable with a tool boundary** — their reasoning cores are already cleanly separated from tenant state Cortex-side, and a Garrison **soul** is a clean per-agent identity home ([soul-as-agent-identity]). The architecture the brief proposes (reasoning in Garrison, all tenant-scoped state in Cortex behind a tool/MCP boundary) is **sound and is the only safe shape** — Garrison's categorical single-tenancy ([knowledge-single-tenant]) *forces* tenant state to stay in Cortex rather than merely recommending it.

But it is **not free**, gated on three things: (1) the **runtime per-call gap** ([runtime-percall]) — deliverable for per-call system prompts (cheap) and per-call MCPs (new wiring), but per-call skills need new infra unless skill sets are fixed per soul (§5.3); (2) **building the inbound tool/MCP contract** of §5.2, which does not exist today (UNKNOWN [integrations-tool-contract]) and must carry the governance token, JIT-decrypt credentials server-side, and re-plumb billing; (3) **reconstituting the agent-process governance** that external reasoning loses (FLAGS 3, 4, 5, 6, 7). The outpost does **not** serve as the tenant-data path ([bridge-vs-outpost], FLAG-2) — tenant reach is the §5.2 contract only — and Cortex's self-healing automation is a capability to **preserve Cortex-side**, not relocate (FLAG-8).
