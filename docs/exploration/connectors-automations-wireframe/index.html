Connectors & Automations for Agent Garrison

Bring ekoa's automation engine — bash & browser steps, vision-driven self-healing, live streaming, per-step feedback, human-in-the-loop — into Garrison as two new Fittings that compose capabilities Garrison already owns. Plus a new Connectors faculty so external services (Slack, Google, GitHub, Stripe, …) are connected once and called from anywhere. The ekoa-specific ekoa_action step is deliberately left out.

This is a wireframe, not an implementation. Nothing here is built. Framed dashed boxes (LIVE BROWSER STREAM RENDERS HERE) mark areas that render live data at runtime. Everything is styled in Garrison's real visual language so it reads as "what it will look like in the app." The governance section argues why this is allowed to exist in Garrison at all.

The one-paragraph thesis

ekoa hand-rolled a Playwright browser harness, a CDP screencast streamer, an AES-256-GCM credential store, an artifact/screenshot store, a scheduler, and an MCP tool surface — all of which Garrison already has as first-class capabilities. So the heavy substrate is not rebuilt. Connectors and the Automations engine are thin Fittings that wire those existing capabilities together. That reuse is exactly what makes the addition pass Garrison's Honesty Test instead of being "ekoa leaking into the platform."

01Governance

Does this belong in Garrison?

Garrison's job is compose · run · observe · quarters. Anything beyond that lives in Fittings. The Honesty Test gates every choice: "Does this make sense for Claude Code on its own merits, with no reference to any specific downstream consumer?" ekoa is a consumer — so the design must be justifiable without ever saying "because ekoa needs it."

The Honesty-Test framing (passes)

  • Connectors — "A long-running operative needs authenticated, reusable connections to the external services it acts on, with a discoverable catalog of actions it can call. Trello / Slack / ekoa happen to use this too." → load-bearing for any Claude Code operative.
  • Automations — "An operative benefits from saving a proven multi-step procedure (browser + shell + HTTP + service calls) and re-running it reliably, with self-correction and a human handoff when it hits a CAPTCHA / MFA." → load-bearing on its own merits.
  • Both are compositions of existing capabilities, which is precisely Garrison's purpose. No new shell surface duplicates a Fitting's UI: the rich run-viewer lives in the Automations Fitting's own view, surfaced like Kanban Loop — Garrison only renders the view, doesn't own it.

Precedent already in the tree

  • faculty: automations is already a live deprecation alias — the browser-automation seed declares it today.
  • data-source capability kind exists (Trello) — Honesty-Test-added "because a real Fitting needed it." Connectors generalizes the same idea to read+write+auth+triggers.
  • automation-runner kind exists (Scheduler, Google Calendar, Improver).
  • The historical Faculty list literally had #3 Data Sources and #5 Automations ("actions the operative can take in external UIs or systems; example: Playwright browser automation seed").
Where leakage would happen — and how this avoids it. ekoa's ekoa_action step (operate on an ekoa-built artifact via its manifest) is consumer-specific → excluded. ekoa's Portuguese strings, its "needs the local ekoa daemon" model, and its bespoke credential crypto do not come across — Garrison uses its own Vault, its own Browser Fitting, its own Artifact Store. No hardcoded user paths, no "EKOA port," no consumer naming in the capability definitions.

New vocabulary this introduces (each Honesty-Test-gated)

04Connectors faculty

Connectors — connect once, act everywhere

A connector is a connected external service: a catalog of actions (each marked mutating or read-only), an auth method (OAuth2 / API key / service account / none) backed by the Vault, and optional triggers (webhook in, or a polling listener). Automations call them via the connector step; channels and the Operative can call them too.

09Self-healing

Self-healing — the fixer loop

When a step fails recoverably, an LLM "fixer" looks at the screenshot + accessibility tree + the plan window and proposes one patch. The engine applies it and retries the same step — within a budget (max fixer calls, max patches per step, wall-clock). Every move is shown inline.

proposing

Fixing step 3 · attempt 2

Spinner + "Asking the model to patch the plan. Usually 5–15s." Shows what failed: "button not found".

applied

Inserted a step before this one

Verb by patch kind — Inserted / Rewrote / Skipped. Shows the new step + one-line reasoning ("a cookie banner is covering the button — dismiss it first").

aborted

Fixer aborted

When the plan is unrecoverable. Shows the reasoning; the run ends failed (honestly — never edited to look passed).

Patch kinds: insert_before (add a prerequisite) · replace_current (rewrite the action) · skip_current (no longer needed) · pause_for_user (CAPTCHA / MFA / payment — hand to the human) · abort. A fast regex path catches CAPTCHA/MFA/payment without an LLM call. Failure kinds: verify_failed · browser_failed · navigate_failed · connector_failed.

Also shown in the live activity bar

✦Fixing step 3 (attempt 2)…— rewriting the click target. What failed: element intercepted.Cancel

A sticky activity bar surfaces the current state (running / fixing / paused / completed) and auto-hides terminal states — so you can scroll the steps while still seeing what's happening.

12Decide together

Open questions before we build

The wireframe assumes a recommended shape, but these are genuine forks worth settling first — they affect phasing and the core vocabulary.

  1. Connector vs data-source. Make connector a new capability kind that supersedes data-source (Trello migrates), or keep data-source for read-only and add connector only for action+auth+trigger services? Recommendation: one connector kind; data-source becomes a read-only profile of it.
  2. Connectors faculty: role or capability-faculty? A 10th core role, or an optional Agent-tier capability faculty like Knowledge/Research? Recommendation: capability faculty (multi), so it stays optional and discovery-driven.
  3. Where do connector definitions live? One connector = one Fitting (clean APM packaging, heavier), or a registry of HTTP-templated connector specs inside one "Connectors" Fitting (lighter, ekoa-like)? Recommendation: a registry Fitting for the long tail + real Fittings for the marquee ones (Slack/Google), mirroring the Hybrid model the capability faculties already use.
  4. Automation storage. Automations as data inside one engine Fitting (recommended — they're user content, not packages), or each automation as its own APM Fitting? Quarters already shows the precedent for "owned content vs packages."
  5. The planner LLM. Route goal→steps and the fixer through the Model Router / gateway (so it uses the operative's own auth + tiering), rather than a hardcoded model. Strongly recommended — keeps it Honesty-Test clean and free of API-key billing.
  6. Phasing. Suggested order: (1) Connectors faculty + Vault-backed auth + catalog → (2) Automations engine with the non-browser steps (navigate/wait/command/api_call/connector/sub) + run viewer → (3) browser + verify steps on the Browser Fitting + live stream → (4) self-healing fixer + pause/consent → (5) triggers (webhook via Gateway, listener via Scheduler) + the call_automation MCP tool.
  7. The dropped ekoa_action. Confirm we omit it entirely, or replace it later with a Garrison-native capability step that calls any installed Fitting's capability via the any discovery mechanism. Recommendation: omit for v1, revisit as a native step.
Next step. This is exploration only — nothing is built. When the forks above are settled we can turn the phasing into a real plan (beads epic + slices) and start with Phase 1 (Connectors), which is the smallest, most independently-useful piece and unblocks everything else.
