Connectors & Automations for Agent Garrison

Bring ekoa's automation engine into Garrison — bash & browser steps, vision self-healing, live streaming, human-in-the-loop — but instead of reusing Garrison's early substrate as-is, audit it against ekoa's and take the best of each. That audit (and the upgrades it forces) is itself a core reason Garrison was built. Paired with a new Connectors faculty and a Vault that locks every key — the headline differentiator — and a chat-to-build authoring flow that reuses the Discuss handoff we just shipped.

Wireframe, not implementation. Nothing here is built. Framed dashed boxes (LIVE BROWSER STREAM) mark live-render areas. Everything is grounded in a verified read of the real code (Discuss handoff, Vault internals, Model Router, scheduler, artifact-store consumers, browser substrate) — cited where load-bearing. Styled in Garrison's actual visual language.

01Substrate audit

Audit each substrate against ekoa — take the best, chuck the rest

Garrison's browser/vault/scheduler/store were built early and barely touched since. The point of Garrison is to be the place where these get good. So this isn't "reuse what exists" — it's measure each one against ekoa's battle-tested equivalent, keep the winner, and upgrade or chuck the loser. Verdicts: keep upgrade decouple drop / chuck

SubstrateGarrison today (verified)ekoa does betterVerdict
Browser / CDP
browser-default :7084		 Stateless CDP service: JPEG screencast (/viewport), input (/input), raw CDP + full DevTools proxy, multi-tab REST, console/network capture, element-pick. No vision, cache, fingerprint, or persistent profile. Persistent per-profile context + stealth (survives cookies/consent, evades bot-detection), fingerprint-keyed action cache, vision action resolution, locator fallback ladder, accessibility snapshots, act-and-observe envelopes. keep the CDP service + DevTools proxy + element-pick. upgrade it with observation/fingerprint/a11y endpoints + opt-in persistent-profile & stealth. Build the cache→vision→execute orchestration layer above it (port ekoa's fingerprint.ts/executor.ts/cache.ts). chuck the thin browser-automation seed — it's subsumed.
Vault
src/lib/vault.ts		 AES-256-GCM, scrypt KDF (N=16384), data/vault.json 0600, in-memory passphrase. Crypto is solid. But: writes non-atomically (writeFile+chmod), delivers the entire vault to any fitting that consumes it (no per-secret scope — secret-ref is a placeholder), no OAuth-refresh storage, no audit log, no keychain, dev passphrase hardcoded. ekoa's per-service credential store had OAuth refresh + per-integration scoping — but used a plain ENCRYPTION_KEY env (weaker than Garrison's scrypt). So ekoa wins on scoping/lifecycle, Garrison wins on crypto. keep the crypto/KDF. upgrade: atomic write (the immediate correctness fix), per-connector secret scoping, OAuth refresh-token storage + rotation, an access audit log, an OS-keychain-only master key (no passphrase to leak). This becomes the marquee.
Scheduler
fittings/seed/scheduler		 Node CLI (add/remove/list/tick/daemon), cron grammar, jobs in ~/.garrison/scheduler-jobs.json. Core is already platform-neutral — but the only "always-on" path is an unbuilt io.garrison.scheduler launchd agent, and it's labelled platforms:[claude-code]. ekoa ran a long-lived listener-supervisor (one worker per polling trigger) as an always-on service — the model for connector listeners (IMAP poll, etc.). keep the job store + cron + scheduler-beats.mjs. decouple: extract daemon() into a standalone always-on Node service (systemd / Docker / PM2 / launchd), independent of Claude Code & macOS, that fires whether or not the operative is up. Fold in ekoa's listener-supervisor pattern for connector polling.
Artifact Store
src/lib/artifact-store.ts		 A file organizer: <root>/<namespace>/<file> + .meta.json sidecars. Only 3 HTTP consumers (documents + artifact-store list/[id]). It's "barely used," and it's drop-safe. ekoa worked in a run-dir of plain files — closer to how Garrison actually wants to work (files, not artifacts). drop the artifact-store abstraction + the artifact-store capability kind. Replace with plain files + the new File Browser. Documents/voice/recordings just become folders.
Data Sources
data-source kind		 One read-only kind (Trello), re-added 2026-06-10 for a single Fitting. Consumed by knowledge / morning-briefing / personal-operative. — drop the data-source kind entirely. Trello becomes a Connector; a future DB (Supabase) is also a Connector. Migrate the knowledge / morning-briefing consumers to connector; the Trello-backed Tasks wiring is dropped, not migrated (retiring it for our own Kanban).
Net new code this implies (beyond wiring): an automation orchestration layer (cache→vision→execute, ported from ekoa) above the upgraded Browser Fitting; the Vault upgrades (atomic write, per-connector scope, OAuth refresh, audit); the scheduler daemon extraction; the File Browser fitting (reusing the existing Monaco FittingEditor.tsx + marked); and the Connectors faculty + per-service Fittings. The substrate that already exists is the foundation — not skipped, but levelled up.

03★ The marquee

The Vault — your keys, sealed

Vault-sealed why this matters more than anything

Most agents leak your keys. Garrison seals them.

The biggest, least-talked-about risk with agent platforms like openclaw and hermes is how they store your credentials — plaintext .env files, tokens pasted into config, secrets handed wholesale to every tool. When you connect Slack, Google, or Stripe to a Garrison operative, the key goes into the Vault and never comes back out: encrypted at rest, scoped to exactly one connector, materialized only into the single process that needs it, redacted from every log, and rotated without a redeploy. This is the thing we advertise everywhere — because when Garrison goes public, it's the thing people should trust it for.

Sealed at rest

AES-256-GCM, master key held in the OS keychain — no passphrase anywhere to leak or phish. Atomic 0600 writes (fixes today's non-atomic writeFile+chmod).

Scoped per connector

A Slack connector can read the Slack key and nothing else. Replaces today's all-or-nothing delivery — secret-ref becomes real per-connector binding.

Never plaintext to you

You connect; you never see the token again. The UI shows sk_live_••••6f2a, never the value. No copy-paste of secrets anywhere.

OAuth, handled

Refresh tokens stored + auto-rotated; expiry tracked; a revoked grant flips the connector to "Reconnect." Today there is no OAuth-refresh storage at all.

Materialized just-in-time

The secret enters only the one process that runs the step, for the duration of the call, then is gone. Auto-redacted in stdout/stderr and run records.

Audited

Every read is logged: which connector, which secret, when, outcome. A tamper-evident trail you can show a security reviewer.

05Authoring

Chat-to-build — describe it, don't draft it

No more plain-English goal box. Authoring starts with a Discuss skill that opens a real conversation: "What would you like to automate?" The operative (James) asks questions, makes suggestions, and writes an automation brief; back in the fitting, the planner (via the Model Router) turns the brief into steps, rehearses them, and hands you a reviewable automation — with the same live visibility ekoa has. This reuses the exact Discuss → web-channel handoff shipped for Kanban, verified in the code.

Verified mechanism (kanban-loop/scripts/discuss.mjs + web-channel-default): the Discuss button builds a URL with a base64 context blob + an auto-sent kickoff message + mode=james. The web-channel is a generic relay (never parses context). 10Decoupled

Scheduler — a platform-agnostic always-on Node daemon

The operative is meant to be always running, so triggers must fire whether or not Claude Code is up — and on any OS. Today's scheduler's only "always-on" path is an unbuilt macOS launchd agent and it's tagged Claude-Code-only. The core (job store, cron, scheduler-beats.mjs) is already platform-neutral; we extract the daemon and run it as an independent service.

What it runs (automation triggers)

  • Cron automations — "every weekday 8am" fires a run.
  • Listener automations — poll a connector (IMAP, a board) on a cadence; one worker per trigger (ekoa's listener-supervisor pattern folded in).
  • Webhook automations — registered with the Gateway, not the scheduler (inbound).
  • Existing Kanban ticks + Improver nightly keep working unchanged.

Smaller forks — decided

  • Vault master key — keychain only. The master key lives in the OS keychain (macOS Keychain / libsecret / Windows Credential Manager) with no passphrase fallback. The threat model is explicit: a passphrase is a stealable, phishable attack surface and a leaked one is unacceptable; losing keychain access in a catastrophic failure is an acceptable loss (the vault is simply reconnected). So there is no passphrase anywhere — Garrison would rather be unrecoverable-if-the-keychain-is-gone than hold a secret someone could take.
  • Browser orchestration home — the Automations Fitting. There is no existing src/lib home for it, so the cache→vision→execute layer lives inside the Automations Fitting (lib only if it were already there). The Browser Fitting stays a pure service; the Automations Fitting owns the orchestration.
  • Automation file scope — machine-local. ~/.garrison/automations/<id>.yml.
  • Derived Tasks — disconnected, not rewired. Move the Trello Fitting to Connectors but don't wire derived Tasks to anything. Trello-backed Tasks is being retired in favour of our own Kanban, so the tasks: wiring is dropped now rather than migrated. The Trello connector stays for direct list/card actions; it just no longer backs a "Tasks" surface.
Still exploration — nothing built. When you're happy with this shape, Phase 0 (Vault + scheduler daemon) is the place to start: smallest, most independently valuable, and it's the trust story that makes the rest worth advertising.
