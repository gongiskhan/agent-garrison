# Cortex — Findings (Garrison-as-agent-engine feasibility)

**Repo explored:** `/Users/ggomes/dev/ekoa-dev/cortex` (active source `cortex/src/`). Related: `/Users/ggomes/dev/ekoa-dev/supabase/migrations/`, the daemon at `/Users/ggomes/dev/ekoa-local/`, the shared content pool `/Users/ggomes/dev/ekoa-dev/ekoa-data/`.

**The decision this feeds:** can each Cortex agent's *reasoning* run as a Garrison operative, with everything it needs from Cortex (integrations, tenant data, served-app context) arriving through a tool/MCP boundary, while ALL tenant-scoped state stays in Cortex on its own infrastructure?

**The one-line answer:** the substrate already is the Claude Agent SDK on managed OAuth, and three of the four agents are *content-defined* (markdown skills / hardcoded prompts) with a clean reasoning↔state seam. The reasoning is movable. What blocks a naive lift is that today **almost all tenant context is injected as prompt text, not exposed as live tools**, and several governance teeth (autonomous-loop tool confinement, billing/quota accounting, SDK sealing) live *inside the agent process configuration*, so they are lost — not inherited — the moment a different process does the reasoning.

## Shared substrate (applies to all four agents)

These findings are common to every agent and are stated once here; per-agent sections reference them.

FINDING: [model-call] The agent/model-call substrate is the **Anthropic Claude Agent SDK** (`@anthropic-ai/claude-agent-sdk` v`0.2.118`, `package.json:2`), NOT raw `@anthropic-ai/sdk`, NOT `claude -p` headless, NOT the Agents API. No `@anthropic-ai/sdk` dependency exists; a repo-wide grep for `claude -p` / headless `spawn('claude')` returns zero hits.

FINDING: [model-call] Three model-invocation mechanisms in two categories: (1) **SDK `query()`** is the primary agent path (`src/adapters/external.ts:1`, `src/agent-face/index.ts:29`); (2) **SDK `startup()`** pre-warms a subprocess (`external.ts:5,86`); (3) a **raw-HTTP fallback** `callAnthropic()` does a direct `fetch` to `https://api.anthropic.com/v1/messages?beta=true` (`src/anthropic-client.ts:63,106`), used only by `turn-classifier.ts:162` and `visual-vibe.ts:206`.

FINDING: [model-call] A fourth distinct surface is the **LLM Gateway** (`src/llm-gateway.ts`): Cortex exposes its own managed OAuth as an Anthropic-Messages-compatible endpoint at `POST /api/v1/llm/messages` (`llm-gateway.ts:454-455`), a `node:https` reverse proxy to `api.anthropic.com` that **clamps every request to Haiku** (`wireTier = LlmTier.FAST`, `llm-gateway.ts:333`) because OAuth `/v1/messages` only authorizes Haiku (`llm-gateway.ts:18-21,319-335`). This is the "fast local loop" provider seam, not the agent loop.

FINDING: [model-call] **Auth is managed-OAuth only; no API key billing.** `buildSdkEnv()` (`src/adapters/external.ts:132-152`) deletes `CLAUDECODE`, `ANTHROPIC_API_KEY`/`ANTH_API_KEY`, `ANTHROPIC_BASE_URL`, sets `CLAUDE_CODE_OAUTH_TOKEN` from `getClaudeToken()` (Supabase-backed refresh loop, `src/services/claude-auth.ts:104`), and sets `CLAUDE_CODE_EMIT_SESSION_STATE_EVENTS='1'` (`external.ts:139,145`). The `claude_code` preset is the **strong-model authorizer on OAuth** — Opus/Sonnet are reachable only via the preset (`llm-gateway.ts:18-21`, `agent-face/index.ts:10-20`).

FINDING: [model-call] **Model ids / params** live in `src/llm-router.ts:95-124`: Haiku=`claude-haiku-4-5-20251001`, Sonnet=`claude-sonnet-4-6`, Opus=`claude-opus-4-8[1m]`. Four tiers map to `{modelId, thinking, maxOutputTokens}` and SDK `effort` (FAST→low, REASONING_LIGHT→low, WORKHORSE→medium, EXPERT→high; `llm-router.ts:130-135`). Default external model is `claude-opus-4-8[1m]` with effort `high` (`config.ts:72-73`). Temperature/max_tokens are NOT passed to `query()` (the SDK manages them); the router's `thinking`/`maxOutputTokens` are advisory on the SDK path.

FINDING: [model-call] **System prompt is the `claude_code` preset + appended Ekoa identity** by default: `systemPrompt: { type: 'preset', preset: 'claude_code', append: appendContent }` (`external.ts:926-930`), gated by `config.external.usePreset` (default true). When `systemPromptOverride` is set it bypasses the preset as a direct string (`external.ts:923-924`). `appendContent` is assembled by `buildAppendContent()` (`external.ts:303-580`) — see per-agent sections for what each pulls.

FINDING: [model-call] **MCP servers**: the primary `external.ts` path passes **no `mcpServers`**. They are used only on the **agent-face** path: `mcpServers: { [EKOA_LOCAL_SERVER_NAME]: mcpServer }` (`agent-face/index.ts:228`), `mcpServer = createSdkMcpServer({ name:'ekoa-local', tools:[bash, fs, browser] })` (`daemon-tools.ts:264-272`), seen by the model as `mcp__ekoa-local__{bash,fs,browser}` (`daemon-tools.ts:278-282`).

## Part 1 — The four agents, decomposed

> **An "agent" in Cortex is not uniform.** Two are content-defined markdown **skills** loaded into the SDK `agent` plugin (chat, coding); two are **hardcoded TypeScript** orchestrating one SDK call (brand research, integrations). All four drive the same external adapter (`src/adapters/external.ts`). The reasoning↔state seam is the `executeExternal(...)`/`executeUnified(...)` call in each handler.

### 2.1 — Integrations storage / access + single tool/MCP contract

**Two stores joined at execution time by `integrationKey`/`type`.**

FINDING: [integrations-storage] Per-user **connection rows** (credentials) live in one flat JSON file `~/.ekoa/data/integration-configs.json` via a mutex-serialized `JsonStore` (`persistence/integrations.ts:32-34`). Row shape `StoredIntegrationConfig { id, name, type, config, credentials?:string /*encrypted*/, enabled, platformProvider?, oauthState?, needsReauth?, ownerUserId? }` (`:5-30`); `ownerUserId` is the tenant key (`undefined`=global, else creator's id, `:23-29`). **No tenant/company table** — multi-tenancy is pure post-read array filtering on `ownerUserId` (`integrations-handler.ts:74`); `UserContext.companyId` exists but is NOT used to scope rows.

FINDING: [integrations-storage] **Action definitions ("skills")** are on disk: global `ekoa-data/integrations/<key>/`, per-user `~/.ekoa/sandboxes/user-<id>/integration-skills/<key>/`, each holding `SKILL.md`+`config.json`+`history.json` (`integration-storage.ts:8-49`), cache keyed `"<scope>:<integrationKey>"`. Platform integrations have a **third** static source: a checked-in `config.json` of HTTP action templates under `VERSIONED_DIR/<integrationKey>/config.json` (`platform-integration-call.ts:24-27,90-96`).

FINDING: [integrations-access] The agent→action path forks on a hardcoded platform set: `call_integration_action` (`call-integration-action.ts:17,29-42`) — `if PLATFORM_KEYS.has(integrationKey) return callPlatformIntegration(...) else executeUserIntegrationAction({..., ownerUserId:ctx.userId})`. **Tenant scope is taken from `ctx.userId`, never from agent-supplied args.** Platform path `callPlatformIntegration` (`platform-integration-call.ts:423-542`) looks up by *provider only* via `getValidTokens(provider)` (`:293-312`) which does `configs.find((c)=>c.platformProvider===provider)` — **it does NOT filter by `ctx.userId`** (`:294-295`; `_ctx` unused `:425`), so the platform path is effectively single-tenant / first-match-wins. User-defined path `executeUserIntegrationAction` (`integration-action-executor.ts:148-226`) DOES filter: `configs.find((c)=>c.type===integrationKey && (c.ownerUserId===input.ownerUserId || (c.ownerUserId===undefined && input.superAdmin)))` (`:176-180`).

FINDING: [integrations-oauth] Credentials are **encrypted at rest** as `iv:authTag:ciphertext` (base64) AES-256-GCM, key from `process.env.ENCRYPTION_KEY` padded to 32 bytes (`crypto.ts:10-23,50-101`; platform path re-implements identical crypto inline `:29-65`). Plaintext = `OAuthTokens {access_token, refresh_token, token_type, expires_at, scope, email?, tid?, provider}` (`platform-oauth.ts:15-31`) or, for user-defined, the JSON `configValues`. **Injection is just-in-time and never returned to the agent**: tokens interpolated as `{{access_token}}`/`{{<field>}}` into the HTTP template; only the HTTP response is returned (`platform-integration-call.ts:453-535`, `integration-action-executor.ts:192-225`; contract documented `call-integration-action.ts:9-11`). Token refresh is transparent/self-healing with singleflight per config id, `needsReauth` on `invalid_grant`, and a startup proactive-refresh timer (`platform-oauth.ts:145-327`, `platform-integration-call.ts:320-417`). OAuth **client secrets** live in env, never per-row. The Anthropic managed OAuth token is **separate** and must not be conflated (`external.ts:132-152`).

**Single tool/MCP contract — feasibility.**

FINDING: [integrations-tool-contract] **Yes — the seam already exists and is clean.** Discovery (`buildAutomationCatalog`, `catalog.ts:81-87`) + execution (`call_integration_action`) are already two pure functions whose only tenant authority is `ctx.userId`/`ownerUserId`, with the credential never crossing the function return boundary. The four tools `list_integration_actions`/`call_integration_action`/`list_ekoa_actions`/`call_ekoa_action` are already registered with names, descriptions, arg summaries, `requiredScopes` (`tools/index.ts:241-305`), and Cortex already builds `createSdkMcpServer` MCP servers for the agent face — so wrapping these in an MCP server is **mechanical**.

FINDING: [integrations-tool-contract] **Sketched external MCP contract** (thin wrappers, tenant-bound by the verified token, never by an argument):
```
integrations.list           in {query?, includeActions?}                out {integrations:[{integrationKey, displayName, provider,
```

**Auth token shape**: reuse the session JWT — `Authorization: Bearer <jwt>` → `verifyToken` → `UserContext{id,role,scopes[],companyId}` (`auth.ts:19-57`); the wrapper derives `userId`/`superAdmin` server-side and builds `ToolExecutionContext`. ### 2.2 — Storage / data layer (Supabase) + one-interface difficulty

FINDING: [storage-layer] The PRIMARY data layer is **local JSON files**, not a database: `JsonStore<T>` reads/writes one `.json` array per domain at `~/.ekoa/data/`, with an in-process `Mutex`, atomic temp+rename writes, `.tmp` crash recovery (`persistence/store.ts:39-170`). A SECOND swappable layer exists just for user-built-app data: a `StorageBackend` interface (`app-data-backend.ts:22-42`) with `FsStorageBackend` (default) and `MongoStorageBackend` ("Firestore with MongoDB compatibility"), selected by `EKOA_APP_DATA_BACKEND=fs|mongo` (`app-data.ts:42-60`). **Supabase is NOT the app data layer** — it is a control-plane/billing store via raw-`fetch` PostgREST (no `@supabase/supabase-js` dep) for Claude OAuth tokens, license/company lookup, installation heartbeat (`supabase-client.ts:1-212`; tables `standalone_credentials`, `companies`, `installations`); optional (`isSupabaseConfigured()`, `:13-15`).

FINDING: [storage-tenant-scoping] Tenant scoping is enforced **ENTIRELY in application code, never by the store**. Per-user isolation is post-read filtering: `sessionStore.query(s=>s.userId===userId)` (`persistence/sessions.ts:67-69`); `all.filter(i=>i.userId===user.id && !i.featured)` (`artifacts-handler.ts:96-104`, repeated at 197,234,308,592,772). Some stores scope by **path** (stronger): projects at `~/.ekoa/data/projects/user-<userId>/<projectId>.json` (`projects.ts:23-31`); settings at `~/.ekoa/sandboxes/user-<userId>/settings.json` (`settings.ts:61-70`). The `appDataStore` is scoped by `(appId, collection)` with **NO per-user partition** — explicitly "single-company model" (`app-data-backend.ts:11-13`). Role/scope/ownership gates are helpers in handlers (`requireScope`/`requireAdmin`/`requireSuperAdmin`/`requireOwnershipOrSuperAdmin`, `shared.ts:61-148`), not the store. **Supabase RLS is effectively inert**: migration `001_license_system.sql:160-171` enables RLS then grants "Service role full access", and Cortex always connects with `SUPABASE_SERVICE_ROLE_KEY` (`supabase-client.ts:47-48`) which bypasses RLS.

FINDING: [storage-boundary] Access is **PARTIALLY funnelled**. Store *construction* is centralized — every `new JsonStore(...)` (15 occurrences) lives in `src/persistence/*.ts`; barrel `persistence/index.ts:1-106` re-exports all + a `getStore(domain)` factory (5 domains). But there is **NO single uniform interface** — ~24 separate per-domain modules with inconsistent shapes (`JsonStore` instances vs hand-rolled object literals like `companyStore`/`projectStore` with `*ForUser` methods/`appDataStore` facade). Direct importers = 61 files under `src/`. The `appDataStore`/`StorageBackend` boundary is the ONE place that already meets the "one interface, swappable backend, single chokepoint" bar (and has a test seam `setAppDataBackendForTesting`).

FINDING: [storage-boundary] **One-interface effort = MODERATE.** Natural seam = the persistence barrel + `JsonStore`. A unified `TenantStore` interface (`list(ctx)/get(ctx,id)/create(ctx,item)/update(ctx,id,patch)/delete(ctx,id)` taking `UserContext`) would cover the bulk because 61 callers already import only from this module. The hard part is normalizing ~3 divergent store shapes and pushing the per-caller `.filter(i=>i.userId===user.id)` predicates DOWN into the store.

FINDING: [storage-leaks] Stores are NOT context-aware today, so the interface is "one place to construct" but not "one place that enforces tenancy" (`store.ts:47-150` take no `UserContext`). **Filesystem writes that BYPASS the persistence layer entirely** (direct `readFileSync`/`writeFileSync`): `integration-storage.ts`, `branding-save.ts`, `app-data-backups.ts`, `app-data-migration.ts`, `brand-asset-proxy.ts`, `brand-color-filter.ts`, `github/provider.ts`, `automation/platform-primitives.ts`, `automation/manifest-parser.ts`, `memory/seed.ts`, `artifacts-handler.ts` — these are the leaks that prevent "EVERY read/write behind one interface." `companyStore` is a hard-coded single-row store at one fixed path (`company.ts:29`) with no tenant key — it embeds the single-company assumption. `appDataStore` having no per-user partition is a deliberate design leak if the tenant unit is the user. Supabase access is a separate parallel path (raw `fetch`) the JSON interface does not cover.

### 2.3 — App-serving lifecycle + the artifact a coding operative must return

**The unit served is an artifact instance ("web_app"); its on-disk form is a sandbox project dir; its served form is that project's `dist/` mounted at `/apps/<id>/`.**

FINDING: [app-serving-lifecycle] First-build order (`execute-handler.ts:300-407`): `projectDir = join(sandboxRoot, 'user-${user.id}', project)` → `scaffoldApp(...)` → `appBuilder.build(...)` → `appBuilder.watch(...)` → `appRegistry.register(...)`. Sandbox root default `~/.ekoa/sandboxes`, env-overridable (`app-registry.ts:61`). `instance.data.projectDir` is the durable pointer used to re-resolve/heal later (`server.ts:1550`). On run complete Cortex stops the watcher, **wipes `dist/`, rebuilds from scratch with up to 2 retries, validates the bundle is real IIFE** (`execute-handler.ts:715-771`) — this authoritative final build is Cortex's, not the agent's. Then it auto-commits a per-artifact git snapshot + GitHub backup (`:776-792`).

FINDING: [app-serving-lifecycle] Build modes (`app-builder.ts`): plain HTML (root `index.html` → copy web-extension files, no esbuild, `:495-542`); JSX app (esbuild → `dist/bundle.js`+`bundle.css`, IIFE/browser/es2020, React from Cortex `node_modules`, `:232-273`); then Cortex generates `index.html` from a template (`:60-69`). Sandboxes **never `npm install`** — bare/CDN imports resolve locally first then fetch from esm.sh at build time and bundle (`cdnResolverPlugin`, `:136-223`). `AppRegistry` is an in-memory map keyed by appId (`distDir`, `projectDir`, `userId`, parsed `manifest`, hot-reloaded skills/recipes/instructions), chokidar-watching the project (`app-registry.ts:23-43,297-366`); startup scans `<sandboxRoot>/user-*/<project>/` for valid `manifest.json` (`:197-235`). Serve route `app.use('/apps/:appId', ...)` (`server.ts:2007-2158`): bare→trailing-slash redirect, slug→id canonicalization, `resolveAppDistDir`, lazy heal from the persisted instance (`tryRegisterAppFromInstance`, can git-clone the working copy back on a wiped box), "Building…" placeholder, `injectAppContext`, `express.static` with SPA fallback. URL is purely `/apps/<artifactInstanceId>/` or `/apps/<slug>/` — **no port allocation, no per-app process** (the older "spawn `npx serve`" design in `docs/cortex-architecture.md:102,535-539` is superseded). End-user SSO is server-mediated Microsoft OIDC+PKCE; the app JS never sees a token (`server.ts:712-764,837`). Server-side event delivery into an app is the **Artifact Backend (Layer 2)** runtime: `ArtifactBackendRuntime.invoke(...)` spawns a warm worker importing `dist-backend/backend.mjs`, calling `handler(input, ekoa)` with a credential-free capability-scoped RPC handle (`docs/artifact-backends.md:13-74`).

**The artifact shape a coding operative must RETURN.**

FINDING: [app-artifact-shape] The operative's product is a **populated project directory (source tree), NOT a built bundle** — Cortex owns the build. Canonical layout (`app-scaffold.ts:9-58`):
```
<projectDir>/
  manifest.json            (required: registry + startup discovery)
  frontend/src/index.jsx   (JSX entry)   — or —   index.html (root → plain-HTML mode)
  frontend/src/index.css
  skills/ recipes/ instructions/   (optional per-app content)
  backend/index.js         (optional Layer-2 server code)
  dist/  dist-backend/      (Cortex-generated; NOT authored by the operative)
```
`manifest.json` is the handoff contract: required `id` (== artifact instance id), `name`, `version`, `entryPoint`, `outputDir`, `type ∈ {jsx-app, html-app, static, static-html}` (`app-manifest.ts:22-49,116-143`); optional `description`, `dependencies`, `cortexApi.baseUrl`, `extends`, `backend:{entryPoint, handlers[]}`. Paths must be relative, no `..` (`:170-206`). For server-side events the operative declares `backend:{entryPoint:"backend/index.js", handlers:["onEmail"]}` and writes `(input, ekoa) => …` handlers using ONLY the `ekoa` capability handle — no creds, no imported `ekoa` (`docs/artifact-backends.md:49-74,150-167`). Browser-side the operative consumes injected globals `window.__EKOA_APP_ID` + `window.__ekoa.{fetch,list,get,create,update,delete,uploadFile,deleteFile,signIn,whoami,signOut}` (`server.ts:1658-1739`) instead of bringing its own backend/auth/data layer; data scoping is the auto-injected `X-Ekoa-App-Id` header. It may read `frontend/src/tokens.json` (resolved design tokens written at scaffold time).

FINDING: [app-handoff] The handoff protocol is **filesystem + record, not a return payload**: the operative leaves a populated `<projectDir>` with a valid `manifest.json`; Cortex's post-run hook rebuilds, validates IIFE, re-registers, commits, and the next `/apps/<id>/` request serves it (`execute-handler.ts:715-792`, `server.ts:2074-2157`). ## Part 3 — Automations and the local bridge

### 3.1 — The automation system (object / self-healing / trigger / run-state)

**Automation as a data object.**

FINDING: [automation-object] An automation is a user-facing spec — plain English, no code/locators/Playwright internals: `Automation { id, name, description, steps:Step[], inputSchema?, ownerUserId, trigger?, createdAt, updatedAt }` (`automation/types.ts:306-321`). A `Step` is the editable unit: `{ id, description, type:StepType, expectedOutcome?, ...discriminated extras }` (`:238-269`); `StepType` is a 9-way union `browser|verify|integration|sub_automation|navigate|wait|local_command|api_call|ekoa_action` (`:156-165`). Stored as one JSON file via `JsonStore` at `~/.ekoa/data/automations.json` (`persistence/automations.ts:16-18`), "no compilation, no pre-resolution". The deterministic-replay forms (`Locator`, `PlaywrightAction`, `PlaywrightAssertion`, resolved local/api/ekoa specs) are a hidden **cache layer** never surfaced to the user (`types.ts:23-128`). The cache is NOT a parallel store — resolved actions/assertions are saved as `StoredMemory` records (`type:'fact'`, `scope:'operational'`) keyed by tags `automation:<id>`+`step:<id>`+`action-cache`|`assertion-cache`, payload in `attachments.payload` (`automation/cache.ts:1-15,32-53,136-167`); per-record key is the `fingerprintKey`.

**Self-healing (rehearsal · fingerprint · replan-on-failure).**

FINDING: [automation-self-healing] Two modes share one loop (`runOrRehearse`): `runAutomation` (deterministic replay) and `rehearseAutomation` (the per-step loop may mutate the spec) (`engine.ts:222-254`). The self-correction (fixer) path runs in BOTH modes — only difference is rehearsal persists refined steps, normal applies patches transiently (`:681-687`). Per-step two-tier resolution: Tier 1 cache hit → `lookupActionCache(...)` → execute cached `PlaywrightAction` (`:1294-1332`); Tier 2 vision → `resolvePlaywrightAction` (Opus 4.7, max effort) → execute, `writeActionCache` (`:1334-1413`). **FINGERPRINT** is the cache key discriminator: `PageFingerprint {origin, pathname, pathSuffix, titleHash, headingHash, domShapeHash, viewport}` (`types.ts:142-150`, `fingerprint.ts:32-114`) — stable across runs of the same page state yet discriminating same-shape SPA templates. **REPLAN-ON-FAILURE** (the rehearsal fixer): on `status:'failed'`, `proposePatch` (Opus 4.7, max effort, post-failure screenshot + a11y tree) returns ONE local `RehearsalPatch` (`insert_before|replace_current|skip_current|abort|pause_for_user`, `rehearsal.ts:68-76,134-181`); "the fixer does not get to rewrite downstream steps" (`:8-10`). Loop retries the SAME index against the patched plan (`engine.ts:880-947`). **Budget-capped**: `REHEARSAL_BUDGET {maxFixerCalls:25, maxWallClockMs:4min, maxPatchesPerIndex:5, maxNormalPauses:5}` (`rehearsal.ts:350-361`) → outcomes `ok|budget_exhausted|stuck|aborted|failed`. **Human-in-the-loop** (3 detection layers before the slow fixer): verifier `humanAction`, regex `detectHumanActionable()` (CAPTCHA/MFA/payment/login, `:384-428`), Haiku classifier backstop (`engine.ts:586-640`) → synthetic `pause_for_user` → `pauseRunForUser` blocks on the resume signal (polled 250ms, no timeout). Normal runs never mutate the saved spec (`engine.ts:2029-2044`). Cache eviction (`evictCacheForFingerprint`) forces re-resolution on the next run (`cache.ts:258-282`).

**Scheduling / triggered — what drives a run.**

FINDING: [automation-trigger] At most one `trigger` per automation, 3 kinds: `manual` / `webhook {triggerId, integrationKey, eventName}` / `listener {triggerId, integrationKey, pollAction, pollIntervalMs}` (`types.ts:278-318`). Trigger metadata is a separate JSON store `~/.ekoa/data/triggers.json` (`StoredTrigger` with `secretCiphertext`, `registrationState`, `pollConfig`, EventEmitter bus, `persistence/triggers.ts:40-141`); dispatch can target `automation` (default) or `artifact-backend {artifactId, entrypoint}`. **The engine loop is the `TriggerDispatcher`** — it drains a durable event queue (not a cron): wakes on `event_enqueued` + a 5s safety-net `setInterval` (`trigger-dispatcher.ts:147-208`); each tick `claimNextEvent()` atomically flips pending→dispatching via `UPDATE ... RETURNING` (no double-dispatch), up to `maxConcurrent=4` (`:46,197-208`, `event-queue.ts:188-199`). `dispatchEvent` loads the trigger, for `automation` calls `runAutomation(id, {ownerUserId: trigger.ownerUserId /* NEVER from payload */, triggeredBy:webhook|listener, triggerEvent, traceId})` (`:90-134`, security invariant `engine.ts:81-89`). Producers: (1) **webhook** `POST /hooks/<triggerId>` HMAC-verified + audited + `enqueueEvent(...source:'webhook')` (`webhooks-handler.ts:29-133`); (2) **listener** `ListenerSupervisor` — long-lived per-trigger timer polling `executeUserIntegrationAction({pollAction, args:{since:cursor}})` then enqueue one row per new item, reschedule at `pollConfig.intervalMs ?? 60_000` (`listener-supervisor.ts:141-256`) — interval-polling, not cron. Direct drivers: `call_automation` tool (`triggeredBy:'agent'`) and the UI run button (`triggeredBy:'user'`).

UNKNOWN: [automation-trigger] No cron-expression / time-of-day scheduler exists — the only periodic driver is the listener's fixed `pollIntervalMs` plus the dispatcher's 5s sweep. If a calendar-style scheduler exists it is outside the files in scope.

**Where run state lives.**

FINDING: [automation-run-state] Live per-run record `RunRecord { id, automationId, startedAt, endedAt?, status, inputs, steps:StepRecord[], triggeredBy, parentRunId?, awaitingIntegration?, kind?, rehearsalSummary?, pauseRequest?, consentRequest? }` (`types.ts:442-469`); `RunStatus` 8-way `running|completed|failed|cancelled|awaiting_integration|paused_for_user|awaiting_consent|awaiting_daemon` (`:432-440`). The engine persists at every step boundary (create `running` → update `steps` per step → finalize, `engine.ts:278-289,459-472,1736-1753`). **Trigger/queue state is a SEPARATE SQLite store** (better-sqlite3 WAL) at `~/.ekoa/data/triggers.db` — tables `events` (`pending→dispatching→dispatched|failed|dead`, idempotency `UNIQUE(trigger_id, dedup_key)`), `listener_state`, `webhook_audit` (`event-queue.ts:1-161`). Crash recovery is at the queue level: `recoverStuckDispatches()` moves rows stuck in `dispatching` >10min back to `pending` (`:363-371`); retries `[30s,2m,10m,1h,6h]` then `dead` after `MAX_ATTEMPTS=5`. ### 3.2 — The local bridge (design / status / capabilities / protocol / driver)

> The bridge spans **two repos**: Cortex = hosted WS *server*; the `ekoa-local` daemon (at `/Users/ggomes/dev/ekoa-local/`, a sibling — NOT under `ekoa-dev/`) *dials out*.

FINDING: [bridge-design] The bridge is a **two-faced RPC over one outbound WebSocket**: the daemon on the user's machine dials OUT to hosted Cortex (NAT-friendly; keeps the real browser/fingerprint local while the brain is remote), so Cortex is the WS server and the daemon is the client (`cortex/src/bridge/server.ts:2-6,89-90`). ONE local execution layer (daemon), TWO hosted drivers ("faces") both ending at the same daemon capabilities — the **agent face** (Claude Agent SDK loop, open-ended) and the **executor face** (typed automation engine) (`ekoa-local/docs/architecture.md:5-9`; `connection.ts:11-15`). One socket multiplexes both, correlated by uuid: `invokeCapability → agent_tool_call → agent_tool_result`; `runStep → exec_step → step_start/step_progress → step_result` (`connection.ts:11-14`). Deep "why" (agent face): Cortex keeps the `claude_code` preset (authorizes Opus/Sonnet on OAuth) but MUST NOT let the preset's built-in Read/Bash/Edit run on the *Cortex host* — each daemon capability is registered as an SDK custom `tool()` whose handler RPCs over the bridge (`daemon-tools.ts:5-11`).

FINDING: [bridge-status] **Shipped and wired, not a spike.** Server attached unconditionally: `attachBridgeServer(httpServer)` (`server.ts:254`); active count surfaced as `bridgeConnections` (`:382`). Token-mint `POST /api/v1/bridge/token` (`:943-956`) and `POST /api/v1/agent-face/run` (`:1111-1158`) are live; daemon connects on startup `void channel.connect()` (`ekoa-local/src/index.ts:75`). Maturity: design doc marked "determined"; schemas carry numbered security-review dispositions (M02/M03/M04/M11/M28/M32/M33/M34/M43); a substantial committed test suite exists (`bridge/__tests__/{server,correlator,registry}.test.ts`, daemon-side `client.test.ts`, `control-channel.test.ts`); a bridge-e2e gate-status record at `ekoa-dev/docs/autothing/slices/bridge-e2e/gate-status.json`. A model-free transport proof endpoint `POST /api/v1/bridge/debug-invoke` is fail-closed behind `EKOA_BRIDGE_DEBUG_INVOKE==='true'` (`:963-979`).

FINDING: [bridge-capabilities] Exactly **three capabilities** via a `CapabilityRegistry` (`ekoa-local/src/capabilities/contract.ts:31-38`): **bash** = arbitrary argv exec on the user's machine (no shell, cwd default home, mandatory `timeoutMs`, 1 MB output cap, `bash.ts:1-66`); **fs** = read/write/edit/list/glob/grep on the real filesystem (discriminated on `op`, optional `root` confinement blocking `..`, 1 MB cap, `fs.ts:1-67`); **browser** = drives the user's REAL local browser via Playwright (HEADED, PERSISTENT per-owner context at `~/.ekoa/sandboxes/<owner>/automation-profile/`, anti-bot stealth, ~25 actions, composite `page` observation, `browser.ts:1-46`). **Act-and-observe contract**: every capability returns a `ResultEnvelope` (never bare success); `CapabilityRegistry.invoke` validates input with its own zod schema and never throws (`contract.ts:44-68`). Inputs are **re-validated daemon-side as the single source of truth** — Cortex's zod shapes only mirror them (`daemon-tools.ts:14-18`).

FINDING: [bridge-protocol] Transport = a single WebSocket (`ws` both ends), Cortex `noServer:true` hooking `httpServer.on('upgrade')` scoped to `/api/v1/bridge/connect/` (`server.ts:31,63-73`); daemon opens `new WebSocket(url, {headers:{authorization:'Bearer <token>'}})` (`client.ts:68-71`). All frames JSON text, one object per message, discriminated by field `t`. URL `/api/v1/bridge/connect/<connectionId>` (single path segment). The two protocol files are deliberate mirrors in lockstep (`cortex/src/bridge/protocol.ts` mirrors the canonical `ekoa-local/src/protocol/control-channel.ts`). **Hosted→Daemon**: `agent_tool_call {callId, capability, input}`, `exec_step {stepId, runId?, capability, input}`, `cancel {id}`. **Daemon→Hosted**: `agent_tool_result {callId, result}`, `step_start`, `step_progress`, `step_result`, `log`. **ResultEnvelope** `{ ok, observation?{kind, text?, screenshotB64?, data?}, error?{message, retryable?}, meta? }` (kinds `stdout|fs|dom|screenshot|text|page|none`). **Never-trust-the-wire**: inbound frames zod-parsed, dropped on failure without settling (`connection.ts:209-226`). **Correlation/timeouts**: pending map keyed by callId/stepId; agent calls 120 000 ms default, steps 600 000 ms; `step_progress` rearms the step timer; on timeout Cortex first sends `cancel(id)` before rejecting (review M11). **Liveness**: 30s ping/pong, `terminate()` on no-pong-within-10s, reconnect exp-backoff 1s→30s with jitter (M28/M34/M43).

FINDING: [bridge-driver] The daemon initiates the socket; **hosted Cortex initiates work**, resolving "the daemon for this user" by ownerUserId via `getBridgeConnection(userId)` → `getConnectionByOwner` (`server.ts:110-115`, `registry.ts:46-58`). Agent-face flow: `POST /api/v1/agent-face/run` → 409 if no daemon, else traceId + async SSE; the SDK loop registers MCP server `ekoa-local` (bash/fs/browser), each handler calls `conn.invokeCapability(...)` → daemon → observation as MCP `CallToolResult` (`daemon-tools.ts:205-256,150-182`). FINDING: [bridge-auth] **Two-tier credential**: durable = the user's Cortex session JWT; from it the daemon mints a short-lived **bridge token** before every (re)connect via `POST /api/v1/bridge/token` (`mint.ts:11-30`, `server.ts:943-956`). Daemon scrubs `EKOA_CORTEX_JWT`/`EKOA_CORTEX_TOKEN` from every child env (M02). **"Pairing" model**: no interactive pairing handshake — possession of a valid session JWT is sufficient; `connectionId` identifies an owner/device. Cross-user isolation lives in the registry (per-owner index, identity-guarded unregister, multi-device).

UNKNOWN: [bridge-status] Whether the bridge runs in any production deployment vs local dev only — `ekoa-local/src/index.ts:43-49` hardcodes a localhost Cortex default; the only no-LLM proof is dev-gated.

UNKNOWN: [bridge-driver] The root `ekoa-dev/CLAUDE.md` contains no bridge/daemon sections; the authoritative narrative is `ekoa-local/docs/architecture.md` §2–§5.

### 3.3 — Bridge-vs-outpost finding (cross-repo analysis)

> Both move "execution on machine B" under "control from process A" over a WebSocket where the **executor dials out** to the controller (NAT-friendly). Same archetype; opposite trust postures.

FINDING: [bridge-vs-outpost] Same archetype — controller drives capability execution on a remote node over a WS where the executor dials OUT (so the controller is the WS server); roles do NOT invert. Cortex: daemon dials Cortex (`cortex/src/bridge/server.ts:3-6`). Garrison: the outpost-host is the WS server, `garrison-outpost-bridge` daemons dial in (`scripts/outpost-host.mjs:1-12`, `bootstrap-outpost.sh:11`). The role mapping aligns 1:1 (**Cortex hosted brain ≙ Garrison machine/Operator**; **ekoa-local daemon ≙ garrison-outpost-bridge remote daemon**) — the same sentence with driver/executor on opposite physical machines: Cortex's driver is the cloud and the executor is *your* laptop; Garrison's driver is *your* laptop and the executor is *another* laptop. **The code shape is identical; the privacy gradient is inverted.** Cortex's driver has two faces over one socket; Garrison has two consumption faces split across transports (blocking HTTP RPC + streaming PTY WS) not unified into one correlator.

FINDING: [bridge-outpost-transport] Cortex = **single-transport** (one WS, JSON frames discriminated by `t`, strict zod validation, uuid correlation in one pending-map). Garrison = **split-transport over a broker**: (a) blocking RPC over HTTP POST `/outposts/:name/rpc` (`outpost-host.mjs:478-490`); (b) interactive PTY over WS `/outposts/:name/io` (`outpost-stream.ts:24,117-131`). **Topology**: Cortex 2-hop (in-process correlator); Garrison 3-hop (separate broker at 127.0.0.1:3702). Frame discriminators differ entirely (`t:` flat vs `type:`+`version`+`payload`). Streaming: Cortex `step_progress` on the same socket *rearms* the step timeout; Garrison fans out `process.output` by handle to a subscriber set, fixed `RPC_TIMEOUT_MS=10_000` with no progress extension. Cancellation: Cortex has first-class `cancel` frames mapped to the daemon AbortSignal and aborts on timeout; Garrison's HTTP RPC has **no cancel** and the PTY path deliberately does **not** reap remote PTYs on subscriber disconnect.

FINDING: [bridge-outpost-auth] **Different auth classes.** Cortex = short-lived signed JWT/HMAC (600s TTL, audience `ekoa-bridge`, `sub`+`connectionId`, `Authorization: Bearer`, connectionId path-bound, `auth.ts:25-85`). Garrison = a **long-lived static pre-shared bearer token** (no expiry, no signing): first WS frame `{type:"auth", payload:{token, machine_name}}`, looked up in a plaintext registry (`outpost-host.mjs:136-151`), stored cleartext in `~/.garrison/outpost-registry.json` + `~/.garrison-outpost/config.json`. **Pairing**: Cortex identity-derived (token `sub` IS the owner, registry indexes by ownerUserId); Garrison name-based provisioning, **no user/owner concept** — machines, not users. **Auth-stage timing**: Cortex at the HTTP upgrade (401 before WS opens); Garrison post-upgrade as the first frame; Garrison's `/io` subscriber socket has **no auth whatsoever** — anyone reaching 3702 can spawn a PTY on any outpost.

FINDING: [bridge-outpost-trust] **The trust gradient is inverted.** In Cortex the *executor* (user's machine) is the privacy-sensitive asset and the *driver* (multi-tenant SaaS) must not be trusted with raw user data; in Garrison the *driver* (your laptop) is the trusted single-user asset. **Multi-tenant isolation is mandatory in Cortex, absent in Garrison**: Cortex has per-owner registry indexing + ownership-mismatch rejection; Garrison has one global machine namespace, no tenant, no per-user scoping. **"Never trust the wire"** is load-bearing in Cortex (zod validation, drop-on-invalid) and a no-op in Garrison (`JSON.parse` + trust `frame.payload.handle`). **Bridge is HMAC-secret-grade; outpost is LAN/tailnet-grade** (Garrison leans on Tailscale confinement + cleartext token; Cortex assumes a hostile public internet). **Lifecycle/abort encodes the trust difference**: Cortex aborts orphaned destructive work on the tenant machine (it is liable); Garrison deliberately keeps remote PTYs alive (the operator owns both ends).

FINDING: [bridge-vs-outpost] **VERDICT: ONE mechanism could serve both at the wire-shape level but NOT as-is** — same archetype, opposite ends of the trust spectrum; the gap is auth/isolation/validation, not topology. **Reusable core**: executor-dials-out-WS-server (NAT-friendly), capability-call RPC + streaming-step output, per-call-id correlation with pending-map + timeout, reconnect/online-offline signalling, JSON discriminated control channel. **Hard differences blocking naive unification**: (1) auth class (signed short-TTL JWT vs static plaintext token); (2) tenancy (mandatory per-owner isolation vs single global namespace); (3) wire trust (strict zod vs unvalidated parse); (4) topology (2-hop in-process vs 3-hop broker + split transports); (5) unauthenticated surface (Garrison's `/io` PTY has no auth); (6) abort liability (Cortex aborts orphaned work, Garrison keeps PTYs alive). **To unify**: extract a shared executor-dial-out WS correlator + capability/step protocol (the Cortex `BridgeConnection` shape is the better base — single socket, validated, cancel-capable), then make **auth and tenancy pluggable policy-injection points** — Garrison plugs a LAN-trust/no-isolation/static-token policy, Cortex plugs the JWT/owner-scoped/strict-validation policy. Unification is feasible **only if auth/isolation/validation are policy injection points, never the default** — Cortex's secure defaults must not be downgraded by sharing code with the localhost-trusting outpost.

UNKNOWN: [bridge-vs-outpost] The remote-side daemons themselves were not on disk (`ekoa-local/src/control/client.ts` not in the explored tree; Garrison's `garrison-outpost-bridge` cloned at runtime from `github.com/gongiskhan/garrison-outpost-bridge`), so executor-side capability implementations and any executor-side sandboxing are inferred from the controller/broker contract only.

### FLAG — where running reasoning in Garrison would break governance or risk tenant-data exposure

FINDING: [governance-external-risk] **FLAG 1 — Billing/quota is bypassed by external reasoning.** Billing is gated only at the two Cortex agent entry points and metered *after* the SDK call (`index.ts:334-348`, `execute-handler.ts:561`, `external.ts:1213-1240`). If reasoning + token spend run in Garrison and only side-effecting tool calls return to Cortex, spend is invisible to `checkBillingAllowance` and the pre-alpha hard cap (`middleware.ts:64-73`) is bypassed. **Mitigation**: route every operative turn through `/api/v1/request` (or an equivalent metered gateway) so Cortex still accounts the tokens.

FINDING: [governance-external-risk] **FLAG 2 — The autonomous-loop containment does not transfer.** The chat/build path has **no per-tool-call server interception** (no `canUseTool`, no registry call from `external.ts`); containment is pure SDK-query config (`bypassPermissions` + `cwd:projectDir`). If reasoning runs in Garrison, only Garrison's own sandbox/`allowedTools` stop arbitrary host commands. The strongest containment (agent-face default-DENY `canUseTool`, `agent-face/index.ts:234-248`) is entirely in-process. **Mitigation**: Garrison must reconstitute equivalent tool confinement; do not assume Cortex's gate applies.

FINDING: [governance-external-risk] **FLAG 3 — Tenant-data exposure if the operative reaches state directly.** Auth/scope/ownership/credential governance survives ONLY if Garrison is forced to act through Cortex's authenticated handlers + governed-tool registry. The danger surface is (i) any path where the operative gets filesystem/host access outside Cortex's handler-mediated tools (the JsonStore files at `~/.ekoa/data/*`, the daemon directly), and (ii) the already-unauthenticated `POST /api/apps/:appId/compile` (`server.ts:398`). **Mitigation**: never hand the operative raw store paths, the `ENCRYPTION_KEY`, decrypted credentials, or the privileged `integrations execute` intent; expose only `{success,status,data}`-returning tools.

FINDING: [integrations-tool-contract] **FLAG 4 — Pre-existing multi-tenant scoping bug must be fixed before any external boundary.** The platform integration path `getValidTokens(provider)` does first-match on the whole connection store, ignoring `ctx.userId` (`platform-integration-call.ts:294-295`). Safe today only because deployments are effectively single-tenant for platform connections; an external multi-tenant boundary would leak tenant A's Google/M365 connection to tenant B. Fix to mirror the user-defined path's `ownerUserId` filter first.

FINDING: [governance-external-risk] **FLAG 5 — SDK sealing + managed-OAuth posture is replaced, not inherited.** `settingSources:[]` (no developer `~/.claude`) and managed-OAuth-only / API-keys-blocked are properties of how Cortex spawns the SDK. A Garrison operative brings its own auth/model posture (the user's Max account per Garrison's CLAUDE.md), so "no developer profile, no API-key billing" does not transfer — it becomes Garrison's responsibility to match.

**Net Cortex-side conclusion:** the reasoning of all four agents is movable to a Garrison operative — integrations cleanest, coding most coupled — *provided* every tenant-state and infra touchpoint becomes a Cortex-exposed, JWT-scoped tool/MCP (the `platform-integration-call` + App Data + build/scaffold seams first), and *provided* billing, containment, and the platform-tenant-scoping bug are explicitly re-plumbed. The data-boundary governance survives; the agent-process governance must be reconstituted in Garrison. The bridge and the outpost are the same archetype and could share a wire-level core only if auth/tenancy/validation are pluggable policy with Cortex's strict defaults never downgraded.
