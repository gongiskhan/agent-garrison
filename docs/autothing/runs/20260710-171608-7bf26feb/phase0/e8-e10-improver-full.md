E8 — THE IMPROVER

FINDING-E8-fittings: Two fittings exist — the LIVE one is `improver` (faculty observability, own-port 7088, provides automation-runner; fittings/seed/improver/), which owns the review queue + apply API; `improver-nightly` (faculty sessions, disabled-by-default, fittings/seed/improver-nightly/) is a separate proposal-only markdown-artifact generator, legacy.

FINDING-E8-registration: Behaviours are "rules" = plain `lib/<name>-rule.mjs` modules exporting a PURE analyze fn + a `run*Rule({now})` collector wrapper, imported at scripts/improver.mjs:40 and invoked inside `runSkills()` (improver.mjs:336-345), pushing proposals into the shared queue via `upsertQueue`. There is NO live watcher/event-subscription — rules are POLL-based, run once per nightly invocation, reading files off disk. So "watch interference events" / "the Probe" plugs in as a new `lib/<name>-rule.mjs` (collectors read the interference/run files, pure analyze emits proposals in the standard shape) wired one line into runSkills(); "watch" means "re-scan these files each nightly run", not a daemon.

FINDING-E8-queue-location: Feedback queue = `~/.garrison/improver/review-queue.json` (QUEUE_FILE = $IMPROVER_DATA|$GARRISON_HOME/improver/review-queue.json), plus full per-proposal files at `~/.garrison/improver/proposals/<id>.json`; autonomy at `~/.garrison/improver/autonomy.json`, last run at `last-run.json`. On THIS box only `~/.garrison/improver/dream-config.json` exists — no queue yet (improver has never completed a nightly run here).

FINDING-E8-queue-schema: Full queue record (review-queue.mjs `enqueue`): `{ id, rule, targetClass, claim, diff, decision, applyVia, status, at, appliedAt?, evidence?, rejectedAt?, reapplyFailureReason? }`; status ∈ pending|applied|rejected|reapply-failed. Canonical proposal shape from every rule: `{ id, rule, targetClass, claim, evidence{}, diff, decision, applyVia, at }`. NOTE a thin-vs-full mismatch: improver-core.mjs `upsertQueue` writes a THINNER index row `{id,rule,targetClass,claim,status,at}` (no diff/decision/applyVia) while review-queue.mjs `enqueue` writes the full one — two writers, slightly different shapes; the full proposal always lives in proposals/<id>.json.

FINDING-E8-appends: Appends are NOT atomic-append — saveQueue (and improver.mjs) rewrite the WHOLE JSON array via writeFile each time; idempotent by id (enqueue skips existing, upsertQueue filter-then-push). The only append-only JSONL in this area is the modes switch-log (`~/.garrison/.../switch-log.jsonl`), unrelated to the improver queue.

FINDING-E8-trigger: Nightly is scheduler-fitting-driven, NOT OS cron. `improver` setup.sh registers cron `30 3 * * *` via the scheduler CLI; `improver-nightly` setup.sh registers a DISABLED job `improver-nightly` (`17 3 * * *`) via `scheduler.mjs register improver-nightly <cron> --disabled` (jobs file `<composition>/data/scheduler-jobs.json`). Manual run: `scheduler.mjs run-now improver-nightly` or the own-port UI POST /api/run-now.

FINDING-E8-approve-flow: Approve/reject served by fittings/seed/improver/scripts/server.mjs (port 7088): GET /api/queue → {queue,autonomy,promotionThreshold}; POST /api/proposals/:id/apply → applyWithRetry (never-clobber baselineSha → 409 → reconcile) → status "applied"+evidence, records rule "accept"; POST /api/proposals/:id/reject → status "rejected", records "reject". Autonomy state machine (improver-core.mjs): manual default; 5 consecutive accepts (PROMOTION_THRESHOLD, zero rejects) → "promotion-suggested"; human toggles PUT /api/autonomy or POST /api/autonomy/promote; ANY reject/revert resets streak and instantly demotes auto→manual.

FINDING-E8-ghost-edits: Composer ghost-edit rendering is DOCUMENTED INTENT, not wired. orchestrator-policy proposals are explicitly NOT applied via the improver's local targetFileFor() (server.mjs:158); their applyVia = "PUT /routing (baselineSha, Orchestrator fitting)". Comments say they "render in the composer as ghost edits" but grep of src/ for orchestrator-policy / port 7088 finds NO Garrison-side consumer reading the improver queue to render routing ghost edits. Only the improver's own dist/ review UI renders apply/reject today.

FINDING-E8-policy-rule: orchestrator-policy-rule.mjs reads two read-only inputs — run outcomes = every `~/.garrison/runs/<project>/<runId>/evidence-index.json` (+per-slice gate-status, cap 200, collectRunOutcomes) and autothing friction logs = `~/dev/<project>/docs/autothing/friction-log.md` lines (cap 300, collectFrictionLines). It aggregates per `${workKind}::${gateKey}` counting {runs,clean,failed,skipped} (status read from slice.gateStatus.gates[].status||verdict||result) and proposes 4 conservative kinds: (a) gate clean ≥5 runs → step effort DOWN; (b) gate failed ≥50% of ≥3 runs → step UP; (c) gate skipped in ALL ≥3 runs → turn phase OFF in that workKind's phasePlan; (d) a skill named in ≥3 friction lines → review/swap its phase binding. All applyVia "PUT /routing (baselineSha, Orchestrator fitting)", never auto-applied.

E10 — SESSION TAGGING (attended vs idle vs pool)

FINDING-E10-owner: The dev-env fitting owns session tagging at `~/.garrison/sessions/state.json` (writer fittings/seed/dev-env/scripts/state.mjs). Per-session fields: branch, worktreePath, ports, createdAt, lastStatus, lastStatusAt, lastHookEvent, id (garrison-local uuid), claudeSessionId (Claude UUID), title, status:"active", source, openedInDevEnv.

FINDING-E10-hook-sessionid: session_id IS available to hooks via the STDIN PAYLOAD, not an env var. dev-env/scripts/install-hooks.mjs installs global hooks into ~/.claude/settings.json for UserPromptSubmit/Stop/Notification/PostToolUse, matcher "" (all sessions); buildHookCommand does `payload=$(cat)` and POSTs it verbatim to http://127.0.0.1:7086/_hook?event=<name>. Payload carries `session_id`, transcript_path, hook_event_name, cwd. There is NO `CLAUDE_CODE_SESSION_ID` anywhere in the repo; the only Claude env var referenced is `$CLAUDE_PROJECT_DIR` (cwd fallback). So a Stop hook reads the session id as payload.session_id (applyHookEvent reads body.session_id||body.sessionId, state.mjs:458).

FINDING-E10-status: statusFromHookEvent (state.mjs:139): UserPromptSubmit/PostToolUse→"working", Stop→"idle", Notification→"waiting". applyHookEvent matches an existing row by EXACT claudeSessionId first, else worktreePath, else auto-creates source "hook-autocreated". A 60s WORKING_IDLE_FALLBACK demotes stale "working"→"idle" unless the PTY screen shows a live turn.

FINDING-E10-attended: The ONLY positive "attended by operator" tag today is `source: "dev-env-open"` + `openedInDevEnv: true` on the state.json row (set by openSessionByClaudeId, state.mjs:441 — operator opened it as a Dev Env tab). Ambient operator sessions are merely `source:"hook-autocreated"` / openedInDevEnv:false — indistinguishable from anything else (all 15 live rows on this box are exactly that). At the TURN level the http-gateway distinguishes `X-Garrison-Origin: ui-tab|channel` (orchestrator-prefix.mjs) but that is per-turn, never persisted per session.

FINDING-E10-dispatched: Engine-dispatched workers = "soul" sub-sessions tracked in the http-gateway's IN-MEMORY SessionRegistry (fittings/seed/http-gateway/scripts/lib/session-registry.mjs), keyed by the `--session-id <uuid>` the gateway mints (spawn-soul.mjs buildClaudeArgs). Per-soul fields: soul (engineer|architect|assistant|researcher|companion), mode ("headless"|"interactive"), status, parentSessionId, worktreeId, channel, terminalTabId, tier. Queryable over HTTP GET /sessions (serialize → session_id, soul, mode, parent_session_id, worktree_id, terminal_tab_id) or gateway MCP list_souls. NOT persisted to disk (only the orchestrator's own id → sidecar). So a Stop hook can only classify a session_id as a dispatched soul by querying the gateway's live session list and matching session_id.

FINDING-E10-pool: Idle pool = WarmPtySessionPool (packages/claude-pty/src/warm-pool.mjs). Pre-spawned OperativePtySession records; status() exposes {id, state:"available"|"checked-out", claudeSessionId} IN-MEMORY ONLY. No on-disk registry, no session-id-keyed marker, and a freshly-warmed session has NO claudeSessionId until its first turn. So there is NO signal a Stop hook could read to identify a pool member today — a genuine gap.

FINDING-E10-no-env-marker: Spawned soul/pool sessions carry NO distinguishing env var. session.mjs spawn (stripNestingMarkers) only deletes CLAUDECODE/ANTHROPIC_API_KEY/ANTHROPIC_BASE_URL and otherwise inherits process.env; no GARRISON_SOUL/GARRISON_POOL/role is injected. Net: a Stop hook cannot read the role from environment — it must cross-reference payload.session_id against state.json (attended: source/openedInDevEnv) and/or the gateway's in-memory session list (dispatched soul); pool membership is currently unobservable to a hook.
