Nightly is scheduler-fitting driven (not OS cron). Approve/reject goes through `server.mjs` with a never-clobber baselineSha apply + a per-rule manual→auto autonomy state machine (5-accept promotion, instant demotion). Notably, the "composer ghost edits" for orchestrator-policy proposals are documented intent only — no Garrison-side consumer of the improver queue exists in `src/` yet. `orchestrator-policy-rule.mjs` reads evidence-index.json run outcomes + autothing friction logs and proposes effort up/down, phase-off, and skill-binding-review edits via `PUT /routing`.

**E10 — Session tagging.** The dev-env fitting owns `~/.garrison/sessions/state.json`. The session id is available to a Stop hook via the stdin payload's `session_id` field (hooks are installed globally with matcher "" and POST the verbatim payload to port 7086); there is no `CLAUDE_CODE_SESSION_ID` env var anywhere. The three-way distinction is only partially observable: **attended** has a positive tag only when opened as a Dev Env tab (`source:"dev-env-open"` + `openedInDevEnv:true`) — ambient operator sessions are indistinguishable `hook-autocreated` rows; **dispatched soul workers** live in the http-gateway's in-memory `SessionRegistry` (soul/mode/parentSessionId, queryable over HTTP but not on disk); **idle pool** members (`WarmPtySessionPool`) have no on-disk registry, no session-id marker, and no env var — a genuine gap where pool membership is currently unobservable to a hook.
