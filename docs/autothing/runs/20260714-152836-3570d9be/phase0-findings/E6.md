## Finding

The V3-planned "5 lifecycle events" bridge (S1c/WS1c: created/needs-input/blocked/failed/finished flowing to the originating thread) was NEVER built as a slice - no slices/S1c dir, no WS1c row in LANDING.md, no MARATHON3-WS1C ledger sentinel; it is part of the run's PARTIAL tail. What shipped post-V3 (commit be5a9f1, 2026-07-14 "channel outcome feedback") is a 3-event subset: (1) created - the gateway's inline "Registered as a run ... Card: <url>" ack, (2) finished and (3) needs-attention - emitted edge-triggered from the ONE card write path (board.mjs saveCardCAS -> notifyOriginTransition) and POSTed to the originating web-channel thread (needs-attention conflates the planned needs-input/blocked/failed trio into one message). Needs-input reply routing exists ONLY for live interactive turns via the AskUserQuestion path (D28): a JSONL watcher in gateway-pty emits a `tool` SSE event, the channel renders tappable buttons, and the answer POSTs back to /chat/answer which drives the live TUI picker via key sequences; the kanban engine's dispatch stream ignores `tool` events entirely, and a thread reply to a parked card does NOT resume it (cardIsLive excludes needs-attention, so the reply registers a fresh card). Subscribers get SSE (per-turn /chat/stream: open/chunk/tool/done/error; persistent /claude/stream rich events; board /cards/:id/watch log tail), files (card events[] timeline, thread JSONs, ui-fittings status files), and a READ-ONLY pull-only garrison-control MCP tool - there is no MCP event subscription.
