# FINDING-E4

## Finding

Post-V3, per-duty evidence is split across three stores: (1) the kanban card store ~/.garrison/kanban-loop/cards/<cardULID>/ (card.json pointers+events timeline, per-iteration log-N.md with the full operative reply, brief.md), written by the engine; (2) the run dir ~/.garrison/runs/<project>/<runULID>/ (D19, minted per card on first agent-list entry), where the OPERATIVE self-writes gate-status.<phase>.json sidecars / a run-level gate-status.json gates{} map / slices/<sliceId>/gate-status.json, FLOW_PLAN.md, touch-set.json, and an evidence/ bundle (enforced only for walkthrough); (3) the legacy doorway layout docs/autothing/runs/<YYYYMMDD-HHMMSS-hex8|ULID>/ with RUN_SPEC.md, LANDING.md, status.json, owner.lock.json, evidence-index.json, report-summary.md, slices/, security/, codex-checkpoint/. What is MISSING for a standard per-duty summary: the engine writes nothing into the runDir itself and only checks gate-record EXISTENCE (next_phase key), never summary content; there is no per-duty summary file contract (summary/notes are free-form optional fields inside heterogeneous gate records); card.sessionIds and videoUrl are defined but never written by anything (transcript/video links never resolve); and on the engine path a card reaching `done` triggers only a channel notify with a 280-char lastReply snippet — LANDING.md/report-summary.md/evidence-index.json exist only on the legacy doorway path, and the live compiled policy has EMPTY phaseSkills.bindings/workKinds/phasePlans so no report duty is bound into any sequence (observed sequences end plan→implement→review→test→done with no summary step).

GAPS for a standard per-duty summary artifact (what was done / decisions / outputs at every duty completion):
1. No summary CONTENT contract: hasPhaseGateEvidence checks only that a gates key/sidecar EXISTS and parses; gateEvidenceNextList reads only next_phase. summary/notes/artifacts fields are free-form and optional — nothing engine-side requires or validates them, and shapes are heterogeneous (sidecar vs gates{} vs phases{}, summary vs notes vs detail, camelCase vs phase-id keys).
2. The engine itself writes NOTHING under runDir — every runDir artifact is operative-self-attested; the only disk verifications are the gate record and (walkthrough-only) evidence/. Non-walkthrough duties can complete leaving nothing but a minimal {status,next_phase} record (observed: 01KXGGTWQWRP... runs have only FLOW_PLAN.md + 4 tiny gate sidecars, no evidence/, no summary).
3. Dead pointer fields: card.sessionIds is initialized [] (board.mjs :129) and NO writer exists anywhere in the repo (grep across src/ + fittings/), so transcript links never resolve; card.videoUrl has no writer either (board.mjs :131 says "set by the Walkthrough list" — never implemented); sliceId stays null on engine-driven cards so the gateMarkers link never resolves.
4. Terminal completion writes no summary: on `done` the only output is notify-origin.mjs outcomeMessage (:51-66) — lastReply snippet (≤400 chars) + videoUrl (always null) + card link. evidence-index.json / LANDING.md / report-summary.md belong to the legacy doorway pipeline only; the engine path never produces a run-level rollup.
5. No report/summary duty in the live model: the compiled policy at ~/.garrison/orchestrator/policy.json has phases[] including "report" but phaseSkills.bindings = {}, workKinds = {}, phasePlans = {} — so skillForPhase returns null for every phase, railForCard falls back to all-on, and the resolved sequences (model.json) for real cards were plan→implement→review→test with no report step; the `report` list is not even on the live board.
A standard per-duty summary would need: (a) a required summary/decisions/outputs block in the gate record (or a <runDir>/summary.<phase>.md) validated the way next_phase already is (policy.mjs), (b) an engine-side completion write folding the reply it already has (engine.mjs :823) into a durable per-duty record under runDir, (c) writers for sessionIds/videoUrl, and (d) a bound terminal report step compiled into policy/sequences.
