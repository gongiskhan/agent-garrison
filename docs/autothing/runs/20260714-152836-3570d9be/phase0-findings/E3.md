# FINDING-E3

## Finding

Native auto-compact in Claude Code 2.1.209 is NOT a flat percentage: the proactive trigger is `usedInputTokens >= effectiveWindow - 13,000`, where effectiveWindow = min(configured auto-compact window, model context) - min(modelMaxOutput, 20,000). On the common 200k window that is 167,000 tokens (~83.5%); on Sonnet 5 the default window is 967k (fires ~934k); on models where no window is configured (source "auto", e.g. local Opus 4.8 at 1M) the proactive path is skipped entirely and compaction is REACTIVE at the model's context limit - confirmed by this machine's transcripts showing trigger:"auto" events at preTokens 999,682-1,001,493. It is fully configurable/disable-able: `autoCompactEnabled` setting (default true, /config toggle), `DISABLE_AUTO_COMPACT=1` env (auto only), `DISABLE_COMPACT` env (kills /compact too), `CLAUDE_CODE_AUTO_COMPACT_WINDOW` env / `/autocompact` command (window 100k-1M, capped by model), and `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE` (1-100, threshold = min(window*pct, window-13k), proactive path only). A Garrison controller compacting at 60% would NOT fight native in steady state (60% of window is far below the ~83.5% native trigger; native becomes a backstop), but the cleanest design is to not run a parallel controller at all: set `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE=60` + `CLAUDE_CODE_AUTO_COMPACT_WINDOW` at spawn (both already whitelisted in Garrison's settings schema), which makes native itself compact at 60% and forces the proactive path on 1M/"auto" models where the pct override is otherwise ignored.

GARRISON-AT-60% INTERACTION ANALYSIS
- No fight in steady state: native proactive threshold is ~83.5% of a 200k window (167k) or reactive-at-limit on unconfigured 1M models. A controller compacting at 60% keeps usage below both, so native never fires and acts purely as a backstop. They only "meet" in races.
- Race window: native's check runs in the query loop before each model turn; a single huge tool-result turn can jump from <60% straight past the native threshold, so native may compact before an out-of-band Garrison controller reacts -> back-to-back double summarization (lossy). Mitigation: watch the session JSONL/stream for the `compact_boundary` system entry (trigger "auto") and skip the Garrison compaction for that cycle.
- Recommended integration (no parallel controller): inject `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE=60` AND `CLAUDE_CODE_AUTO_COMPACT_WINDOW=<model window>` at spawn. The window env is required to force the proactive path on models whose source would be "auto" (local 1M Opus), where pct override is otherwise ignored and compaction stays reactive-at-limit. Effective trigger becomes min(floor(window*0.60), window-13000). Garrison already validates both vars: src/lib/claude-settings-schema.json:513 (CLAUDE_AUTOCOMPACT_PCT_OVERRIDE) and :561 (CLAUDE_CODE_AUTO_COMPACT_WINDOW); env injection points exist at src/lib/runner.ts:1419-1437 (spawnGateway, GARRISON_SYSTEM_PROMPT_PATH env block at :1437, second block :1525-1528) and src/lib/runner.ts:359 (buildPrimaryRuntimeEnv).
- If Garrison must own compaction exclusively: set DISABLE_AUTO_COMPACT=1 (keeps manual /compact usable); Garrison then MUST compact before used >= modelContext - min(maxOutput,20k) - 3,000, or the session hits level "blocked" ("Context low ... /compact or /clear to continue") and turns stop. Do NOT use DISABLE_COMPACT - it disables manual /compact too, leaving only /clear.
- Denominator caveat: "60%" must be defined against the auto-compact window, not the statusline used_percentage (which always uses the full model window); on a 1M model with a 500k configured window these diverge by 2x.
- Thrash caveat: native's rapid-refill breaker (3 refills within 3 turns each) only guards the native path; a Garrison 60% controller needs its own thrash guard for context-heavy workloads.
