# FINDING-E8

## Finding

Confirmed: messaging a web-channel thread while its card is mid-run does nothing useful. The message queues on the gateway's single global turn chain BEHIND the card's running phase turn (which the engine dispatches through the same /chat/stream front door with a timeout up to 25 min), and when it finally runs, runRoutedTurn detects the thread's live card via attachedCard(sessionKey) but only logs a "card-attached" event and falls through to a plain inline conversational turn on the orchestrator — the run engine, the card JSON, and the card's next duty prompt never see the message (not dropped, but never delivered to the run). Steering could be attached as a card-owned sidecar file mirroring the Discuss brief (cards/<id>/brief.md read fresh at every dispatch by readCardBrief and folded into buildCardPrompt), since a mid-run PATCH to the card JSON is blocked by the D16 engine-owned lock and would break the engine's CAS saves. The exact loop that advances a card from one duty to the next is processChain (fittings/seed/kanban-loop/lib/engine.mjs:1240), which calls processCard (engine.mjs:523) per hop; the duty-boundary check points are processCard's pre-dispatch guard region (:529-542, where waitingOn is already checked) and processChain's between-hop seam after outcome.status === "moved" (:1247).
