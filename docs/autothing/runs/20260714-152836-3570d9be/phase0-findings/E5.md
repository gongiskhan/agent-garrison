# FINDING-E5

## Finding

The Kanban engine is the kanban-loop fitting: storage in fittings/seed/kanban-loop/lib/board.mjs, the transition function in lib/engine.mjs (processCard for gateway-dispatched turns, advanceCardPhase for the in-session garrison-doorway path), REST surface in scripts/server.mjs. "Done" is decided when the phase verdict (parseNextList / durable gate-status next_phase) plus rail fast-forward yields effectiveNext whose list is terminal (`landedTerminal = landed?.terminal || effectiveNext === "done"`, engine.mjs:1049 and :1421); the actual move is the saveCardCAS write that sets card.list = effectiveNext (engine.mjs:1177 / :1461). Side effects at done: a `routed` timeline event, lease release + coordination-intent cleanup, an optional commit fence, and notifyOriginTransition fired from inside saveCardCAS (board.mjs:291) which posts a short "Run complete" message (title + lastReply snippet + videoUrl + card link) to the originating web-channel thread. NO completion summary artifact is generated today - the only completion text is that channel outcomeMessage (notify-origin.mjs:51-66) built from card.lastReply; a Handoff Packet generator would hook either at the landedTerminal branches in engine.mjs (dispatched: ~1049/1095-1099; in-session: ~1421/1457-1459) or, to cover ALL movers including manual PATCH and gateway quick-card completion, at the single write choke point saveCardCAS in board.mjs:274-294 using the terminalTransition edge predicate (notify-origin.mjs:72). Cards are file-per-card JSON at ~/.garrison/kanban-loop/cards/<ulid>/card.json, created via POST /cards (server.mjs handleCreateCard:856) with fields including id, title, origin, originChannel, duty/level/sequence, runId/runDir/sliceId/sessionIds/videoUrl pointer fields, events timeline, fences, and lastReply.
