# FINDING-E2

## Finding

(a) The gateway can already read live context percentage TODAY via the statusline-to-screen path: the user's ~/.claude statusLine script computes ctx% from Claude Code's statusline JSON (.context_window.current_usage / context_window_size) and renders it into the TUI status row, which claude-pty's parseStatus() scrapes off the headless xterm into `contextPct`, streamed as SSE `status` events from GET /claude/stream and one-shot GET /claude/status on both the gateway and dev-env; the transcript path (~/.claude/projects/<enc-cwd>/<sessionId>.jsonl) is deterministic (pre-minted --session-id) and — verified empirically, contradicting a stale in-repo comment — gateway-spawned operative transcripts DO carry per-assistant-event `usage` fields, but nothing parses them yet (jsonl.mjs drops usage), so a transcript-based reader and a statusline-JSON tee are build items while the screen scrape is the only live source. (b) `/compact <free text>` passes everything after the command as custom summarization focus instructions (the PreCompact hook stdin JSON carries trigger + custom_instructions); completion is detectable from the transcript's `{"type":"system","subtype":"compact_boundary","compactMetadata":{trigger,preTokens,postTokens,durationMs,…}}` line followed by an `isCompactSummary:true` user message (verified in real transcripts on this machine), via the PostCompact hook event (already in Garrison's settings catalog; dev-env's install-hooks curl-forward pattern is the drop-in plumbing), or on-screen via prompt-return plus a sharp contextPct drop — but note OperativePtySession's 45s command timeout is shorter than observed real compactions (106-143s). (c) codex-runtime and opencode-runtime expose NO compaction and no context/token signals: codex is a stateless one-shot `codex exec` per turn (resume flag is set but never consumed, so context never accumulates and compaction is moot); opencode resumes sessions via `-s <id>` so context does accumulate server-side, but the adapter parses only text/error/sessionID NDJSON events, dropping any usage events, and never drives opencode's summarize/compact. (d) All three API-loop runtimes read token usage per call: agent-sdk-runtime from SDK `result` message `msg.usage`, openai-agents-runtime by summing `rawResponses[].usage`, and garrison-call maps anthropic/openai/ollama usage envelopes into `{inputTokens, outputTokens}`; only garrison-call surfaces usage in its return envelope — the two runtime fittings accumulate it internally solely for budgetTokens enforcement and omit it from the delegate {summary, artifacts} result and the gateway's routed-turn results (cost_usd:null, no tokens).
