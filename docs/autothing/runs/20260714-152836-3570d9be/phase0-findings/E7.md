# FINDING-E7

## Finding

The web channel (fittings/seed/web-channel-default, port 7083) is a stateless proxy + thread store: threads are opaque-keyed JSON transcripts under ~/.garrison/web-channel/threads/, and every POST /api/chat forwards the thread id as the gateway's sessionId, which the routed gateway stamps onto spawned cards as originChannel:{channel:"web", threadId} (D19) plus an in-RAM session-to-card attach map; the kanban board already routes TERMINAL outcomes back to the exact thread via notify-origin.mjs POSTing /api/threads/:id/messages, so origin-thread routing plumbing exists end-to-end but fires only on done/needs-attention, only for channel "web", and only as plain assistant text. There is NO standing session per conversation - there is exactly ONE rolling operative session per gateway (module-level `session` in gateway-pty.mjs, `claude --continue`, all threads share its model context and serialize through one `inflight` chain); the web-channel server holds no session at all, and thread history is never replayed into the model (it only seeds the UI). Per-duty summaries would render through the existing pipeline: thread messages are flat {role:user|assistant, text, ts}, paired by toHistory() into ClaudeChat Turn objects and rendered as markdown bubbles in ClaudeChat.tsx's turns.map loop; a server-side append is picked up by a 10s idle poll that re-mounts the chat, so posting per-duty summaries needs only a new caller of the append endpoint (plus a message `kind` field if distinct styling is wanted, since threads.mjs strips everything but role/text/ts). A user message flows UI -> POST /api/chat (thread id attached) -> web-channel handleChat/pipeChatSse -> gateway /chat/stream -> runRoutedTurn (classify, attach-or-card via createAutonomousCard -> board POST /cards + PATCH move to plan/implement) -> SSE done -> server-side tee of the exchange into the thread.

## (b) Standing session per conversation - THERE ISN'T ONE (it's one per gateway, shared by all threads)

threads.mjs header (lines 1-12) is explicit: "The operative is one rolling `claude --continue` conversation; 'threads' are a generic, opaque-keyed organizer ON TOP of it". The web-channel server holds NO Claude/PTY session at all - handleChat is a per-turn SSE proxy (server.mjs pipeChatSse:315-392).

The standing session lives in the gateway, ONE per composition:
- gateway-pty.mjs module state `session` (line 73) + `inflight` promise chain serializing ALL turns across ALL threads (line 76; enqueueTurn:716-721).
- Legacy path: spawnOperative (320-358) - OperativePtySession.spawn with continueSession decided by the marker file .garrison/operative-session-id (SESSION_ID_FILE line 70; hasPriorSession/markPriorSession 278-294; continue-wedge detection 304-318).
- Routed path (active): initRouting (205-265) -> createRoutedGateway -> MultiRuntimePool (from @garrison/claude-pty, gateway-routing.mjs:27, pool built at 1248) - ONE operative checkout + ONE warm classifier (gateway-routing.mjs:305-307); session = router.getOperativeSession() (gateway-pty.mjs:242, re-fetched per turn at 587); dead sessions re-checked out in ensureOperative (gateway-routing.mjs:512-524); provider/soul switches respawn with `--continue` + a context-carryover ring `_lastTurns` (gateway-routing.mjs:279, 766-772, respawnOperative:838-918).
- Souls mode (gateway.mjs): a standing headless orchestrator child (bootOrchestrator:317-366, session id persisted at .garrison/orchestrator-session-id:302-315, reboot-on-demand ensureOrchestrator:371-381) plus per-SOUL sessions in lib/session-registry.mjs - keyed by soul/session-uuid, never by web thread.

What removing/replacing it (e.g. per-thread sessions) would entail: (1) model-side continuity today comes ONLY from this shared rolling session - thread history is never replayed to the model (main.tsx passes initialHistory to ClaudeChat at line 602 for UI seeding only; the gateway gets just the new message), so per-thread isolation requires either per-thread pool checkouts keyed by sessionId (pool.checkout is single-slot per runtime id today) or replaying thread transcripts (which the thread store already holds in full); (2) the global `inflight` serialization (gateway-pty.mjs:716-721) must become per-session; (3) the resume machinery (marker file + `claude --continue` + wedge detection) is per-composition, not per-thread; (4) the AskUserQuestion watcher tails the ONE session's project JSONL (startAskWatcher:120-137) and /claude/* rich console binds session.handle (936-938) - both assume a single live PTY; (5) cross-thread context bleed (currently by design) would disappear, which also changes the "attach follow-up to live card" semantics.
