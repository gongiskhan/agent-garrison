The walkthrough machinery (skill at ~/.claude/skills/walkthrough, per-repo state at /home/ggomes/dev/garrison/.walkthrough/) records per-segment webm via playwright-cli 0.1.15's page.screencast API (a CDP-screencast wrapper with an HTML overlay layer), normalizes/stitches with ffmpeg, and CAN capture a 10-15 min horizon today: total video length is explicitly uncapped (segments concat; a ~90-min re-roll is referenced in notes), with the only per-segment limits being the run-code timeout (default 600 s, overridable via runTimeoutMs) and the 300 s default waitBefore/holdUntil timeouts. ## (a) 10-15 minute horizon - YES, with knobs
- Capture engine: playwright-cli `page.screencast.start({path,size})` producing a webm per browser segment (browser.mjs:485, 565). decisions.md:5-10 records the bake-off: chosen OVER raw Playwright `recordVideo` (no programmatic overlays) and over hand-rolled CDP screencast ("screencast already wraps CDP and adds the overlay layer"). playwright-cli is 0.1.15 at /home/ggomes/.nvm/versions/node/v20.19.4/lib/node_modules/@playwright/cli; its own reference doc is node_modules/playwright-core/lib/tools/cli-client/skill/references/video-recording.md (WebM VP8/VP9, showOverlay/showChapter, overlays are pointer-events:none).
- Per-segment limit: the run-code call is killed at `seg.runTimeoutMs || 600000` ms (browser.mjs:598; util.mjs exec SIGKILL at 22-24). storyboard-schema.md:68: "A browser segment may run up to 600 s before timing out (override per-segment with runTimeoutMs)". Overrun -> "produced no parseable result" (browser.mjs:601; noted in notes.md 2026-07-12: waitBefore timeouts must fit INSIDE runTimeoutMs).
- Wait limits: `waitBefore` default timeout 300000 ms (browser.mjs:483; schema line 139), `holdUntil` default 300000 ms (browser.mjs:552). Both must be raised above the real op duration for long waits.
- Total-length limit: NONE. "There is no beat cap and no duration cap" (SKILL.md:47, storyboard-schema.md:66). Segments are ffmpeg-concat'd (record.mjs:256-259). Session continuity across segments via `continue: true` (browser.mjs:576-594 keepOpen; record.mjs:83-98 validation), and the Playwright session survives across title/evidence segments. Practical proof of scale: notes.md 2026-07-12 "avoids a full 90-min re-roll" (island-splice re-record technique), and the 2026-07-14 run-engine entry uses waitBefore card-in-list selectors + title-card cuts to hide multi-minute phase waits honestly.
- So a 10-15 min horizon = either one long segment with runTimeoutMs >= ~900000 + raised holdUntil/waitBefore timeouts, or (preferred per notes) several continue/waitBefore segments with title-card bridges.

## (b) Time-lapse rendering
- Speed factor: segment-level `speed: <n>` (schema storyboard-schema.md:138). record.mjs:147 reads it, 154-160 applies `speedUp()` after normalize; ffmpeg.mjs:46-54 speedUp = `-filter:v setpts=PTS/${factor},fps=${fps},format=yuv420p` (video-only, no atempo needed - silent). Measured beat offsets are divided by speed (record.mjs:170 `_offsetMs: off / speed`) so manifest timestamps stay accurate. Evidence mode defaults speed "aggressive (8x)" (SKILL.md:108).
- Normal-speed key segments: inherent - speed is per-segment, so storyboards mix normal-speed segments with timelapsed `continue` segments; typical pattern (schema worked example lines 143-161): kickoff at 1x -> `continue`+`speed:8`+beat `holdUntil:{selector}` for the wait -> `continue` result segment at 1x. Alternative "cut": `waitBefore` waits off-camera (unrecorded) bridged by a title card "~= 3 minutes later" (SKILL.md:65, schema line 139/184).
- Caption overlays: NOT done with ffmpeg filters. ffmpeg.mjs:1-3: "This ffmpeg build lacks drawtext, so all text is rendered upstream as PNGs... composited with core overlay/vstack" (comment slightly stale - current code has no overlay/vstack filter; text surfaces are: (1) the per-beat caption HUD painted IN-PAGE during recording via page.screencast.showOverlay (browser.mjs caption() 96-109, bottom bar, accent border, FAILED red variant, tokens in style.mjs), so captions are baked into the webm and survive the setpts speedup; (2) title cards and evidence panels rendered HTML->PNG via a throwaway playwright-cli "render lane" browser (render.mjs titleCardHtml:12-25, evidencePanelHtml:54-114, renderHtmlToPng:128-140) then looped to clips with pngToClip (ffmpeg.mjs:20-28, `-loop 1 -t <dur>`)).
- ffmpeg filters actually used today: scale+pad+fps+format normalize chain (ffmpeg.mjs:6 `V()`), setpts speedup (46-54), concat demuxer + libx264 + `-movflags +faststart` (31-40), `-ss` single-frame extraction for the vision gate (57-63, driven by extract_frames.mjs), and lazy 480px thumb.jpg generation in serve.mjs.
