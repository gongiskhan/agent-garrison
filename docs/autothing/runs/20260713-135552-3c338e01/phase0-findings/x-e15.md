## FINDING-E15 — deepgram-voice today

**(a) Live streaming, not batch-only.** `fittings/seed/deepgram-voice/scripts/server.mjs` already implements a real WS live-STT relay at `WS /stream?sample_rate=<n>&utterance_end_ms=<ms>` (lines 384-467): it opens `wss://api.deepgram.com/v1/listen?model=...&encoding=linear16&interim_results=true&endpointing=300&utterance_end_ms=<n>&vad_events=true` and relays `{type:"ready"|"speech_started"|"transcript"|"utterance_end"|"error"}` events back to the client. Batch STT also exists as `POST /stt` → `https://api.deepgram.com/v1/listen` (REST, one-shot). TTS is **batch/REST only**: `POST /tts` → `https://api.deepgram.com/v1/speak?model=aura-asteria-en` (whole-utterance in, audio bytes out) — no streaming TTS. The standalone `scripts/connector.mjs` (automation catalog actions `transcribe`/`synthesize`) is REST-only by design, separate from the live server.

**(b) Vault key path.** `DEEPGRAM_API_KEY` is declared in `secret_scope` (`apm.yml:50-51`), consumed via `consumes: kind: vault`, and materialized server-side only: the runner's `src/lib/own-port-lifecycle.ts` (`vaultEnvForEntry`, referenced at line 176) injects it into the spawned process's env for own-port Fittings. `server.mjs` reads it from `process.env.DEEPGRAM_API_KEY` (line 42) and never forwards it to a client — `/stt`/`/tts` return 503 when unset, and the browser only ever talks to the same-origin web-channel proxy, which forwards to the voice Fitting without ever seeing the key.

**(c) What LIVE already has vs. what's missing.** STT side matches current Deepgram live API surface (confirmed via Deepgram docs): `interim_results`, `endpointing`, `utterance_end_ms`, `vad_events` are the real current parameters on `wss://api.deepgram.com/v1/listen`. Nothing to add there. TTS is the gap: Deepgram now offers **streaming TTS via WebSocket** at `wss://api.deepgram.com/v1/speak` (Aura-2 model, 40+ voices, token-by-token input streaming, ~3x faster than batch/ElevenLabs Turbo per Deepgram's own benchmark) — this is a genuine upgrade opportunity: swap the batch `POST /v1/speak` for the WS variant so read-aloud can start speaking before the full LLM reply is generated, mirroring the STT side's architecture.

Sources: [Text-to-Speech Streaming | Deepgram's Docs](https://developers.deepgram.com/docs/tts-websocket), [Now Available: Aura WebSocket TTS](https://deepgram.com/learn/aura-text-to-speech-adds-websocket-support-for-input-streaming)

## FINDING-E16 — web-channel client audio on iPhone over Tailscale

**(a) Serving.** Own HTTP server on port 7083 (`scripts/server.mjs`), plain `http` by default, `127.0.0.1` bind. TLS is optional and in-app: `config_schema.tls_cert`/`tls_key` (apm.yml:29-36) — when both are set and readable, it switches to `https.createServer` (server.mjs:782-849). The apm.yml comment explicitly recommends `tailscale serve` for a managed cert over this in-app fallback.

**(b) Secure-context reality.** `src/lib/tailnet-serve.ts` already exists and does exactly the needed mapping: it shells `tailscale serve status --json` and maps a local loopback port to its HTTPS tailnet URL (`https://<host>.ts.net:<port>`), used elsewhere (e.g. `browser-view-url.ts`) to hand the browser a reachable, secure-context URL instead of a loopback one. **Nothing in `web-channel-default` itself invokes `tailscale serve`** — the mapping utility exists at the platform level but the Fitting doesn't configure or check for it. Today, without the user manually running `tailscale serve --https=... http://127.0.0.1:7083` (or supplying `tls_cert`/`tls_key`), a Tailscale-only URL like `http://100.x.x.x:7083` is **not** a secure context, and iOS Safari's `getUserMedia`/mic capture will fail. `micCaptureAllowed()` in `legacy-voice.tsx:68-79` correctly gates on `window.isSecureContext`, so the failure mode is a disabled/greyed mic button, not a hard crash — but it's dead weight right now (see (c)).

**(c) PWA / mic-capture gap analysis.** No `manifest.json`, no service worker, no `apple-touch-icon` anywhere under `fittings/seed/web-channel-default` (confirmed via file search — none exist). `ui/index.html` only carries the legacy iOS meta-tag pair `apple-mobile-web-app-capable` + `apple-mobile-web-app-status-bar-style` (lines 7-8), which is enough for "Add to Home Screen" standalone mode on iOS *without* a manifest, but there's no icon declared (falls back to a screenshot thumbnail) and no offline/installability signal for Chrome/Android (which requires `manifest.json` + service worker for the install prompt).

More significant: **mic capture is currently entirely unwired.** `legacy-voice.tsx` (which contains all the `getUserMedia`/`AudioContext`/`MediaRecorder`/PCM-streaming code, including the AudioWorklet-adjacent `ScriptProcessorNode` streaming path at lines 210-334) is **not imported anywhere in `main.tsx`** — confirmed via grep, zero hits. The apm.yml's own `for_consumers` doc says it's "preserved... for a future voice re-integration via the component's `composerAdornment` slot." So today's active chat UI (`@garrison/claude-chat`) has no mic button, no speaker button, no read-aloud toggle wired in at all, even though the server-side proxy endpoints (`/api/voice/stt`, `/api/voice/tts`, `/api/voice/stream`) are fully built and working.

**Gaps to close, in order:**
1. Re-wire `legacy-voice.tsx`'s mic/speaker controls into `main.tsx` via the `composerAdornment` slot (currently dead code).
2. Modernize the streaming path off `ScriptProcessorNode` (deprecated) onto an `AudioWorkletNode` for PCM capture, since AudioWorklet is what iOS Safari standalone-mode expects for reliable low-latency mic streaming.
3. Add `manifest.json` (icons, `display: standalone`, `start_url`) + at least one `apple-touch-icon` link for a real home-screen icon and Android installability.
4. Either have `web-channel-default`'s setup/verify step detect and prefer a live `tailscale serve` mapping (via the existing `tailnet-serve.ts` utility) for HTTPS, or default-generate the in-app `tls_cert`/`tls_key` pair — right now neither is automatic, so the secure-context requirement for iOS mic capture over Tailscale is a manual, undocumented user step.
