**(c) Goal hook**: There is no discrete "goal hook" file — goal-mode is just a prompt prefix (`/goal`) plus the per-card **iteration cap** as the convergence guard (`engine.mjs:18` comment: "the convergence GUARD is the per-card iteration cap", enforced at `engine.mjs:457` and `1454`, default `cap = 10`). "Next phase" is decided per-turn by the operative's own reply being exact-matched against that list's `validNext` (`engine.mjs` `firstValidNextIn`/`parseNextList`-style logic around line 1386), not by consulting an external file.

**(d) Lifecycle event kinds**: Only 9 event `kind` values are ever emitted in `engine.mjs`: `coordination, deferred, dispatch, failed, interference, parked, phase-off, recovered, routed` (confirmed via `rg 'kind: "' engine.mjs`), plus `created`/`inference` from `board.mjs`. **`needs-input`, `blocked`, and `finished` are never emitted as event kinds.** "Blocked" is only observable indirectly (a card `list === "needs-attention"` with a `parked` event); "finished" is only observable as `list === "done"` (a terminal list, no dedicated event fires on arrival).

### E9 — Empty-output failure

`engine.mjs:1005` (`processCard`) and its batch counterpart `engine.mjs:1679-1688` (`processBatch`) **do** have explicit empty/no-verdict checks (`!replyText` → `reason: "empty-reply"`; no per-card verdict token → `reason: "no-exact-match"`) — these are not missing. Live evidence on disk shows the checks fire correctly but reveal a **race, not a missing check**:

- Card `~/.garrison/kanban-loop/cards/01KXD5E1PA43ZSYWN50W8T5M75/card.json` ("build a task list app") was parked with `attentionReason: "The Implement run produced no output..."` (`parkedFrom: "implement"`), logged at `2026-07-13T07:22:05Z`.
- Its iteration log `.../cards/01KXD5E1PA43ZSYWN50W8T5M75/log-2.md` is literally 2 lines (`# iteration 2`, blank) — confirming the router-level reply text really was empty when the engine read it.
- But `~/.garrison/runs/no-project/01KXD5ENZ19FY3QAV4Q6F8QWN6/gate-status.implement.json` shows `"status":"passed"` with a detailed notes field: task-list-app fully implemented, typecheck/lint/unit(7/7)/build/E2E all green, **committed** — and its file mtime is `08:24:17` local time, **~2m34s after** the card was already parked (`log-2.md` mtime `08:21:43`, card parked at 08:22:05 local).

So the real work finished and the phase gate file (and a git commit) landed successfully, but the **turn's reply-text stream** (consumed at `gateway-client.mjs:200` — `done.reply ?? done.text ?? ""` — the sole signal `processCard` uses for "did this turn conclude") went empty/closed before the underlying session's file writes caught up, and the engine's `!replyText` check (correctly, per its own contract) parked a card whose work had, in fact, succeeded. The gap is the **timing contract between the streamed `done` reply and the asynchronous on-disk gate evidence**, not a missing null-check — `hasPhaseGateEvidence` is only consulted *before* the empty-reply branch in the success path (`engine.mjs` ordering: gate-evidence-missing → evidence-missing → empty-reply), so a gate file that lands *after* the empty `done` event is never re-checked.
