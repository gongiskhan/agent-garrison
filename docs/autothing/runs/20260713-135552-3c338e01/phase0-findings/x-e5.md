**FINDING-E11 — probe generator: premise is wrong, there is no LLM call**

I'd flag this before seeding a fitting off it: `fittings/seed/improver/scripts/probe-generate.mjs` and `fittings/seed/improver/lib/probe-core.mjs` never call Ollama, or any model. The question text is built by pure deterministic string templates — `buildProbeQuestion()` (`probe-core.mjs:166-184`) and `buildRetrospectiveQuestions()` (`probe-core.mjs:224-238`) — driven off the correlated routing decision + Kanban card, not a model round-trip.

This is explicit in a comment at `probe-core.mjs:159-164`:
> "v1 phrases the question deterministically from the real classification so the operator sees a grounded question with no model round-trip on the Stop path (a model call would block every attended Stop on runtime availability + latency; the resolved target is recorded regardless...)"

What *does* happen: `resolveProbeTarget(policy)` (`probe-core.mjs:250-263`) reads the compiled policy's `matrix["probe-question"]` cell and returns `{targetId, runtime, provider, model, tier}` — resolving to target `cc-ollama-qwen` (provider `ollama-local`, model `qwen2.5:3b`, per `fittings/seed/orchestrator/config/routing.seed.json:144-196`). This resolved target is only logged to stderr for provenance/attribution (`probe-generate.mjs:73`, `:124-126`) — it's never fetched or invoked. The actual relay happens by handing the pre-built question to the *host* Claude session via a Stop-hook block reason (`relayReason()`, `probe-core.mjs:333-350`), which the model must relay verbatim via `AskUserQuestion` — no separate model call at all, local or Anthropic.
